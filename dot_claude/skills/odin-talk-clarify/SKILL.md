# odin-talk-clarify

要件ヒアリング・曖昧点解消スキル。odin司令塔のtalkフェーズで使用する。
要件や仕様の曖昧な箇所を洗い出し、対話を通じて1つずつ解消し、明確化された要件をQ&A形式で文書化する。

## Instructions

### 完了チェックポイントの原則

各ステップの最後には「完了チェックポイント」を設けている。
チェックポイントに記載された全ての条件を満たさない限り、次のステップに進んではならない。

### コンテキスト検出

$ARGUMENTS を確認し、odinコンテキストJSONが含まれているか判定する。

odinコンテキストJSONの例:
```json
{
  "odin_context": {
    "task": "通知機能の要件をヒアリング",
    "requirements_source": "https://confluence.example.com/...",
    "requirements_text": "通知機能を追加したい...",
    "artifacts": {
      "requirements": ".claude/artifacts/requirements-20260320-1500.md"
    }
  }
}
```

- odinコンテキストがある場合: artifactsから入力ファイルのパスを取得する
- odinコンテキストがない場合: 成果物管理ルールに従い入力ファイルを探索する

### 成果物管理

入力ファイルの探索順序:
1. $ARGUMENTSにファイルパスが明示指定されていればそれを使う
2. odinコンテキストのartifactsにパスがあればそれを使う
3. `.claude/artifacts/` 内の最新の `requirements-*.md` を使う
4. 該当ファイルが見つからない場合はAskUserQuestionでユーザーに確認する

### ステップ1: 対象分析

要件ソースを取得・読み込む。

方法A: 既存の要件ドキュメントがある場合
- `requirements-*.md` または指定されたファイルを読み込む
- 文書全体を把握してからステップ2に進む

方法B: ConfluenceページのURLがある場合
- AtlassianプラグインのgetConfluencePageツールでページ内容を取得する
- ページ内容を把握してからステップ2に進む

方法C: 口頭・テキストで要件を受け取る場合
- AskUserQuestionで以下を確認する:
  1. 実現したい機能の概要
  2. 対象ユーザー（誰が使うのか）
  3. 背景・動機（なぜ必要か）
  4. 参考情報（ConfluenceページのURL、既存ドキュメント等）

#### 完了チェックポイント（ステップ1）

- 要件の生データ（テキスト）が取得できていること
- 機能の概要が把握できていること

### ステップ2: 曖昧点の洗い出し

以下の5軸で要件を分析し、曖昧な箇所を列挙する。

軸1: 主語不明確
- 「誰が」行うのか明記されていない操作
- 例: 「削除できる」→ 誰が削除できるか不明

軸2: 動詞曖昧
- 「適切に」「なるべく」「うまく」等の定義不明な表現
- 例: 「適切なエラーメッセージを表示」→ 何が適切か不明

軸3: 条件不完全
- エラーケース・例外処理の記載なし
- 境界値・上限下限の未定義
- 例: 「ファイルをアップロードできる」→ サイズ制限・形式制限が未定義

軸4: 数値未定義
- パフォーマンス要件・SLA・件数制限等が「高速」「十分に」等の曖昧な表現
- 例: 「高速に表示する」→ 何秒以内かが未定義

軸5: 矛盾
- 複数の箇所で異なる記述がある
- 前提条件と要件が矛盾している

全ての曖昧点を番号付きリストとしてまとめる（A-001, A-002 等）。

#### 完了チェックポイント（ステップ2）

- 5軸で分析が完了していること
- 曖昧点が番号付きリストとして列挙されていること（0件の場合は「曖昧点なし」と記録）

### ステップ3: AskUserQuestionで対話的に解消

ステップ2で列挙した曖昧点を1つずつAskUserQuestionで確認する。

- 一度に1つの質問をする（まとめて質問しない）
- 選択肢がある場合は2〜4個に絞って提示する
- 回答を記録し、次の質問へ進む
- 全ての曖昧点を確認し終えるまで繰り返す

優先度の高い確認事項:
1. スコープ境界（何が入って何が入らないか）
2. エラーケース・異常系の扱い
3. 権限・認可（誰が何をできるか）
4. 数値・パフォーマンス要件

確認が困難な場合の扱い:
- 「現時点では決定できない」場合は未解決ポイントとして記録する
- 後から決定できる事項については暫定値を設定する

#### 完了チェックポイント（ステップ3）

- 全ての曖昧点に対して確認が完了していること（または未解決として記録済みであること）
- 各曖昧点に対する回答が記録されていること

### ステップ4: 明確化ドキュメントの出力

1. `.claude/artifacts/` ディレクトリが存在しなければ作成する
2. 現在時刻を取得し、`yyyyMMdd-HHmm` 形式のタイムスタンプを生成する
3. 以下のフォーマットで明確化ドキュメントを出力する

出力先: `.claude/artifacts/clarified-{yyyyMMdd-HHmm}.md`

```
# 要件明確化ドキュメント

作成日: {yyyy-MM-dd HH:mm}
対象機能: {機能の概要}
参照元: {入力した要件ドキュメントのパスまたはURL}

## Q&A形式の明確化

### A-001: {曖昧点の概要}

軸: {主語不明確 / 動詞曖昧 / 条件不完全 / 数値未定義 / 矛盾}
元の記述: {原文の引用}
質問: {確認した質問内容}
回答: {ユーザーからの回答}
明確化後: {明確になった要件の記述}

### A-002: {曖昧点の概要}
...

## 未解決の確認ポイント

| No. | 質問 | 影響する箇所 | 暫定対応 |
|-----|------|------------|---------|
| U-001 | {質問内容} | {影響する要件・機能} | {暫定値または「要確認」} |

## 明確化サマリー

- 解決済み: {N}件
- 未解決: {N}件
- 新たに発見したスコープ外事項: {あれば記載}
```

4. ドキュメントの概要をユーザーに提示する
5. odinから呼ばれた場合は、出力ファイルパスをodinに返す

#### 完了チェックポイント（ステップ4）

- `.claude/artifacts/clarified-*.md` が出力されていること
- 全てのA-XXXに回答が記録されていること
- 未解決ポイントが明記されていること（0件でもセクションを記載する）

## Examples

### 通知機能の要件をヒアリングして明確化する

ユーザー: 「通知機能の要件を詰めたい。requirements-20260320-1500.md を確認して」

```
ステップ1: 対象分析
  requirements-20260320-1500.md を読み込む
  → 通知作成・既読・削除の3機能が記載

ステップ2: 曖昧点の洗い出し
  A-001 [主語不明確]: 「通知を削除できる」→ 誰が削除できるか
  A-002 [条件不完全]: 通知の最大件数が未定義
  A-003 [動詞曖昧]: 「リアルタイムで表示する」→ どの程度がリアルタイムか
  A-004 [矛盾]: 「管理者のみ既読にできる」と「全ユーザーが既読にできる」が混在

ステップ3: 対話で解消
  Q: 「通知の削除は誰ができますか？1. 通知受信者のみ 2. 管理者のみ 3. 両方」
  A: 「受信者本人のみ」
  Q: 「通知の最大保存件数を設けますか？1. 設けない 2. ユーザーごとに上限を設ける」
  A: 「ユーザーごとに100件まで」
  ...

ステップ4: ドキュメント出力
  → .claude/artifacts/clarified-20260320-1600.md
  解決済み: 3件、未解決: 1件（既読権限の詳細は後日確認）
```
