# odin-talk-triage

入力解析・タスクトリアージスキル。odin司令塔のtalkフェーズで使用する。
ユーザーの雑な入力（口頭メモ・アイデア断片・複数要望の混在）を解析し、4種類に分類してodin-boardに登録する。

## Instructions

### 完了チェックポイントの原則

各ステップの最後には「完了チェックポイント」を設けている。
チェックポイントに記載された全ての条件を満たさない限り、次のステップに進んではならない。

### コンテキスト検出

$ARGUMENTS を確認し、odinコンテキストJSONが含まれているか判定する。

odinコンテキストJSONの例:
```json
{
  "odin_context": {
    "task": "複数の要望をトリアージして",
    "input_text": "認証を直したい。あとパフォーマンスが気になる。あと新しい通知機能も欲しい",
    "artifacts": {}
  }
}
```

- odinコンテキストがある場合: `input_text` フィールドから入力テキストを取得する
- odinコンテキストがない場合: $ARGUMENTS をそのまま入力テキストとして扱う。引数がない場合はAskUserQuestionで入力を求める

### ステップ1: 入力解析

入力テキストから以下の要素を抽出する。

目的:
- ユーザーが最終的に達成したいこと（1文で要約）

課題:
- 現在困っていること・不満な点・解消したい問題

背景情報:
- 関連するコンテキスト・経緯・制約

暗黙の前提:
- 明示されていないが前提になっていると推測されること
- 例: 特定の技術スタックへの依存、既存機能への影響範囲

複数の要望が混在している場合は、それぞれを個別のタスク候補として分解する。

#### 完了チェックポイント（ステップ1）

- 入力テキストが取得できていること
- タスク候補が1件以上抽出されていること
- 各候補の目的が1文で要約されていること

### ステップ2: タスク分類・提案

各タスク候補を以下の4分類に振り分ける。

分類基準:

investigate（調査タスク）:
- 原因や現状が不明な問題、技術的な調査が必要な事柄
- 例: 「パフォーマンスが遅い気がする」「○○が時々壊れる」
- odin-boardステータス: odin、tags: [investigate]

task（実装タスク）:
- 具体的な実装・修正・追加が明確なもの
- 例: 「通知機能を追加したい」「ログイン画面のバリデーションを修正する」
- odin-boardステータス: inbox、tags: [odin-task]

human（人間判断タスク）:
- 設計判断・優先度決定・ビジネス要件の確認など、人間の意思決定が必要なもの
- 例: 「どの機能を先に実装するか決める」「ライセンス方針を検討する」
- odin-boardステータス: human

inbox（未整理・保留）:
- 分類が困難、情報が不足している、後で検討すべきもの
- 例: 「なんかもっとよくしたい」「全体的に見直したい」
- odin-boardステータス: inbox

分類結果をまとめ、各タスクに分類理由を付けて提示する。

#### 完了チェックポイント（ステップ2）

- 全タスク候補に分類が付与されていること
- 各分類の理由が1文で説明されていること

### ステップ3: アンチパターン検出

分類されたタスク全体を俯瞰し、以下の3つのアンチパターンに該当しないか検出する。

アンチパターン1: 丸投げ検出

トリガー: 設計判断や技術選定を含むタスクで、判断基準が示されていない
（例: 「いい感じのアーキテクチャにして」「ベストな方法で実装して」）

警告: "この判断はあなたの成果物に直接影響します。Why/Whetherを自分で評価してからodinに実行を委譲しましょう"

アクション: 分類をhuman（人間判断タスク）に変更し、判断すべき観点をbodyに追記する

アンチパターン2: 理解不足の委譲

トリガー: ユーザーが理解していない技術領域のタスクを直接実装依頼している
（例: ドキュメントを読まずに「WebSocketを使って実装して」「BFFパターンで書いて」）

警告: "この技術について先に調査タスクを作り、結果を理解してから実装に進みましょう"

アクション: 元のtaskをinvestigate + taskの2件に自動分割して提案する

アンチパターン3: 検証なし受け入れ

トリガー: odin-boardのdoneタスクの成果物に対して「そのまま使って」「提出して」等の指示がある

警告: "成果物のレビューポイントを明示します。最低限ここは自分で確認してください"

アクション: レビューすべき観点（機能・セキュリティ・パフォーマンス・UX）をリストアップしてbodyに追記する

アンチパターンが検出された場合は、該当タスクの横に警告マークと修正提案を表示する。

#### 完了チェックポイント（ステップ3）

- 全タスクに対してアンチパターン検出が完了していること
- 検出されたアンチパターンの修正提案が示されていること（0件でも「検出なし」と記録する）

### ステップ4: ユーザー承認

AskUserQuestionで以下を確認する。

提示内容:
- 抽出されたタスク一覧（タイトル・分類・理由）
- アンチパターン警告（あれば）
- 自動分割の提案（あれば）

確認事項:
1. このタスク分類で登録してよいか
2. タイトルや分類に修正があるか
3. 追加・削除するタスクがあるか

ユーザーが承認したタスクのみを次のステップで登録する。

#### 完了チェックポイント（ステップ4）

- ユーザーが承認したタスク一覧が確定していること
- 修正がある場合は反映済みであること

### ステップ5: 登録・実行

承認されたタスクをodin-boardに登録する。

登録コマンド（1件ずつ実行する）:
```bash
~/.config/tmux/odin-board.sh add "{タイトル}" -t "{tags}" --body "{body}" --pri medium
```

docsの登録（参考URLがある場合）:
```bash
~/.config/tmux/odin-board.sh docs add {task-id} "{タイトル}" "{URL}" web
```

investigate分類のバックグラウンド実行:

investigate分類のタスクは、登録直後にバックグラウンドで調査を開始する。

1. ステータスをodinに変更する
```bash
~/.config/tmux/odin-board.sh move {task-id} odin
```

2. odin-paneフィールドに"agent"マーカーを設定する
```bash
~/.config/tmux/odin-board.sh field {task-id} odin-pane agent
```

3. タスクの性質に応じてスキルを選択し、Agent(run_in_background: true)で調査を実行する
   - 技術調査・ベストプラクティス調査 → odin-think-research
   - 不具合・原因不明の問題 → odin-think-investigate

4. 調査結果を `~/.local/share/odin-board/docs/{task-id}/` に保存する

5. 調査完了後、ステータスをodinからhumanに変更し、odin-paneを空に戻す
```bash
~/.config/tmux/odin-board.sh move {task-id} human
~/.config/tmux/odin-board.sh field {task-id} odin-pane ""
```

#### 完了チェックポイント（ステップ5）

- 承認された全タスクがodin-boardに登録されていること
- investigate分類のタスクがバックグラウンドで実行開始されていること

### ステップ6: 完了報告

以下の形式で報告する。

```
## トリアージ完了

登録タスク: {N}件

| # | タイトル | 分類 | ステータス |
|---|--------|------|---------|
| 1 | {タイトル} | investigate | odin（調査中） |
| 2 | {タイトル} | task | inbox |
| 3 | {タイトル} | human | human |

{アンチパターン警告があれば再掲}

次のアクション:
- investigate: 調査完了後にhumanへ移動します（バックグラウンド実行中）
- task: odinに「{タイトル}を実装して」と依頼してください
- human: あなたの判断が必要なタスクです。odin-boardで確認してください
```

#### 完了チェックポイント（ステップ6）

- 登録タスクの一覧が報告されていること
- 各タスクの次のアクションが明示されていること

## Examples

### 複数の雑な要望をまとめてトリアージする

ユーザー: 「トリアージして。認証が時々落ちる気がする、通知機能を追加したい、あとダッシュボードのデザインどうするか考えたい」

```
ステップ1: 入力解析
  タスク候補A: 認証が時々落ちる
    目的: 認証の安定性を確保する
    課題: 散発的な認証失敗が発生している
    暗黙の前提: 本番環境で発生している可能性あり

  タスク候補B: 通知機能を追加したい
    目的: ユーザーへの通知を実装する
    課題: 通知機能が未実装

  タスク候補C: ダッシュボードのデザインを考えたい
    目的: ダッシュボードUIの方向性を決める
    課題: デザイン方針が未決定

ステップ2: 分類
  A → investigate（原因不明の問題、調査が必要）
  B → task（実装要件が明確）
  C → human（デザイン方針は人間の判断が必要）

ステップ3: アンチパターン検出
  検出なし

ステップ4: ユーザー承認
  Q: 「3件のタスクを確認しました。この分類で登録しますか？」
  A: 「OKです」

ステップ5: 登録
  A: odin-board.sh add "認証が時々落ちる（調査）" -t "investigate"
     → move {id} odin → Agent(run_in_background)でodin-think-investigate実行
  B: odin-board.sh add "通知機能を追加" -t "odin-task"
  C: odin-board.sh add "ダッシュボードのデザイン方針を決める"（human）

ステップ6: 完了報告
  3件登録完了。Aは調査中（バックグラウンド）
```

### アンチパターンを検出して修正提案する

ユーザー: 「いい感じのマイクロサービス構成で全部作り直して。あとWebSocketも追加して」

```
ステップ1: 入力解析
  タスク候補A: マイクロサービス構成で再設計
  タスク候補B: WebSocket機能の追加

ステップ2: 分類
  A → task（実装依頼として解釈）
  B → task（実装依頼として解釈）

ステップ3: アンチパターン検出
  A → アンチパターン1（丸投げ検出）
     「いい感じの」という主観的判断基準のみで、分割方針・サービス境界・移行戦略が未定義
     → humanに変更。判断すべき観点: サービス境界の基準、移行戦略（段階的/一括）、インフラコスト
  B → アンチパターン2（理解不足の委譲）
     WebSocketの適用箇所・プロトコル設計が未確認
     → investigate（WebSocket適用箇所の調査）+ task（WebSocket実装）に分割提案

ステップ4: ユーザー承認
  Q: 「2件の警告があります。修正後の分類で登録しますか？」
  → 修正内容を提示してユーザー確認を得る
```
