# odin-do-refactor

TDDリファクタリングスキル。odin司令塔のdoフェーズで使用する。
テストカバレッジを確認しながら、外部振る舞いを変えずにコードの内部品質を改善する。
小さなステップで変更し、常にテストが緑の状態を保つ。

## ガードレール（安全制御）

このスキルは以下の安全制御を遵守する:

- git push / git commit は実行しない（それぞれodin-do-pr / odin-do-commitスキルの責務）
- `git reset --hard` / `git clean -f` 等の破壊的操作を実行しない
- テスト失敗時の変更戻しは `git checkout -- ファイル名`（特定ファイルのみ）を使用し、`git checkout .`（全ファイル一括戻し）は使用しない
- `git checkout .` や `git reset --hard` が必要な状況ではAskUserQuestionでユーザーの承認を得る

## Instructions

### 完了チェックポイントの原則

各ステップの最後には「完了チェックポイント」を設けている。
チェックポイントに記載された全ての条件を満たさない限り、次のステップに進んではならない。

### コンテキスト検出

odin司令塔からの自動呼び出し時（odinコンテキストJSONあり）:
- リファクタリング対象・方針はコンテキストから取得する
- 設計ドキュメント（design-*.md）があれば参照する

ユーザー直接呼び出し時（odinコンテキストなし）:
- ユーザーにリファクタリング対象と目的を確認する
- 対象コードを分析して改善点を提案する

### テストコマンドの自動検出

リファクタリング開始前に、プロジェクトのテストコマンドを自動検出する:

1. package.jsonの "scripts.test" を確認 → あればそのコマンドを使用
2. vitest.config.* の存在確認 → あれば `pnpm vitest run`
3. jest.config.* の存在確認 → あれば `pnpm jest`
4. Cargo.toml の存在確認 → あれば `cargo test`
5. pytest.ini / pyproject.toml の [tool.pytest] 確認 → あれば `pytest`
6. 上記いずれにも該当しない場合はAskUserQuestionでテストコマンドを確認する

E2Eテストコマンドの自動検出:

1. playwright.config.* の存在確認 → あれば `pnpm playwright test`
2. cypress.config.* の存在確認 → あれば `pnpm cypress run`
3. 上記いずれにも該当しない場合はE2Eテストをスキップする

以降のステップでは、検出したテストコマンドを使用する。

### リファクタリングの基本原則

- 外部から見た振る舞いを変えない（入力と出力が変わらない）
- 1つの変更ごとにテストを実行する（小さなステップ）
- テストが赤になったらすぐに変更を戻す
- 完璧を目指さず、段階的に改善する

### ステップ1: 対象の確認とテストカバレッジの評価

1. リファクタリング対象を特定する:
   - $ARGUMENTSに対象ファイル・関数が指定されていればそれを使う
   - 指定がない場合はAskUserQuestionで確認する

2. 現在のテストカバレッジを確認する:
   ```bash
   # 自動検出したテストコマンドに --coverage オプションを付けて実行する
   ```

3. カバレッジが不十分な場合（目安: 重要なパスが80%未満）:
   - リファクタリング前にテストを追加する
   - AskUserQuestionで「テスト追加してからリファクタリングしてよいか」を確認する
   - odin-do-testスキルでテストを追加する

4. リファクタリングが必要な箇所を分析する:

   コードの臭い（Code Smell）を確認する:
   - 重複したコード（DRY違反）
   - 長すぎる関数（1関数20行以上を目安）
   - 多すぎる引数（4つ以上を目安）
   - 深すぎるネスト（3レベル以上）
   - 意図不明な変数名・関数名
   - コメントで補足が必要な複雑なロジック

#### 完了チェックポイント（ステップ1）

- リファクタリング対象が特定されていること
- テストカバレッジが十分であること（主要なパスがカバーされている）
- 不十分な場合はテスト追加が完了していること
- コードの臭いが特定されていること

### ステップ2: リファクタリング計画の作成

特定した問題を小さなステップに分解する:

分解の原則:
- 1ステップ = 1つの独立した改善（名前変更、関数抽出、重複除去など）
- ステップ間で依存関係がある場合は順序を明記する
- 各ステップは5〜15分で完了できる粒度にする

代表的なリファクタリングパターン:

| パターン | 対象 | 方法 |
|---------|------|------|
| 関数の抽出 | 長すぎる関数 | 一部を別関数として切り出す |
| 変数名の改善 | 意図不明な変数 | 役割がわかる名前に変更 |
| 重複の除去 | 同じロジックが複数箇所 | 共通関数に抽出 |
| 早期リターン | 深いネスト | ガード節で早期にリターン |
| 型の明示化 | any型の使用 | 具体的な型を定義 |
| 定数の抽出 | マジックナンバー | 名前付き定数に変更 |

リファクタリング計画を提示し、AskUserQuestionで承認を得る。

#### 完了チェックポイント（ステップ2）

- リファクタリングステップが5〜15分粒度に分解されていること
- ステップの順序と依存関係が明確になっていること
- ユーザーの承認が得られていること

### ステップ3: TDDリファクタリングの実行

各ステップを以下のサイクルで実行する:

#### 各ステップの実行手順

1. 変更前にテストを実行して「緑」を確認する:
   ```bash
   # 自動検出したテストコマンドで全テストを実行する
   ```

2. 1つのリファクタリングを実施する（小さな変更のみ）

3. テストを実行して「緑」を維持していることを確認する:
   ```bash
   # 自動検出したテストコマンドで全テストを実行する
   ```

4. テストが赤になった場合:
   - 変更を即座に戻す（`git checkout -- 対象ファイル名` で特定ファイルのみ戻す、またはEditツールで手動修正する）
   - `git checkout .`（全ファイル一括戻し）は使用しない。必要な場合はAskUserQuestionでユーザーの承認を得る
   - 変更が小さく問題を特定できる場合は直接修正する
   - 問題が複雑な場合はAskUserQuestionで相談する

5. 緑を維持できたら次のステップへ進む

#### リファクタリング中の注意事項

- 機能追加は同時に行わない（別のTDDサイクルで実施）
- テストコードもリファクタリングの対象に含める
- コードコメントを整備する（日本語で記述）
- 不要なコード・コメントは削除する

#### 完了チェックポイント（ステップ3）

- 全ステップが実行されていること
- 各ステップ後にテストが緑を維持していること
- 外部から見た振る舞いが変わっていないこと（テストが証明している）

### ステップ4: リファクタリング後の検証

superpowers:verification-before-completion のパターンに従い検証する。

> superpowersプラグインが未インストールの場合: 以下の検証手順を直接実行する。

1. 全テストの通過確認:
   ```bash
   # 自動検出したテストコマンドで全テストを実行する
   ```

2. 型チェックの実施:
   ```bash
   pnpm typecheck
   # または
   pnpm tsc --noEmit
   ```

3. lintの確認:
   ```bash
   pnpm lint
   ```

4. コードレビューの実施（重要な変更の場合）:
   - odin-auto-reviewスキルを使用してセルフレビューを行う

5. 変更のサマリーを提示する:
   - どのコードの臭いを解決したか
   - before/afterの比較（主要な変更のみ）
   - テストカバレッジの変化（あれば）

#### 完了チェックポイント（ステップ4）

- 全テストが通過していること
- 型エラー・lintエラーがないこと
- 変更サマリーがユーザーに提示されていること

## Examples

### 長すぎる関数を分割するリファクタリング

ユーザー: 「src/lib/notifications.ts をリファクタして」

```
ステップ1: 対象確認
  processNotification 関数が50行、深いネスト
  テストカバレッジ: 85%（十分）
  コードの臭い: 長すぎる関数、深いネスト

ステップ2: 計画作成
  S-001: validateNotificationInput 関数を抽出（10分）
  S-002: saveNotificationToDB 関数を抽出（10分）
  S-003: sendNotificationEmail 関数を抽出（15分）
  S-004: マジックナンバーを定数化（5分）
  → ユーザー承認

ステップ3: TDDリファクタリング
  各ステップ前後でテスト実行（緑を維持）
  S-001: 抽出 → テスト緑 ✓
  S-002: 抽出 → テスト緑 ✓
  S-003: 抽出 → テスト緑 ✓
  S-004: 定数化 → テスト緑 ✓

ステップ4: 検証
  全テスト通過・型エラーなし・lintクリーン
  「processNotification が50行→8行に削減」と報告
```
