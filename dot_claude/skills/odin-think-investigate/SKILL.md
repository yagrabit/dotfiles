# odin-think-investigate

不具合原因調査スキル。odin司令塔のthinkフェーズで使用する。
バグ・エラー・予期しない動作について体系的に調査し、根本原因を特定した上で企画向けと開発者向けの2形式で調査レポートを出力する。

## Instructions

### 完了チェックポイントの原則

各ステップの最後には「完了チェックポイント」を設けている。
チェックポイントに記載された全ての条件を満たさない限り、次のステップに進んではならない。

### コンテキスト検出

$ARGUMENTS を確認し、odinコンテキストJSONが含まれているか判定する。

odinコンテキストJSONの例:
```json
{
  "odin_context": {
    "task": "ログイン後にリダイレクトが失敗する",
    "symptom": "ログイン成功後に404ページに遷移する",
    "error_message": "Error: NEXT_REDIRECT at /api/auth/callback",
    "reproduction_steps": "1. /login にアクセス 2. 正しい認証情報を入力 3. ログインボタンクリック",
    "artifacts": {}
  }
}
```

- odinコンテキストがある場合: そこから症状・エラーメッセージ・再現手順を読み取る
- odinコンテキストがない場合: ステップ1からAskUserQuestionで確認する

### ステップ1: 症状の把握

AskUserQuestionで以下を確認する（未回答の項目のみ質問する）:

1. エラーメッセージ・スタックトレース（正確なテキスト）
2. 再現手順（何をすると発生するか、ステップバイステップ）
3. 期待する動作（本来どうあるべきか）
4. 実際の動作（実際に何が起きているか）
5. 発生頻度（毎回/特定条件/まれに）
6. 発生環境（本番/ステージング/ローカル/特定ブラウザ等）
7. いつから発生しているか（最後に正常動作した時点）
8. 最近の変更（デプロイ、設定変更、依存関係更新等）

#### 完了チェックポイント（ステップ1）

- エラーメッセージまたは症状の具体的な説明が得られていること
- 再現手順が明確であること
- 期待する動作と実際の動作の差が明確であること

### ステップ2: 体系的なデバッグ（5フェーズ）

superpowers:systematic-debugging のパターンに従い、以下の5フェーズで調査を進める。

> superpowersプラグインが未インストールの場合: 以下の5フェーズを直接実行する。
各フェーズは Explore エージェントまたは直接ツールで調査する。

フェーズ1: 根本原因の仮説立案
- エラーメッセージからエラーが発生している箇所を特定する
- スタックトレースを追って呼び出し元を特定する
- エラーパターンから考えられる原因を3〜5つ列挙する
- 調査ツール: Grep（エラーメッセージ、関数名で検索）、Read（関連ファイル）

フェーズ2: コードパターン分析
- 仮説と関連するコードを読む
- 最近の変更（git log, git diff）を確認する
- 類似の実装と比較して差異を特定する
- 調査ツール: Bash（git log, git diff）、Read（関連ファイル）、Grep（パターン検索）

フェーズ3: 仮説の検証
- 各仮説について「この原因だとしたら他に何が起きるはずか」を検証する
- 仮説を裏付ける証拠を集める
- 仮説を否定する証拠を探す
- 最も証拠が多い仮説を根本原因として特定する

フェーズ4: 根本原因の確定と影響範囲の特定
- 根本原因を1文で明確に記述する
- 同様の問題が他の箇所でも発生しうるか確認する
- 影響を受けるユーザー・機能の範囲を特定する
- 修正に伴うリスク（副作用）を評価する

フェーズ5: 上流課題の分析
根本原因の背後にある仕様・デザイン・アーキテクチャ上の課題を洗い出す。
技術的な直接原因を修正するだけでなく、本質的な解決に必要なステークホルダーとの調整項目を特定する。

以下の観点で根本原因を再評価する:

- 仕様・要件の曖昧性: 「期待する動作」自体が明文化されているか。仕様が曖昧で解釈が分かれていないか。仕様自体が誤っている可能性はないか
- デザイン・UXへの影響: 修正によってUIや操作フローが変わるか。ユーザー体験の観点でデザイナーの判断が必要か
- アーキテクチャ判断: 応急処置で済ませるか、設計の根本的な見直しが必要か。技術的負債が蓄積していないか
- 他チーム・他機能への波及: 修正が他チームの担当機能やAPIの契約に影響するか
- ビジネス判断: 暫定対応で進めるか、リリースを止めるか。優先度の再評価が必要か

各項目について「調整が不要」と判断した場合はその理由を記録する。

#### 完了チェックポイント（ステップ2）

- 根本原因が1文で明確に記述できていること
- 根本原因を裏付ける証拠（コード・ログ等）が特定されていること
- 影響範囲が明確になっていること
- 上流課題の分析が完了し、各観点について「調整が必要」または「不要（理由付き）」が記録されていること
- 不明点が残る場合は「調査限界」として記録されていること

### ステップ3: 調査レポートの出力（3セクション）

1. `.claude/artifacts/` ディレクトリが存在しなければ作成する
2. 現在時刻を取得し、`yyyyMMdd-HHmm` 形式のタイムスタンプを生成する
3. 1つのファイルに企画向けセクションと開発者向けセクションの両方を出力する

出力先: `.claude/artifacts/investigation-{yyyyMMdd-HHmm}.md`

```

作成日: {yyyy-MM-dd HH:mm}
調査対象: {不具合の概要}
ステータス: {調査中 / 根本原因特定済み / 修正方針決定済み}
```

## Examples

### 例1: ライブラリ更新起因のバグ（調整不要ケース）

ユーザー: 「ログイン後に404が出る。Error: NEXT_REDIRECT at /api/auth/callback」

```
ステップ1: 症状把握
  症状: ログイン成功後に404ページへ遷移
  再現: 毎回発生、全ユーザー
  最近の変更: Next.js 14.1.0 → 14.2.0 にアップグレード

ステップ2: 5フェーズデバッグ
  フェーズ1: NEXT_REDIRECTエラーでGrep → /api/auth/callback/route.ts 発見
  フェーズ2: git diff でredirect()呼び出し方の変更を確認
            Next.js 14.2でredirect()のthrow挙動が変わった
  フェーズ3: try-catchでredirect()を囲んでいる → 14.2ではredirectはthrowする仕様
  フェーズ4: try-catchの外にredirect()を移動すれば修正可能
  フェーズ5: 上流課題なし — 仕様通りの動作を復元するだけ。全5観点「該当なし」

ステップ3: レポート出力
  企画向け: 「ログイン後の画面遷移が壊れています。ライブラリ更新の影響で発生。修正2時間」
  調整項目: 全5観点「該当なし」— 即着手可能
  開発者向け: 「Next.js 14.2でredirect()はthrowするため try-catch 内では動作しない」
  → .claude/artifacts/investigation-20260320-1430.md
```

### 例2: 仕様の曖昧性が根本原因のバグ（調整必須ケース）

ユーザー: 「注文確定後に在庫数がマイナスになることがある」

```
ステップ1: 症状把握
  症状: 同時注文時に在庫がマイナスになる
  再現: 高負荷時に発生（残り1個の商品に複数ユーザーが同時注文）
  期待動作: 在庫切れエラーを表示すべき

ステップ2: 5フェーズデバッグ
  フェーズ1-3: 在庫チェックと在庫減算の間にロックがない（race condition）
  フェーズ4: SELECT FOR UPDATE追加で技術的には修正可能
  フェーズ5: 上流課題を発見
    - 仕様確認（必須）: 在庫切れ時の振る舞いが未定義。
      後勝ちでキャンセル通知？ 先着順で待ち行列？ 要PO判断
    - デザイン調整（必須）: 在庫切れ通知のUI・メッセージが未設計
    - アーキテクチャ: 楽観ロックvs悲観ロック、PO判断に依存するため並行検討
    - 他チーム: 決済チームに影響（キャンセルフロー追加の可能性）
    - ビジネス判断: 暫定対応（悲観ロックで在庫0でエラー）で先行リリースするか要判断

ステップ3: レポート出力
  企画向け: 「同時注文で在庫管理に不整合が発生。技術修正に加え、在庫切れ時のユーザー体験の設計が必要」
  調整項目: 仕様確認・デザイン調整が修正着手前に必須。決済チームへの連絡が並行で必要
  開発者向け: 「race conditionの解消 + 在庫切れハンドリングの実装」
  着手前提条件: 仕様確認（在庫切れ時の振る舞い）とUI設計の完了を待つ
  → .claude/artifacts/investigation-20260322-0930.md
```
