# Test Plan

設計ドキュメントと明確化された要件をもとに、テストシナリオを洗い出し、テスト計画を作成するスキル。
テストコードの実装ではなく「何をテストするか」を定義することに集中する。

## Instructions

### 完了チェックポイントの原則

各ステップの最後には「完了チェックポイント」を設けている。
チェックポイントに記載された全ての条件を満たさない限り、次のステップに進んではならない。

### ステップ1: 入力ファイルの読み込み

1. `.claude/artifacts/` から以下のファイルを探す:
   - `design-*.md`（必須）
   - `clarified-requirements-*.md`（必須）
   - `research-*.md`（任意: 存在すればテストフレームワーク等の情報源として活用）
   - $ARGUMENTSでパスが指定されていればそれを使用
2. 設計ドキュメントのコンポーネント・API・ロジック一覧を把握する

#### 完了チェックポイント（ステップ1）

- design-*.md が読み込まれていること
- clarified-requirements-*.md が読み込まれていること

### ステップ2: テストシナリオの洗い出し

以下の観点でテスト対象を抽出する:

1. 機能要件からの抽出: 各機能要件に対して正常系・異常系のシナリオ
2. エッジケースの洗い出し: 空データ、境界値、大量データ、同時操作
3. 非機能要件からの抽出: パフォーマンス、アクセシビリティ等（該当する場合）

テスト種別に分類する:
- ユニットテスト: 個別関数、カスタムフック、ユーティリティ
- コンポーネントテスト: React コンポーネントの表示・操作
- 統合テスト: API結合、コンポーネント連携
- E2Eテスト: ユーザーフロー全体

#### 完了チェックポイント（ステップ2）

- 全機能要件がテストシナリオに変換されていること
- テスト種別ごとに分類されていること

### ステップ3: テスト計画の出力

1. `.claude/artifacts/` ディレクトリが存在しなければ作成する
2. テスト計画を出力する:

出力先: `.claude/artifacts/test-plan-{yyyyMMdd-HHmm}.md`

```
# テスト計画

作成日: {yyyy-MM-dd}
対象設計: {design-*.md のパス}

## テスト方針

- テストフレームワーク: {vitest / jest 等、research-*.md から判明していれば記載}
- テストファイル配置: {__tests__/ or *.test.ts 等、既存パターンに合わせる}

## ユニットテスト

### {対象モジュール/関数名}

| No | シナリオ | 期待結果 | 優先度 |
|----|---------|---------|-------|
| U1 | {正常系: ...} | {期待される振る舞い} | 高 |
| U2 | {異常系: ...} | {エラーハンドリング} | 高 |
| U3 | {エッジケース: ...} | {...} | 中 |

## コンポーネントテスト

### {コンポーネント名}

| No | シナリオ | 期待結果 | 優先度 |
|----|---------|---------|-------|
| C1 | {初期表示} | {...} | 高 |
| C2 | {ユーザー操作} | {...} | 高 |

## 統合テスト

| No | シナリオ | 期待結果 | 優先度 |
|----|---------|---------|-------|
| I1 | {API連携} | {...} | 高 |

## E2Eテスト

| No | シナリオ | 手順 | 期待結果 | 優先度 |
|----|---------|------|---------|-------|
| E1 | {ユーザーフロー} | {1. ... 2. ... 3. ...} | {...} | 高 |

## 優先度の定義

- 高: ビジネスクリティカルなパス、バグが発生すると致命的
- 中: 重要だが代替手段がある、エッジケース
- 低: UIの細かい挙動、nice-to-have
```

#### 完了チェックポイント（ステップ3）

- `.claude/artifacts/test-plan-*.md` が出力されていること
- 全テストシナリオに優先度が付与されていること

### ステップ4: テスト計画の提示

1. テスト計画の要約をユーザーに提示する（テスト種別ごとの件数、高優先度の一覧）
2. AskUserQuestionで以下の選択肢を提示する:
   - 承認する: テスト計画に問題なし
   - 修正する: テストシナリオの追加・削除・修正（ステップ2に戻る）
   - 中止する: ワークフローを中止する

#### 完了チェックポイント（ステップ4）

- ユーザーが「承認する」を選択していること
- 「修正する」の場合はステップ2に戻って修正が完了していること

## Examples

### 通知機能のテスト計画

ユーザー: 「テスト計画を作って」（design-*.mdとclarified-requirements-*.mdがある状態）

```
ステップ1: 入力読み込み
  design-20260320-1530.md → 通知機能の設計
  clarified-requirements-20260320-1500.md → 通知機能の要件
  research-20260320-1430.md → vitest, @testing-library/react, __tests__/ 配置

ステップ2: テストシナリオ洗い出し
  ユニットテスト: 5件（通知取得ロジック、既読更新、日付フォーマット等）
  コンポーネントテスト: 6件（NotificationBell, NotificationList）
  統合テスト: 3件（API結合）
  E2Eテスト: 2件（通知確認フロー、既読フロー）

ステップ3: 出力
  → .claude/artifacts/test-plan-20260320-1600.md

ステップ4: 提示
  合計16件（高: 8, 中: 6, 低: 2）
  → ユーザー承認
```
