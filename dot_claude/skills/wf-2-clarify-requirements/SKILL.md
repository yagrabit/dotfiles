# Clarify Requirements

要件を取得し、曖昧な点を洗い出し、ユーザーとの対話で明確化するスキル。

## Instructions

### 完了チェックポイントの原則

各ステップの最後には「完了チェックポイント」を設けている。
チェックポイントに記載された全ての条件を満たさない限り、次のステップに進んではならない。

### ステップ1: 要件の取得

入力ソースに応じて取得方法を切り替える:

1. ConfluenceページのURL/タイトルが指定された場合:
   - Atlassian MCPツールでページ内容を取得する
   - `mcp__plugin_atlassian_atlassian__getConfluencePage` でページ本文を取得
   - 子ページがある場合は `mcp__plugin_atlassian_atlassian__getConfluencePageDescendants` で一覧取得
   - 関連ページの参照が必要な場合は `mcp__plugin_atlassian_atlassian__searchConfluenceUsingCql` で検索
2. ユーザーが口頭で要件を伝える場合:
   - AskUserQuestionで要件の全体像を聞き取る
3. ファイルパスが指定された場合:
   - 指定ファイルを読み込む

#### 完了チェックポイント（ステップ1）

- 要件のソースデータが手元にあること
- 要件の全体像が把握できていること

### ステップ2: 要件の構造化

取得した要件を以下の4カテゴリに整理する:

1. 機能要件: ユーザーが実現したいこと（ユーザーストーリーまたは機能一覧）
2. 非機能要件: パフォーマンス、セキュリティ、アクセシビリティ、対応ブラウザ等
3. 制約: 技術的制約、スケジュール、互換性要件
4. 前提条件: 既存システムとの関係、外部サービス依存

#### 完了チェックポイント（ステップ2）

- 4カテゴリに整理されていること
- 各カテゴリが空の場合は「明記なし（要確認）」としてマークされていること

### ステップ3: 曖昧な点の検出

以下の観点で要件の曖昧な点・不足点を洗い出す:

1. 曖昧表現の検出:
   - 「など」「適切に」「必要に応じて」「柔軟に」「可能であれば」等
   - 具体的な数値や条件が不明な表現
2. 不足項目の検出:
   - エラーハンドリング（異常系の振る舞い）
   - エッジケース（空データ、大量データ、同時操作）
   - 権限・認可（誰がこの機能を使えるか）
   - データのライフサイクル（作成・更新・削除・保持期間）
   - 画面遷移（どこから来てどこへ行くか）
3. 矛盾の検出:
   - 要件間で矛盾する記述がないか

検出結果を疑問リストとしてまとめる。

#### 完了チェックポイント（ステップ3）

- 曖昧表現・不足項目・矛盾のチェックが完了していること
- 疑問リストが作成されていること

### ステップ4: ユーザーとの対話で明確化

1. 疑問リストから優先度の高いものを2〜4項目選んでAskUserQuestionで質問する
   - 選択肢がある場合は具体的な選択肢を提示する
   - 背景情報（なぜこの質問が重要か）を添える
2. ユーザーの回答を要件に反映する
3. 残りの疑問がある場合はステップ4を繰り返す
4. 全ての疑問が解消したらステップ5に進む

#### 完了チェックポイント（ステップ4）

- 全ての疑問が解消されていること（または「ユーザー判断で保留」と明示されていること）
- ユーザーの回答が要件に反映されていること

### ステップ5: リスクレベル判定と出力

1. リスクレベルを判定する:
   - 高: 認証、決済、DB設計変更、セキュリティ関連、公開APIの変更
   - 中: 新画面追加、API追加、状態管理の変更
   - 低: UIテキスト変更、スタイル修正、バグ修正
2. `.claude/artifacts/` ディレクトリが存在しなければ作成する
3. 以下のフォーマットで明確化された要件を出力する:

出力先: `.claude/artifacts/clarified-requirements-{yyyyMMdd-HHmm}.md`

```
# 明確化された要件

作成日: {yyyy-MM-dd}
ソース: {Confluenceページ名/URL or 口頭ヒアリング}
リスクレベル: {高/中/低}

## 機能要件

{番号付きリストで整理}

## 非機能要件

{パフォーマンス、セキュリティ等}

## 制約

{技術的制約、スケジュール等}

## 前提条件

{既存システムとの関係等}

## 明確化のログ

{質問と回答の要約。後から「なぜこう決めたか」を追跡可能にする}

## 保留事項

{判断を保留した項目があれば記載}
```

4. 出力内容の要約をユーザーに提示する

#### 完了チェックポイント（ステップ5）

- `.claude/artifacts/clarified-requirements-*.md` が出力されていること
- 機能要件/非機能要件/制約/前提条件が全て記載されていること
- リスクレベルが付与されていること
- 未解決の疑問が残っていないこと（保留は明示されていればOK）

## Examples

### Confluenceから要件を取得して明確化する場合

ユーザー: 「このConfluenceページの要件を整理して: https://example.atlassian.net/wiki/spaces/DEV/pages/12345」

```
ステップ1: 要件の取得
  Confluence MCP経由でページ内容を取得
  → ユーザー管理画面の改修要件（ロール管理機能の追加）

ステップ2: 構造化
  機能要件: ロール作成/編集/削除、ユーザーへのロール割当
  非機能要件: 明記なし（要確認）
  制約: 既存の権限テーブルとの互換性
  前提条件: 現行の認証基盤（NextAuth.js）を使用

ステップ3: 曖昧な点の検出
  - 「適切な権限チェック」→ 具体的にどのエンドポイントで？
  - エラー時の振る舞いが未定義
  - ロール数の上限は？
  - ロール削除時の割当済みユーザーの扱いは？

ステップ4: ユーザーとの対話
  Q1: ロール削除時、そのロールが割り当てられたユーザーはどうなりますか？
      a) デフォルトロールにフォールバック
      b) 削除をブロック（割当解除を先に要求）
      c) ユーザーのロールをnullにする
  A1: → b) 削除をブロック

  Q2: ロール数の上限と、非機能要件（レスポンスタイム等）の目安はありますか？
  A2: → 上限50、レスポンスは特に厳しくないが一覧表示は1秒以内

ステップ5: リスクレベル判定と出力
  リスクレベル: 高（認証・認可関連）
  → .claude/artifacts/clarified-requirements-20260320-1500.md
```

### 口頭で要件を伝える場合

ユーザー: 「ダッシュボードにグラフを追加したい」

```
ステップ1: AskUserQuestionでヒアリング
  - どんなデータのグラフ？
  - リアルタイム更新は必要？
  - 既存のグラフライブラリは使っている？

ステップ2〜5: 通常通り実行
  リスクレベル: 中
  → .claude/artifacts/clarified-requirements-20260320-1600.md
```
