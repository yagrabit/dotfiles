# TDD Cycle

タスク一覧から指定タスクをTDDサイクル（Red-Green-Refactor）で実装するスキル。
tdd-guideエージェントを活用し、テスト作成→実装→リファクタリング→品質検証を1タスクずつ進める。

## Instructions

### 完了チェックポイントの原則

各ステップの最後には「完了チェックポイント」を設けている。
チェックポイントに記載された全ての条件を満たさない限り、次のステップに進んではならない。

### ステップ1: タスクの選択

1. `.claude/artifacts/` から `tasks-*.md` を探す（$ARGUMENTSでパスが指定されていればそれを使用）
2. タスク一覧を読み込み、各タスクのステータスを確認する
3. 実装対象タスクを決定する:
   - $ARGUMENTSでタスク番号が指定されている場合はそのタスク
   - 指定がない場合は「未着手」かつ依存タスクが全て完了しているものから次の1つを選択
   - 該当タスクがない場合はAskUserQuestionでユーザーに確認
4. `test-plan-*.md` が存在する場合は対応するテストケースも読み込む

#### 完了チェックポイント（ステップ1）

- 実装対象タスクが1つ選択されていること
- タスクのTDDサイクル定義（Red/Green/Refactor）が把握できていること

### ステップ2: Red - テスト作成

1. tdd-guideエージェント（`~/.claude/agents/tdd-guide.md`）をAgentツールで起動し、テスト作成を委譲する
   - タスクのTDDサイクル定義の「Red」に記載された内容を指示として渡す
   - `test-plan-*.md` の対応テストケースがあればそれも渡す
   - 既存のテストパターン（`research-*.md` 参照）に合わせるよう指示する
2. テストを実行し、失敗（Red）であることを確認する
   - テストフレームワークの自動検出: package.jsonからvitest/jest/playwright等を判定

#### 完了チェックポイント（ステップ2）

- テストファイルが作成されていること
- テスト実行結果が失敗（Red）であること

### ステップ3: Green - 最小限の実装

1. tdd-guideエージェントを起動し、テストを通す最小限の実装を委譲する
   - タスクのTDDサイクル定義の「Green」に記載された内容を指示として渡す
   - 「テストを通す最小限の実装のみ行う」ことを強調する
2. テストを実行し、成功（Green）であることを確認する
3. 既存テストも全て通ることを確認する

#### 完了チェックポイント（ステップ3）

- 対象テストが成功（Green）であること
- 既存テストが全て通ること

### ステップ4: Refactor - リファクタリング

1. タスクのTDDサイクル定義に「Refactor」の記載がある場合:
   - tdd-guideエージェントを起動し、リファクタリングを委譲する
   - コードの改善（重複除去、命名改善、共通化等）を実施
2. Refactorの記載がない場合でも、以下をチェック:
   - 明らかな重複コード
   - 命名の不整合
3. テストを実行し、全て成功を維持していることを確認する
4. lint、型チェック等の品質チェックを実行する:
   - PostToolUseフックで自動実行されていればスキップ
   - されていなければ手動で実行

#### 完了チェックポイント（ステップ4）

- 全テストが成功していること
- lint/型チェックにエラーがないこと

### ステップ5: タスク完了と次のアクション

1. `tasks-*.md` のタスクステータスを「未着手」→「完了」に更新する
2. 実装結果の要約をユーザーに報告する:
   - 作成/変更したファイル一覧
   - テスト結果（成功件数）
   - 残りの未着手タスク数
3. AskUserQuestionで次のアクションを確認する:
   - 次のタスクに進む: 次の未着手タスクのTDDサイクルを開始（ステップ1に戻る）
   - コミットする: /pr-2-commit スキルを案内
   - PRを作成する: /pr-4-create スキルを案内
   - 中止する: ワークフローを中止

#### 完了チェックポイント（ステップ5）

- `tasks-*.md` のステータスが更新されていること

## Examples

### 通知APIの実装（T3）

ユーザー: 「T3を実装して」（tasks-*.mdがある状態）

```
ステップ1: タスク選択
  tasks-20260320-1545.md を読み込み
  T3: 通知取得API - GET /api/notifications（中）
  依存タスク: T1（型定義）→ 完了済み、T2（DBマイグレーション）→ 完了済み
  → T3を選択

ステップ2: Red
  tdd-guideエージェント起動
  → src/__tests__/api/notifications.test.ts を作成
  → テスト実行: FAIL（3 failed）

ステップ3: Green
  tdd-guideエージェント起動
  → src/app/api/notifications/route.ts を実装
  → テスト実行: PASS（3 passed）
  → 既存テスト: PASS（全47件）

ステップ4: Refactor
  → エラーレスポンスの共通化を実施
  → テスト実行: PASS（全50件）
  → lint/型チェック: エラーなし

ステップ5: タスク完了
  → tasks-20260320-1545.md の T3 ステータスを「完了」に更新
  → 報告:
    作成: src/app/api/notifications/route.ts, src/__tests__/api/notifications.test.ts
    テスト: 3件追加、全50件成功
    残りタスク: T4(未着手), T5(未着手), T6(未着手), T7(未着手)
  → 「次のタスクに進む」
```

### 引数なしで次のタスクを自動選択

ユーザー: 「次のタスクを進めて」

```
ステップ1: タスク選択
  tasks-20260320-1545.md を読み込み
  未着手かつ依存完了: T4(通知既読API), T5(NotificationBell), T6(NotificationList)
  着手順が最も早い T4 を自動選択

ステップ2〜5: 通常通り実行
```
