# odin-think-qa-design

QA試験項目生成スキル。odin司令塔のthinkフェーズで使用する。
Confluence設計ドキュメント・Epicブランチ差分・ソースコードを入力とし、ISTQBテスト設計技法を参考にしたQA試験項目書をMarkdown形式で生成する。

## ガードレール（安全制御）

このスキルは以下の安全制御を遵守する:

- Confluenceページは読み取り専用（更新・削除しない）
- git push / git commit / 破壊的git操作（reset --hard, checkout ., clean -f）を実行しない
- ソースコードの変更を行わない（読み取り専用の分析のみ）

## Instructions

### 完了チェックポイントの原則

各ステップの最後には「完了チェックポイント」を設けている。
チェックポイントに記載された全ての条件を満たさない限り、次のステップに進んではならない。

### コンテキスト検出

$ARGUMENTS を確認し、odinコンテキストJSONが含まれているか判定する。

odinコンテキストJSONの例:
```json
{
  "odin_context": {
    "task": "QA試験項目の作成",
    "target_area": "epic/feature-name ブランチの変更範囲",
    "focus": ["仕様網羅性", "リスクベース優先度"],
    "constraints": ["Confluenceは読み取り専用", "ソースコード変更禁止"],
    "artifacts": {},
    "qa_params": {
      "confluence_url": "https://xxx.atlassian.net/wiki/spaces/XXX/pages/123",
      "epic_branch": "epic/feature-name",
      "base_branch": "develop",
      "scope": "全機能"
    }
  }
}
```

- odinコンテキストがある場合: qa_paramsからConfluence URL・ブランチ名を取得する
- odinコンテキストがない場合: ステップ1でAskUserQuestionを使用する

### 成果物管理

入力ファイルの探索順序:
1. $ARGUMENTSにファイルパスが明示指定されていればそれを使う
2. odinコンテキストのqa_params.confluence_urlがあればConfluenceから取得する
3. 該当ファイルが見つからない場合はAskUserQuestionでユーザーに確認する

出力ファイル: `.claude/artifacts/qa-test-items-{yyyyMMdd-HHmm}.md`

### ステップ1: 入力収集

1. odinコンテキストJSON または AskUserQuestion で以下を取得:
   - Confluence設計ドキュメントのURL
   - Epicブランチ名（またはベースブランチとの比較対象）
   - テスト対象のスコープ（全機能 or 特定機能）
2. `mcp__plugin_atlassian_atlassian__getConfluencePage` でConfluenceページを取得する。関連ページの探索が必要な場合は `mcp__plugin_atlassian_atlassian__searchConfluenceUsingCql` で検索する
3. `git diff {base}...{epic}` でEpicブランチの変更差分を取得する
4. 変更ファイルのimportチェーンをGrep/Globで逆探索し影響範囲を特定する

#### 完了チェックポイント（ステップ1）

- Confluenceドキュメントの内容を取得できていること
- git diffの変更ファイル一覧が把握できていること

### ステップ2: 要件構造化

1. Confluenceドキュメントから機能要件を抽出する
2. 各要件を「主語 + 動詞 + 条件 + 結果」に分解する
3. 要件IDを付与する（REQ-{機能}-{連番}）
4. 要件とgit diff変更ファイルの紐付けを行う（要件トレーサビリティ）

#### 完了チェックポイント（ステップ2）

- 全ての機能要件が構造化されていること
- 要件とファイルの紐付けが完了していること

### ステップ3: 変更影響分析

1. git diffから変更の種別を分類する:
   - 新規追加 / 修正 / 削除 / リネーム

2. 変更種別ごとの影響範囲を以下のチェック表で確認する:

| 変更種別 | 直接影響 | 間接影響 | テスト範囲 |
|---------|---------|---------|-----------|
| APIレスポンス変更 | API呼び出し元の全画面 | データ加工ロジック | API単体 + 呼び出し元UI |
| DBスキーマ変更 | 対象テーブルのCRUD操作 | 関連JOIN/ビュー | データ整合性 + 関連機能 |
| UIコンポーネント変更 | 使用箇所の全画面 | 共通コンポーネント依存先 | ビジュアルリグレッション + 機能テスト |
| ビジネスロジック変更 | 該当関数のユニットテスト | 上位呼び出し元 | ユニット + 統合 + 該当フローのE2E |
| 認証・認可変更 | 認証フロー全体 | 全保護リソース | セキュリティテスト + 権限別アクセス |
| 設定値変更 | 設定読み込み箇所 | 設定依存の全機能 | 影響機能の正常系 + 異常系 |

3. 影響範囲を要件にマッピングする

#### 完了チェックポイント（ステップ3）

- 変更種別ごとの影響範囲が特定されていること
- 影響範囲が要件に紐付いていること

### ステップ4: テスト設計技法の選定（要件モデル起点）

要件から中間モデルを構築し、そのモデルに適切なテスト設計技法を適用する。コード変更パターンは補助的な参照にとどめ、要件・仕様の構造をプライマリな判断基準とする。

中間モデルの構築:
1. 入力条件表: 各機能の入力パラメータ・型・制約・有効値域を一覧化する
2. 状態一覧: 画面・データ・ワークフローの取りうる状態と遷移を整理する
3. 業務ルール表: 条件と結果の対応関係を表形式で整理する

モデルに対する技法選定:

| 要件の特性 | 適用技法 | LLMの着目ポイント |
|-----------|---------|-----------------|
| 入力パラメータに型・範囲制約がある | 同値分割 + 境界値分析 | 入力条件表の有効値域・制約条件 |
| 複数条件の組み合わせで結果が変わる | デシジョンテーブル | 業務ルール表の条件-結果対応 |
| 状態を持ち遷移ルールがある | 状態遷移テスト | 状態一覧の遷移図 |
| 独立したパラメータが3つ以上ある | ペアワイズ | 入力条件表のパラメータ間独立性 |
| エンドツーエンドの業務シナリオがある | ユースケーステスト | 要件のアクターとゴール |

一つの機能に複数の技法が適用される場合は全て適用する。ただし、ケース爆発を防ぐため、各技法の適用結果が重複する場合は統合する。

#### 完了チェックポイント（ステップ4）

- 各要件に適用するテスト設計技法が選定されていること

### ステップ5: テストケース生成

各要件 × 各技法からテストケースを生成する。

テストケースの構造（IEEE 829を参考にしたカスタムフォーマット）:

```markdown
### TC-{機能}-{連番}: {テスト名}

- テスト対象: {機能・画面名}
- テスト目的: {このテストで何を確認するか}
- テスト観点: {正常系/異常系/境界値/セキュリティ等}
- 優先度: {P0-P3}（ステップ6で付与）
- 実行方式: {自動（Playwright）/ 手動}（ステップ7で付与）
- 自動化判定理由: {なぜ自動/手動と判定したか}
- 使用技法: {適用したテスト設計技法}
- 関連要件: {REQ-ID}
- 環境要件: {必要なブラウザ・画面サイズ等}
- 依存テスト: {先に実行が必要なTC-ID、なければ「なし」}
- 前提条件:
  - {テスト実行に必要な初期状態}
- 手順:
  1. {具体的な操作手順}
  2. ...
- 入力データ:
  - {テストに使用する具体的なデータ}
- 期待結果:
  - {各手順後に期待される挙動・表示}
- 実行ヒント（自動テストの場合のみ）:
  - start_url: {テスト開始URL}
  - 検証対象: {確認すべきテキスト・要素・URL・HTTPステータス}
  - 待機条件: {操作後に待つべきイベント（ページ遷移、要素表示等）}
```

テストケース生成の網羅性チェック:
- 正常系: 全ての要件に最低1つの正常系テスト
- 異常系: 入力バリデーション、エラーハンドリング
- 境界値: 数値範囲、文字数制限、日付範囲
- セキュリティ: 認証・認可、XSS、SQLインジェクション（該当箇所のみ）

#### 完了チェックポイント（ステップ5）

- 全要件にテストケースが紐付いていること
- 正常系・異常系・境界値が網羅されていること

### ステップ6: リスクベース優先度付け

各テストケースにリスクスコアを算出し優先度を付与する。

リスクスコア = 発生可能性(1-5) × 影響度(1-5)

発生可能性の判断基準:
- 5: 新規実装・複雑なロジック・過去にバグが多い領域
- 4: 既存機能の大幅改修
- 3: 既存機能の一部修正
- 2: 設定値変更・軽微な修正
- 1: リネーム・コメント変更等

影響度の判断基準:
- 5: データ損失・セキュリティ侵害・システム停止
- 4: 主要機能の障害・課金関連
- 3: 一部機能障害（回避策あり）
- 2: 表示崩れ・UX劣化
- 1: 軽微な表示問題

優先度マッピング:
- P0 (20-25): 最優先。全テストケースを実施
- P1 (13-19): 高優先。標準的に実施
- P2 (7-12): 中優先。変更影響時に追加
- P3 (1-6): 低優先。余裕があれば実施

#### 完了チェックポイント（ステップ6）

- 全テストケースに優先度が付与されていること

### ステップ7: 自動化可否分類

各テストケースに対して以下の判断フローで自動化可否を分類する。

```
1. 手順が明確に定義できるか?
   → NO → 手動（探索的テスト・アドホック操作が必要）
   → YES → 2へ

2. 期待結果を機械的に判定できるか?
   → NO → 手動（主観判断が必要: デザイン、UX、文章トーン等）
   → YES → 3へ

3. ブラウザ操作で完結するか?
   → NO → 手動（印刷、メール受信、外部連携等）
   → YES → 4へ

4. Playwright MCPで技術的に実行可能か?
   → NO → 手動（以下の制約に該当）
   → YES → 自動（Playwright）
```

判断フロー分岐4の技術的制約チェックリスト:
- 多要素認証（MFA/2FA）が必要 → 手動（OTP入力の自動化は不可）
- CAPTCHA/reCAPTCHAが存在 → 手動
- ファイルダウンロード後の内容検証 → 手動（DLファイル内部の確認はPlaywright MCP範囲外）
- 特定の日時・タイムゾーンに依存する挙動 → 手動（時間操作が困難）
- CSSスタイルの視覚的検証（色・サイズ・位置） → 手動（browser_snapshotはアクセシビリティツリーのみ返却し、CSS情報は含まない）

自動化適性が高いテストの例:
- フォームバリデーション（テキスト・要素の存在で判定可能）
- ページ遷移・URL確認
- APIレスポンスのステータスコード検証
- エラーメッセージのテキスト確認
- レスポンシブ対応確認（要素の表示/非表示の切り替え）
- 認証フロー（ID/パスワード方式のみ）

手動が必要なテストの例:
- デザインの完成度・ブランド一貫性
- UXの直感性
- 印刷レイアウト
- メール配信内容
- 外部サービス連携
- 業務知識に基づく妥当性判断
- MFA/CAPTCHA必須のフロー

#### 完了チェックポイント（ステップ7）

- 全テストケースに「自動」「手動」が付与されていること
- 分類理由が記録されていること

### ステップ8: 試験項目書の出力

1. `.claude/artifacts/` ディレクトリが存在しなければ作成する
2. 現在時刻を取得し、`yyyyMMdd-HHmm` 形式のタイムスタンプを生成する
3. 以下のフォーマットで試験項目書を出力する

出力先: `.claude/artifacts/qa-test-items-{yyyyMMdd-HHmm}.md`

```markdown
# QA試験項目書

## メタ情報
- 対象機能: {機能名}
- 対象Epicブランチ: {branch}
- ベースブランチ: {base branch}
- Confluence設計ドキュメント: {URL}
- 作成日: {date}
- 総テストケース数: {n} (自動: {auto} / 手動: {manual})
- 適用テスト設計技法: {使用した技法一覧}

## サマリー

| 優先度 | 自動 | 手動 | 合計 |
|--------|------|------|------|
| P0     | -    | -    | -    |
| P1     | -    | -    | -    |
| P2     | -    | -    | -    |
| P3     | -    | -    | -    |

## 要件トレーサビリティ

| 要件ID | 要件概要 | テストケース |
|--------|---------|-------------|
| REQ-{機能}-001 | ... | TC-001, TC-002 |

## テストケース一覧

### TC-{機能}-{連番}: {テスト名}
（ステップ5のフォーマット）
```

4. レポートの要約をユーザーに提示する
5. odinから呼ばれた場合は、出力ファイルパスをodinに返す

#### 完了チェックポイント（ステップ8）

- `.claude/artifacts/qa-test-items-*.md` が出力されていること
- 全テストケースに優先度・実行方式・実行ヒントが記載されていること
- 要件トレーサビリティ表が完成していること

## Examples

### 新機能のQA試験項目を作成する

ユーザー: 「Confluenceの設計書をもとにQA試験項目を作って」

```
ステップ1: 入力収集
  Confluence URL: https://xxx.atlassian.net/wiki/spaces/XXX/pages/123
  Epicブランチ: epic/notification-feature
  ベース: develop
  → Confluenceからページ取得、git diffで40ファイルの変更を検出

ステップ2: 要件構造化
  REQ-NOTIF-001: ユーザーは未読通知を一覧表示できること
  REQ-NOTIF-002: ユーザーは通知を既読にできること
  REQ-NOTIF-003: 管理者は全ユーザーに通知を送信できること
  → 要件とファイルの紐付け完了

ステップ3: 変更影響分析
  API追加: /api/notifications（CRUD操作）
  UIコンポーネント: NotificationList, NotificationDetail（新規追加）
  → 影響範囲をマッピング

ステップ4: テスト設計技法選定
  入力条件表 → 同値分割 + 境界値（通知タイトル文字数等）
  状態一覧 → 状態遷移（未読→既読→削除）
  業務ルール → ユースケース（通知送信フロー）

ステップ5-7: テストケース生成・優先度・分類
  TC-NOTIF-001: 通知一覧が表示できること（P0, 自動）
  TC-NOTIF-002: 通知を既読にできること（P0, 自動）
  TC-NOTIF-003: 1000件の通知でもページネーションが正しいこと（P1, 自動）
  TC-NOTIF-004: 通知文面のトーンが適切か（P2, 手動）
  ...合計24件（自動: 18件, 手動: 6件）

ステップ8: 出力
  → .claude/artifacts/qa-test-items-20260402-1500.md
```
