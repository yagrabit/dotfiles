# odin-auto-improve

odinシリーズのスキル群自体を客観的に評価し、改善するメタスキル。
WebSearchで最新のベストプラクティスを調査し、メタ認知で自分自身のスキルを批判的に分析する。

「知恵のために片目を犠牲にした」オーディンのように、自己を客観視して進化し続ける。

## Reflexion統合

Reflexion（arXiv:2303.11366）のパターンを適用し、改善サイクルの記憶を蓄積する。

- Actor: ステップ4（改善実行）- サブエージェントがスキルを修正する
- Evaluator: ステップ2後半 - 前回改善の効果を3段階で判定する
- Self-Reflection: ステップ6 - 教訓を言語化しエピソディックメモリに永続化する
- Episodic Memory: `.claude/artifacts/odin-episodic-memory.md` - ステップ0で読み、ステップ6で書く

メモリは2層構造:
- 知識索引（要約層）: 有効パターン・アンチパターン・未解決バックログ。O(1)で参照可能
- エピソードログ（詳細層）: 各改善サイクルの詳細記録。時系列でO(N)

## 発動条件

1. ユーザーが明示的に自己改善を要求した場合
2. odin司令塔が品質向上の必要性を検出した場合

## Instructions

### ステップ0: エピソディックメモリの読み込み

過去の改善履歴を読み込み、重複回避と効果測定の基盤を構築する。

1. `.claude/artifacts/odin-episodic-memory.md` の存在を確認する
   - 存在する場合: 全体を読み込み、以下を抽出する
     - 知識索引（有効パターン、アンチパターン、未解決バックログ）
     - 直近エピソードのSelf-Reflection（次回への指針）
     - 直近エピソードの見送り項目
   - 存在しない場合: 初回実行として空のコンテキストで続行する
2. `.claude/artifacts/insights-*.md` を時系列で読み込み、改善関連のエントリを抽出する
   - フィルタ基準:
     - [スキル品質] タグ: 常に取り込む
     - [進め方] タグ: スキル構造への示唆がある場合のみ取り込む
     - [コミュニケーション] タグ: スキルの指示文に関する示唆がある場合のみ取り込む
     - [ツール選択] タグ: スキルのallowed-toolsや手順に関する示唆がある場合のみ取り込む
     - [実装判断] [学び] タグ: 取り込まない
   - 取り込み済みエントリ（エピソードログに記録済み）はスキップする
3. メモリコンテキストを構築する:
   - 実施済み改善リスト（重複チェック用）
   - 失敗/逆効果だった改善リスト（回避用）
   - 前回の見送り項目（再検討候補）
   - insightsから抽出した改善示唆

#### 完了チェックポイント（ステップ0）
- メモリファイルの有無が確認されていること
- insightsファイルの取り込みが完了していること
- メモリコンテキストが構築されていること

### ステップ0.5: 正典・基準参照（Canonical & Baseline Reference）の取得

Anthropicが公式に発信するドキュメントを「正典」、コミュニティの高精度リポジトリを「基準」として参照し、検索結果の評価基準を確立する。
正典を最優先する理由: Claude Codeの開発元であり、最もモデル特性を理解している情報源であるため。
基準リポジトリ: shanraisshan/claude-code-best-practice はClaude Codeのベストプラクティスを高い精度で反映し続けている外部ソースである。

1. 以下のAnthropic公式ソースをWebFetchで取得する（並列実行）:
   - https://docs.anthropic.com/en/docs/claude-code/overview
   - https://docs.anthropic.com/en/docs/claude-code/skills
   - https://docs.anthropic.com/en/docs/claude-code/hooks
   - https://www.anthropic.com/engineering （最新のエンジニアリングブログ一覧）

2. 取得した情報から以下を抽出する:
   - Claude Codeの推奨設定パターン
   - スキル・フック・エージェントの推奨設計
   - 非推奨パターンや廃止された機能
   - モデルの最新能力に関する記述（何がモデル自身で可能か）

3. 既存のartifactsにAnthropicリファレンスキャッシュがある場合（`.claude/artifacts/anthropic-reference-*.md`）:
   - 前回取得日から7日以上経過していれば再取得する
   - 7日以内であればキャッシュを使用する

4. 取得結果を `.claude/artifacts/anthropic-reference-{yyyyMMdd}.md` に保存する

5. コミュニティ基準リポジトリ（shanraisshan/claude-code-best-practice）を取得する:
   - WebFetchで `https://raw.githubusercontent.com/shanraisshan/claude-code-best-practice/main/README.md` を取得する
   - Bashで直近の変更を確認する:
     ```
     gh api repos/shanraisshan/claude-code-best-practice/commits --jq '.[0:5] | .[] | {date: .commit.committer.date, message: .commit.message}'
     ```
   - 重要な変更（ベストプラクティスの追加・修正）がある場合、該当コミットの差分をWebFetchまたはBashで確認する

6. コミュニティ基準から以下を抽出する:
   - Claude Codeのベストプラクティス（CLAUDE.md設計、スキル設計、プロンプト構造）
   - 直近の変更で追加・修正された推奨事項
   - 非推奨となったパターン

7. コミュニティリファレンスキャッシュの管理:
   - `.claude/artifacts/community-reference-*.md` が7日以内に存在すればキャッシュを使用する
   - 7日以上経過していれば再取得する

8. 取得結果を `.claude/artifacts/community-reference-{yyyyMMdd}.md` に保存する

9. 正典（Anthropic公式）と基準（コミュニティリポジトリ）を統合し、ステップ1の検索結果を評価するための基準を確立する

#### 完了チェックポイント（ステップ0.5）
- Anthropic公式ドキュメントが取得されていること
- コミュニティ基準リポジトリが取得されていること
- 推奨パターン・非推奨パターンが抽出されていること
- 両ソースのリファレンスキャッシュが保存されていること
- 検索結果の評価基準が確立されていること

### ステップ1: ベストプラクティスの調査

#### 情報ソース優先度（正典ファーストの原則）

WebSearchの結果は以下の優先度で評価する:

| 優先度 | ソース | 信頼性 | 処理 |
|--------|--------|--------|------|
| 1（正典） | anthropic.com, docs.anthropic.com | 最高 | そのまま採用 |
| 2（基準） | github.com/anthropics/*, shanraisshan/claude-code-best-practice | 高 | そのまま採用 |
| 3（参考） | 2026年以降の技術ブログ・カンファレンス資料 | 中 | 正典・基準と矛盾しなければ採用 |
| 4（非採用） | 2025年以前の情報（正典・基準ソースを除く） | 非採用 | 原則不採用 |

正典・基準との矛盾チェック:
- WebSearchで得た情報がステップ0.5の正典・基準と矛盾する場合、正典・基準を優先する
- 矛盾がある場合はその旨をinsightsファイルに記録する（「WebSearch結果 X は正典/基準 Y と矛盾するため不採用」）

WebSearchで以下を調査する:

1. AIエージェントのプロンプト設計ベストプラクティス
   - 検索例: "Claude Code skill design best practices 2026"
   - 検索例: "AI agent prompt engineering best practices"
   - 検索例: "LLM tool use orchestration patterns"

2. 開発ワークフロー自動化のベストプラクティス
   - 検索例: "AI-driven development workflow automation"
   - 検索例: "agentic software development best practices"

3. メタ認知・自己改善のフレームワーク
   - 検索例: "AI agent self-improvement metacognition"
   - 検索例: "reflexion AI agent self-reflection"

メモリコンテキストの「次回への指針」に特定テーマがある場合、そのテーマを優先的に調査する。

調査結果を以下の観点で整理する:
- スキル設計のアンチパターン（避けるべきこと）
- 効果的なプロンプト構造
- エージェント間の連携パターン
- フィードバックループの設計
- エラーハンドリングのベストプラクティス

#### 完了チェックポイント（ステップ1）
- 3つ以上の情報源からベストプラクティスを収集していること
- 改善に活かせる具体的な知見が得られていること
- メモリの「次回への指針」に関連する調査を含んでいること（指針がある場合）

### ステップ2: メタ認知による自己分析 + Evaluator評価

全odinスキル（dot_claude/skills/odin*/SKILL.md）を読み込み、以下の観点で客観評価する:

評価軸1: 一貫性
- frontmatterの形式は全スキルで統一されているか
- コンテキスト検出パターンは統一されているか
- 完了チェックポイントの書き方は統一されているか
- 成果物の命名規約は統一されているか

評価軸2: 網羅性
- 各スキルのステップに抜け漏れはないか
- エラーハンドリングは十分か
- エッジケース（入力なし、中断、失敗時）への対応はあるか

評価軸3: 品質
- ステップ1のベストプラクティスと照合して不足はないか
- 指示の明確性は十分か（曖昧な表現はないか）
- 具体例（Examples）は十分か

評価軸4: 連携
- スキル間の連携は適切に定義されているか
- odin司令塔からの呼び出しフローに矛盾はないか
- superpowersスキルとの連携指示は正確か

評価軸5: メタ認知
- 「自分（Claude）がこのスキルを実行するとき、何に困るか」を想像する
- 「このスキルの指示に従ったとき、迷う箇所はどこか」を特定する
- 「ユーザーがこのスキルを使ったとき、期待と結果にギャップが出る箇所はどこか」を予測する

評価軸6: Evaluator評価（前回改善の効果測定）
- ステップ0で読み込んだ直近エピソードの改善項目それぞれについて、現時点のスキル品質を観察する
- 以下の3段階で判定する:
  - 有効: 改善がスキル品質に寄与している証拠がある
  - 効果不明: 判定材料が不足している
  - 逆効果: 改善が新たな問題を生んでいる
- 初回実行（前回エピソードなし）の場合はスキップする

#### 完了チェックポイント（ステップ2）
- 全スキルが読み込まれ評価されていること
- 改善点のリストが作成されていること
- 前回エピソードの改善項目に対するEvaluator評価が完了していること（前回がない場合はスキップ）

### ステップ3: 改善計画の作成

改善計画の作成にあたり、メモリベースの重複・回避チェックを行う:

1. 重複チェック: 各改善案をメモリの実施済みリストと照合する
   - 完全一致: その改善を除外し、理由を記録する（「EP-{N}で実施済み」）
   - 類似: 前回の改善を深化させるか別アプローチにするか判断する
2. 回避チェック: アンチパターンリストと照合する
   - 該当する場合: その改善を除外し、理由を記録する（「EP-{N}で逆効果と判定済み」）
3. バックログ再検討: 前回の見送り項目を再検討対象に含める
   - 再提案回数が3回以上の項目は「低優先度固定」として自動的に後回しにする
4. insightsからの改善示唆を改善候補に追加する
   - insightsは実運用知見のため、同等の改善案がWebSearch由来で出た場合はinsights側を優先する
5. 優先度分類でEvaluator結果を反映する:
   - 前回「逆効果」だった系統の改善は優先度を1段階下げる
   - 前回「有効」だった系統の深化改善は優先度を1段階上げる
6. 改善計画をユーザーに提示する。重複排除・回避の経緯も含める

改善点を優先度で分類する:
- 重大: スキルの動作に影響する問題（誤った指示、矛盾、欠落）
- 改善: 品質向上に寄与する改善（より明確な指示、ベストプラクティスの適用）
- 軽微: スタイルの統一、表現の改善

改善計画をユーザーに提示する:
- 改善点の一覧（優先度、対象スキル、改善内容）
- 推定影響範囲
- 重複排除・回避の経緯
- AskUserQuestionで承認を得る

#### 削減監査（Harness Pruning）

Anthropicの原則: "Every component in a harness encodes an assumption about what the model can't do on its own, and those assumptions are worth stress testing."

改善提案の作成時、「追加」だけでなく「削減」も検討する:

1. 各スキル・フック・エージェントについて以下を問う:
   - このコンポーネントが解決している問題は何か？
   - 現在のモデル（正典から取得した最新能力情報）が自力で解決できるか？
   - 削除した場合のリスクは何か？（安全制約は削除不可）

2. 削減候補の分類:
   - 安全制約（ガードレール）: 削除不可。モデルの判断力に関係なく物理的制約として保持
   - ワークフロー定義: モデルが自発的に同じ手順を踏むか検証し、踏む場合は簡素化を検討
   - 品質チェック自動化: 自動化の便益がオーバーヘッドを上回るか評価
   - 情報提供型: モデルが既知の情報を提供するだけのスキルは統合を検討

3. 改善提案に「削減提案」セクションを含める:
   - 削除候補のスキル/フック名
   - 削除の根拠（正典ベース）
   - 削除時のリスク評価（高/中/低）
   - 代替手段（CLAUDE.mdの1行ルール化、他スキルへの統合等）

#### 完了チェックポイント（ステップ3）
- 改善計画がユーザーに承認されていること
- 削減監査が実施されていること

### ステップ4: 改善実行

承認された改善計画に基づいて、Agentツールでサブエージェントに修正を委譲する:

1. 各スキルファイルの修正をサブエージェントに委譲する
   - 修正内容を明確に指示する
   - dot_claude/skills/odin-*/SKILL.md を直接編集する
2. 修正後、全スキルの整合性を確認する
3. 変更をコミットする

#### 完了チェックポイント（ステップ4）
- 全ての承認された改善が適用されていること
- スキルファイルに構文エラーがないこと

### ステップ5: 改善レポート

改善結果を以下の形式で報告する:

```
## 自己改善レポート

改善日: {yyyy-MM-dd}

### 調査したベストプラクティス
- {情報源1}: {得られた知見}
- {情報源2}: {得られた知見}

### 実施した改善
| # | 対象スキル | 改善内容 | 優先度 |
|---|-----------|---------|--------|

### 改善前後の比較
{主要な改善ポイントのbefore/after}

### 次回の改善候補
{今回見送った改善点}
```

出力先: `.claude/artifacts/improvement-{yyyyMMdd-HHmm}.md`

#### 完了チェックポイント（ステップ5）
- レポートが出力されていること

### ステップ6: Reflection記録

今回の改善サイクル全体を振り返り、教訓をエピソディックメモリに永続化する。

1. 新しいエピソードエントリ（EP-{NNN}）を構成する:
   - トリガー情報（手動起動 / insightsトリガー）
   - 実行サマリー（調査源数、改善項目数、対象スキル）
   - Evaluator評価テーブル（前回改善の効果判定）
   - Self-Reflection:
     - 有効だった施策
     - 期待より効果が薄かった施策
     - 想定外の副作用
     - 次回への指針
   - 見送り項目テーブル

2. 知識索引を更新する:
   - 「有効」と判定された改善パターンを「有効な改善パターン」テーブルに追加（既存と重複しない場合）
   - 「逆効果」と判定された改善を「回避すべきアンチパターン」テーブルに追加
   - 新たに見送った項目を「未解決の改善候補」に追加。既存バックログの再提案回数をインクリメント

3. メタデータ（最終更新日時、累計数）を更新する

4. `.claude/artifacts/odin-episodic-memory.md` に書き込む

#### 完了チェックポイント（ステップ6）
- 新エピソードが追記されていること
- 知識索引が更新されていること
- メタデータが最新であること
