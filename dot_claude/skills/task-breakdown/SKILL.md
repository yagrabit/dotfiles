# Task Breakdown

設計ドキュメントをもとにタスクを分解し、依存関係・並列可否・着手順を整理するスキル。
各タスクにはTDDサイクル（テスト作成→実装→リファクタ）を組み込む。

## Instructions

### 完了チェックポイントの原則

各ステップの最後には「完了チェックポイント」を設けている。
チェックポイントに記載された全ての条件を満たさない限り、次のステップに進んではならない。

### ステップ1: 入力ファイルの読み込み

1. `.claude/artifacts/` から以下のファイルを探す:
   - `design-*.md`（必須）
   - `clarified-requirements-*.md`（必須）
   - `test-plan-*.md`（任意: 存在すればテストケースの対応付けに使用）
   - $ARGUMENTSでパスが指定されていればそれを使用
2. 設計ドキュメントの「変更対象ファイル一覧」と各セクションを把握する
3. リスクレベルを確認する

#### 完了チェックポイント（ステップ1）

- design-*.md と clarified-requirements-*.md が読み込まれていること
- 変更の全体像を把握していること

### ステップ2: タスクの洗い出し

設計ドキュメントの各セクションからタスクを抽出する:

1. 型定義・インターフェース → 型定義タスク
2. DBスキーマ → マイグレーションタスク
3. API設計 → 各エンドポイントごとのタスク
4. コンポーネント設計 → 各コンポーネントごとのタスク
5. 状態管理 → store/state定義タスク
6. 統合 → コンポーネント間の結合タスク

タスク粒度の目安:
- 1タスク = 1つのTDDサイクルで完結する単位
- 目安: 1タスクあたり1-3ファイルの変更
- 大きすぎるタスクは分割する

#### 完了チェックポイント（ステップ2）

- 全ての設計セクションがタスクに変換されていること
- 各タスクの粒度が適切であること（1-3ファイル程度）

### ステップ3: 依存関係と着手順の整理

1. タスク間の依存関係を特定する:
   - 型定義は他のタスクの前提になることが多い
   - APIはコンポーネントより先に実装すべき
   - DBマイグレーションは最初に必要
2. 並列実行可能なタスクを識別する:
   - 依存関係がないタスク同士は並列実行可能
   - 異なるファイルを変更するタスクは並列しやすい
3. 着手順を決定する:
   - 基盤（型定義、DB、共通関数）→ ロジック（API、ビジネスロジック）→ UI（コンポーネント）→ 統合
   - 各グループ内では依存関係に従って順序付け

#### 完了チェックポイント（ステップ3）

- 全タスクに着手順の番号が振られていること
- 依存関係が明記されていること
- 並列実行可能なグループが識別されていること

### ステップ4: TDDサイクルの組み込み

各タスクに以下のTDDサイクルを定義する:

1. Red: 何のテストを書くか（テスト対象の振る舞い）
2. Green: テストを通すために何を実装するか
3. Refactor: リファクタリング観点（該当する場合）

test-plan-*.md が存在する場合は、テスト計画のテストケースと各タスクを対応付ける。

#### 完了チェックポイント（ステップ4）

- 全タスクにTDDサイクル（Red/Green/Refactor）が定義されていること

### ステップ5: タスク一覧の出力

1. `.claude/artifacts/` ディレクトリが存在しなければ作成する
2. 以下のフォーマットでタスク一覧を出力する:

出力先: `.claude/artifacts/tasks-{yyyyMMdd-HHmm}.md`

```
# タスク一覧

作成日: {yyyy-MM-dd}
対象設計: {design-*.md のパス}
リスクレベル: {高/中/低}
タスク数: {N}

## 依存関係図

{テキストベースの依存関係図}

例:
T1（型定義）→ T2（API）→ T5（統合）
T1（型定義）→ T3（store）→ T4（UI）→ T5（統合）
T3 と T2 は並列実行可能

## タスク詳細

### T1: {タスク名}

- 着手順: 1
- 規模: 小
- 依存: なし
- 並列: T1完了後、T2とT3は並列可能
- 変更ファイル:
  - {ファイルパス}（{追加/変更}）
- TDDサイクル:
  - Red: {テスト内容}
  - Green: {実装内容}
  - Refactor: {リファクタ観点、該当なしの場合は省略}
- ステータス: 未着手

### T2: {タスク名}
...
```

#### 完了チェックポイント（ステップ5）

- `.claude/artifacts/tasks-*.md` が出力されていること
- 全タスクに着手順・依存関係・TDDサイクル・ステータスが記載されていること

### ステップ6: タスク計画レビュー（ゲート2）

1. タスク一覧の要約をユーザーに提示する:
   - タスク数、推定規模の内訳
   - 依存関係図
   - 並列実行可能なグループ
2. AskUserQuestionで以下の選択肢を提示する:
   - 承認する: 計画に問題なし、実装に進む
   - 修正する: タスクの粒度・順序・内容を修正する（ステップ2に戻る）
   - 中止する: ワークフローを中止する

#### 完了チェックポイント（ステップ6）

- ユーザーが「承認する」を選択していること
- 「修正する」の場合はステップ2に戻って修正が完了していること

## Examples

### 通知機能のタスク分解

ユーザー: 「タスクに分解して」（design-*.mdとclarified-requirements-*.mdがある状態）

```
ステップ1: 入力読み込み
  design-20260320-1530.md → 通知機能の設計
  clarified-requirements-20260320-1500.md → 通知機能の要件
  リスクレベル: 中

ステップ2: タスク洗い出し
  T1: Notification型定義の作成（小）
  T2: DBマイグレーション - notificationsテーブル追加（小）
  T3: 通知取得API - GET /api/notifications（中）
  T4: 通知既読API - PATCH /api/notifications/:id（小）
  T5: NotificationBellコンポーネント（中）
  T6: NotificationListコンポーネント（中）
  T7: Headerへの通知ベル統合（小）

ステップ3: 依存関係整理
  T1 → T2 → T3 → T7
  T1 → T4
  T1 → T5 → T7
  T1 → T6 → T7
  並列: T2完了後、T3/T4/T5/T6は並列実行可能

ステップ4: TDDサイクル組み込み
  T3 Red: GET /api/notifications が通知一覧をJSON返却するテスト
  T3 Green: APIルート実装、Prismaクエリ
  T3 Refactor: エラーハンドリングの共通化

ステップ5: 出力
  → .claude/artifacts/tasks-20260320-1545.md

ステップ6: タスク計画レビュー（ゲート2）
  タスク数: 7、小: 3, 中: 4
  依存関係図を提示
  → ユーザー承認
```
