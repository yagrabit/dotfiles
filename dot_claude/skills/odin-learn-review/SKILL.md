# odin-learn-review

PRレビューを学習機会に変える教育スキル。
odin-talk-reviewの実行結果に教育レイヤーを追加し、コードの「なぜ」を解説、ソクラテス式の質問対話で理解を促進、学んだパターンをフラッシュカード化して間隔反復学習に活用する。

## ブルームのタキソノミー（学習段階）

スキル評価に以下の6段階を使用する:

| レベル | 名称 | プログラミングでの意味 | 例 |
|--------|------|----------------------|-----|
| 1 | Remember（記憶） | 構文・API名を思い出せる | 「useEffectの引数は？」 |
| 2 | Understand（理解） | コードの動作を説明できる | 「このuseEffectが何をしているか説明して」 |
| 3 | Apply（適用） | 仕様をコードに翻訳できる | 「この仕様でカスタムフックを書いて」 |
| 4 | Analyze（分析） | 構造を分解し問題を特定できる | 「このコンポーネントの再レンダリング原因は？」 |
| 5 | Evaluate（評価） | 設計の代替案を比較判断できる | 「Context vs Zustand、この場面ではどちらが適切？」 |
| 6 | Create（創造） | 抽象的な問題の解決策を設計できる | 「この要件のアーキテクチャを設計して」 |

## Instructions

### 完了チェックポイントの原則

各ステップの最後には「完了チェックポイント」を設けている。
チェックポイントに記載された全ての条件を満たさない限り、次のステップに進んではならない。

### 保存場所に関する注意

skillmap-latest.md と flashcards-latest.md はプロジェクト横断の個人学習データである。
必ずユーザーレベルの `~/.claude/artifacts/` に保存すること。
プロジェクトの `.claude/artifacts/` に保存してはならない。

### ステップ 0: コンテキスト検出

$ARGUMENTS を確認し、odinコンテキストJSONが含まれているか判定する。

odinコンテキストJSONの例:
```json
{
  "odin_context": {
    "task": "PRレビューの教育モード",
    "review_target": "pr:123",
    "artifacts": {
      "skillmap": "~/.claude/artifacts/skillmap-latest.md"
    }
  }
}
```

odinコンテキストがある場合: review_targetとskillmapを使用してステップ1に進む。
odinコンテキストがない場合: AskUserQuestionでレビュー対象を確認する。

### ステップ 1: レビュー実行 + 教育ポイント抽出

1-1. Skillツールで `odin-talk-review` を実行し、通常のレビュー結果を取得する。

1-2. レビュー結果を分析し、以下の観点で「教育ポイント」を抽出する:

教育ポイントの抽出基準:
- パターン/アンチパターン: 良い設計パターンまたは改善すべきアンチパターンが含まれている
- 技術的深掘りの余地: 表面的な指摘の裏に、理解すべき概念がある
- 実践的な代替案: より良いアプローチが存在し、比較学習の価値がある
- 横断的な知識: 他の領域（a11y、パフォーマンス、セキュリティ等）に波及する学び

1-3. 各教育ポイントに以下を付与する:
- 対象スキル領域（スキルマップの中区分）
- ブルームレベル（この学びが求める認知レベル）
- 重要度: 高（必ず学ぶべき）/ 中（学ぶと良い）/ 低（参考程度）

#### 完了チェックポイント（ステップ 1）

- odin-talk-reviewのレビュー結果が取得できていること
- 教育ポイントが1つ以上抽出できていること
- 各教育ポイントにスキル領域・ブルームレベル・重要度が付与されていること

### ステップ 2: スキルマップ読み込み

2-1. `~/.claude/artifacts/skillmap-latest.md` を読み込む。
   - ファイルが存在しない場合: Skillツールで `odin-learn` を `init` 引数付きで実行し、スキルマップを初期化する。

2-2. 教育ポイントの対象領域について、ユーザーの現在のブルームレベルを確認する。

2-3. ユーザーのレベルに応じて教育の深さを調整する:
   - ユーザーのレベル < 教育ポイントのレベル: 前提知識の補足から丁寧に解説
   - ユーザーのレベル = 教育ポイントのレベル: 標準的な解説
   - ユーザーのレベル > 教育ポイントのレベル: 簡潔に済ませ、より高度な関連知識を提供

#### 完了チェックポイント（ステップ 2）

- スキルマップが読み込めていること
- 各教育ポイントの深さが調整されていること

### ステップ 3: ソクラテス式教育対話

重要度「高」の教育ポイントから順に、以下のソクラテス式フレームワークで対話する。
一度に扱う教育ポイントは最大3つまで。残りは次回に回す。

3-1. Clarify（明確化）
まずコードの該当箇所を提示し、質問する:
```
このコードについて質問です:

[コードスニペット]

Q: この実装で[対象パターン]が使われています。
   なぜこのアプローチが選ばれた（または選ばれるべきだった）と思いますか？

ヒント: [ユーザーのレベルに応じた適切なヒント]
```

AskUserQuestionで回答を求める。選択肢は:
1. [考えられる理由A]
2. [考えられる理由B]
3. わからない / ヒントがほしい
4. （自由記述で回答）

3-2. Justify（正当化）+ 解説
ユーザーの回答に対してフィードバックする:
- 正解/部分正解の場合: 回答を肯定し、さらに深い理由を補足する
- 不正解の場合: 回答の論理は認めつつ、正しい理由を丁寧に解説する
- 「わからない」の場合: 段階的にヒントを与え、最終的に解説する

解説には必ず以下を含める:
- なぜそのパターンが良い/悪いのか（理由）
- 実際のコードでどう改善できるか（具体例）
- 関連するベストプラクティスやドキュメントへの言及

3-3. Counterexample（反例提示）
「もしこう書いたらどうなるか？」という反例を提示し、理解を深める:
```
では、もしこう書いていたらどうなるでしょう？

[アンチパターンのコード例]

どんな問題が起きると思いますか？
```

3-4. Synthesize（統合）
学んだことを1-2文でまとめ、ユーザーに確認する:
```
まとめ:
[学んだパターン/原則を1-2文で要約]

このパターンは[スキル領域]の[ブルームレベル]レベルの知識です。
スキルマップを更新します。
```

#### 完了チェックポイント（ステップ 3）

- 重要度「高」の教育ポイントについてソクラテス式対話が完了していること
- ユーザーが各教育ポイントの「まとめ」を確認していること

### ステップ 4: フラッシュカード生成

4-1. ステップ3で扱った各教育ポイントについて、フラッシュカードを生成する。

フラッシュカード形式:
```markdown
### カード {連番}
- 領域: {スキルマップの中区分}
- レベル: {ブルームレベル}
- 作成日: {yyyy-MM-dd}
- 次回復習: {作成日 + 1日}
- 復習回数: 0

Q: {質問（ステップ3の質問を簡潔にしたもの）}

A: {回答（ステップ3のまとめを基にした簡潔な回答）}

コード例:
{該当コードの最小限のスニペット}
```

4-2. `~/.claude/artifacts/flashcards-latest.md` に追記する。
   - ファイルが存在しない場合は新規作成する。
   - 既存のカードがあれば末尾に追記する。

#### 完了チェックポイント（ステップ 4）

- 教育ポイントごとにフラッシュカードが生成されていること
- flashcards-latest.md に保存されていること

### ステップ 5: スキルマップ更新

スキルマップ更新の原則:
- スキルマップはソクラテス式対話での回答精度に基づいてのみ更新する
- 学びポイントを表示しただけではスキルマップを更新しない（表示 ≠ 理解）
- ユーザーが対話に参加し、回答の正誤を判定できた場合にのみ更新する

5-1. 学習した領域について、スキルマップの以下を更新する:
- 現在レベル: ソクラテス式対話の回答精度に基づいて判定
  - 全問正解 → 現在レベルを1段階上げることを検討
  - 部分正解 → 現在レベルを維持
  - 不正解/わからない → 現在レベルを維持（ただし「学習中」フラグを付与）
- 最終学習日: 今日の日付
- 次回復習予定: 間隔反復スケジュールに基づいて計算
  - 初回: +1日
  - 2回目: +3日
  - 3回目: +7日
  - 4回目: +14日
  - 5回目以降: +30日
  - 不正解時: 間隔をリセットして+1日

5-2. `~/.claude/artifacts/skillmap-latest.md` を更新する。

#### 完了チェックポイント（ステップ 5）

- スキルマップが更新されていること
- 間隔反復スケジュールが設定されていること

### ステップ 6: 学習サマリー提示

以下の形式でユーザーに報告する:

```
--- 学習サマリー ---

今回のレビューで学んだこと:
1. [{スキル領域}] {学んだパターン/原則の要約}（レベル: {ブルーム}）
2. ...

スキルマップ更新:
- {領域}: {旧レベル} → {新レベル}（次回復習: {日付}）

フラッシュカード: {N}枚追加（累計: {M}枚）

復習が必要なカード: {期限切れのカード数}枚
（「/odin 復習して」で復習セッションを開始できます）
```

#### 完了チェックポイント（ステップ 6）

- 学習サマリーがユーザーに提示されていること
- 復習が必要なカードがある場合、その旨が通知されていること

## Examples

### 例1: Next.jsコンポーネントのPRレビュー

ユーザー: 「PR #42を教育モードでレビューして」

```
ステップ1: odin-talk-reviewでレビュー実行
  → 教育ポイント抽出:
    1. [高] Server Componentでのデータフェッチ（FEアーキテクチャ / Analyze）
    2. [中] aria-label の欠如（アクセシビリティ / Apply）
    3. [低] CSS Grid の使い方（CSS/デザインシステム / Apply）

ステップ2: スキルマップ確認
  → FEアーキテクチャ: Apply → Analyzeへの成長中
  → アクセシビリティ: Remember → もう少し丁寧に解説

ステップ3: ソクラテス式対話（教育ポイント1と2を実施）

  Q: このコンポーネントでは `fetch()` が直接呼ばれています。
     Next.js App Routerでは、なぜServer Componentで直接fetchするのが推奨されるのでしょう？

  ユーザー: 「サーバーサイドで実行されるからAPIルートが不要？」

  正解の要素があります。補足すると:
  1. Server Componentはサーバーで実行されるためクライアントバンドルに含まれない
  2. fetch結果はReactのキャッシュと統合され、同じリクエストは自動的に重複排除される
  3. API Routeを経由すると追加のHTTPホップが発生し、パフォーマンスが劣化する

  もし `'use client'` を付けてクライアントサイドでfetchしていたら？
  → バンドルサイズ増加、ウォーターフォール問題、SEOへの悪影響

  まとめ: Next.js App Routerでは、データフェッチはServer Componentで直接行うのが原則。
  クライアントサイドfetchは「ユーザーインタラクション後のデータ更新」に限定する。

ステップ4: フラッシュカード生成（2枚）
ステップ5: スキルマップ更新
  → FEアーキテクチャ: Apply → Analyze（部分正解で維持、学習中フラグ）
  → アクセシビリティ: Remember → Understand（基礎解説で理解度向上）
ステップ6: 学習サマリー提示
```
