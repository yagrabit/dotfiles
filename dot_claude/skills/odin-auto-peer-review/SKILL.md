# odin-auto-peer-review

ドキュメント向け自動ピアレビュースキル。
Anthropic公式原則: "separating the agent doing the work from the agent judging it proves to be a strong lever" をドキュメントに適用する。

作成者のコンテキスト（なぜそう書いたか、どんな議論があったか）をレビューアに渡さず、ドキュメント内容のみで品質を評価する。これにより確証バイアスを排除した客観的レビューを実現する。

## auto-review（コードレビュー）との違い

- auto-review: do系スキル完了後のコード差分レビュー。code-reviewer/security-reviewer等を使用
- auto-peer-review: think系スキル完了後のドキュメントレビュー。ドキュメント品質に特化した評価軸を使用

## コンテキスト検出

### odinコンテキストがある場合

```json
{
  "odin_context": {
    "task": "設計書のピアレビュー",
    "document_path": ".claude/artifacts/design-20260320-1600.md",
    "document_type": "design",
    "artifacts": { ... }
  }
}
```

- `document_path`: レビュー対象のファイルパス
- `document_type`: ドキュメント種別（design / requirements / adr / test-plan / research）

### odinコンテキストがない場合（ユーザー直接呼び出し）

ステップ1でレビュー対象を検出する。

## 独立評価の原則

- レビューアは必ず新しいAgentツールで起動する（同一コンテキストで実行しない）
- レビューアにはドキュメント内容のみを渡す。作成時の意図、会話履歴、設計判断の経緯を渡さない
- レビュー結果は「ドキュメントの品質」のみで判断する。作成者の意図を忖度しない

## ステップ1: レビュー対象の特定

### odinコンテキストがある場合

`document_path` と `document_type` を使用する。

### odinコンテキストがない場合

1. `.claude/artifacts/` 内の最新ドキュメントを検出する:
   ```bash
   ls -t .claude/artifacts/*.md | head -5
   ```

2. AskUserQuestionでレビュー対象を確認する:
   ```
   どのドキュメントをレビューしますか？

   1. {最新のファイル名} （{種別}）
   2. {2番目に新しいファイル名} （{種別}）
   3. ファイルパスを指定する
   ```

### ドキュメント種別の自動判定

ファイル名プレフィックスから種別を判定する:
- `design-*` → design
- `requirements-*` / `clarified-requirements-*` → requirements
- `adr-*` → adr
- `test-plan-*` → test-plan
- `research-*` → research
- 判定不可 → AskUserQuestionで確認

### 完了チェックポイント（ステップ1）

- レビュー対象のファイルパスが特定されていること
- ドキュメント種別が判定されていること
- ドキュメント内容が読み込み可能であること

## ステップ2: レビュー基準の構築

ドキュメント種別ごとに異なるレビュー基準を適用する。

### 共通レビュー基準（全種別）

- 明確性: 曖昧な表現がないか、誰が読んでも同じ解釈になるか
- 自己完結性: このドキュメントを初めて読む人が、このドキュメント単体で内容を理解できるか。以下を検出する:
  - 前回セッションや更新差分を前提とした記述（「前回の議論を踏まえ」「更新点として」等）
  - 別ドキュメントの議論・結論を暗黙的に前提とした記述（「先の会議で決まった通り」「Confluenceで合意した方針に基づき」等。ただし文書内で参照リンクとともに要約されている場合は可）
  - 作成・修正過程でのやりとりが補足や付記として残っている箇所
  - レビュー指摘を受けて追加した内容が文書の適切な位置に統合されず、末尾に追記されている箇所
- 目的適合性: ドキュメントの種別・目的に照らして、全ての記述がその目的に貢献しているか。以下を検出する:
  - 参考資料から引っ張られた、このドキュメントの目的に無関係な情報（例: 要件定義書に「デザイナーはここを確認すべき」、設計書に「営業チームへの共有方法」等）
  - ドキュメントの読者・用途に合わない粒度やトーンの記述（例: 技術設計書にエンドユーザー向けの操作説明）
  - 元資料のコピーペースト痕跡（書式の不統一、文脈に合わない敬体/常体の混在）
- 完全性: 必要な情報が欠けていないか
- 一貫性: 文書内で矛盾がないか、用語が統一されているか
- 実行可能性: この文書を元に次の作業に移れるか

### 種別固有のレビュー基準

design（設計書）:
- アーキテクチャの妥当性: 選択した構造は要件を満たすか
- トレードオフの検討: 代替案とその評価が記載されているか
- 実装可能性: 具体的にコードに落とし込めるか
- 非機能要件: パフォーマンス、スケーラビリティ、保守性の考慮

requirements（要件定義）:
- テスト可能性: 各要件に対してテストが書けるか
- 優先度: 必須/任意が明確か
- スコープ: 範囲外が明示されているか
- ユーザーストーリー: 誰が何をなぜしたいかが明確か

adr（Architecture Decision Record）:
- 問題の定義: 何を解決するのかが明確か
- 選択肢の網羅性: 十分な代替案が検討されているか
- 決定の根拠: なぜその選択をしたかが論理的か
- 影響の分析: 決定がもたらす影響が評価されているか

test-plan（テスト計画）:
- カバレッジ: 要件に対するテスト網羅率
- エッジケース: 境界値やエラーケースが含まれているか
- 前提条件: テスト実行に必要な環境・データが明記されているか

research（調査レポート）:
- 調査範囲: 十分な幅と深さで調査されているか
- 情報源: 信頼できるソースに基づいているか
- 結論の妥当性: 調査結果から導かれる結論が論理的か

### 完了チェックポイント（ステップ2）

- 共通基準と種別固有基準が特定されていること

## ステップ3: レビュー実行（反復ループ）

### 3a. レビューア起動

以下のレビューアを並列で起動する。

1. ドキュメントレビューア（必須）:
   Agentツールで汎用エージェントを起動する（model: opus）。

   プロンプトに含めるもの:
   - ドキュメント内容（Readツールで読み込むようファイルパスを指定）
   - ステップ2で構築したレビュー基準
   - 出力形式の指定（後述のフィードバック形式）

   プロンプトに含めないもの:
   - 作成者の意図や議論の経緯
   - 他の成果物の参照（レビュー対象のドキュメント内から参照されているものを除く）
   - 「良い点を褒めてください」等の指示

   レビューアへの指示テンプレート:
   ```
   あなたはドキュメントレビューアです。以下のドキュメントをレビューしてください。

   ドキュメント: {ファイルパス}（Readツールで読んでください）
   種別: {document_type}

   レビュー基準:
   {共通基準 + 種別固有基準をここに展開}

   出力形式:
   各指摘を以下の形式で出力してください。

   ## 総合評価: PASS / REVISE / REJECT

   ### 重大な問題（修正必須）
   - F-001 [{基準カテゴリ}] {問題の概要}
     該当箇所: {文書内の具体的な箇所}
     問題: {なぜ問題か}
     改善案: {どう修正すべきか}
     routeToSkill: {差し戻し先。例: think-requirements, think-design, think-plan}

   ### 改善提案（推奨）
   - F-002 [{基準カテゴリ}] {改善内容}
     該当箇所: {文書内の具体的な箇所}
     改善案: {具体的な改善方法}
     routeToSkill: {差し戻し先スキル名}

   ### 情報（参考）
   - [{基準カテゴリ}] {内容}

   Finding ID規約:
   - F-{NNN} 形式（3桁ゼロ埋め、レビューセッション内で連番）で重大・改善の全指摘に付与する（情報レベルは不要）
   - 反復ループでFinding IDによる解決追跡を行う
   - routeToSkillの判定基準はexecution-loop.md § 5-3.7の正本を参照する。ドキュメントレビューで主に使われる値:
     - 調査の不足 → think-research（注: auto-peer-reviewの自動発動対象外だが、route先としては有効）
     - 要件の欠陥 → think-requirements
     - 設計の問題 → think-design
     - タスク分解の問題 → think-plan
     - 自己完結性・目的適合性の問題 → 当該ドキュメントを生成したスキル（元の種別に対応するthink系スキル）

   注意:
   - 根拠のない賛辞は禁止。問題点と改善案に集中してください
   - 「PASS」は重大な問題が0件の場合のみ
   - 「REVISE」は重大な問題が1件以上だが構造的な書き直しは不要な場合
   - 「REJECT」は根本的な再考が必要な場合
   ```

2. Codexレビューア（任意）:
   Codexプラグインが利用可能な場合のみ実行する。

   Agentツールで subagent_type に "codex:codex-rescue" を指定して起動する。
   プロンプト:
   ```
   以下のドキュメントをレビューしてください。構造的・論理的な問題のみ指摘してください。
   誤字脱字、表現の好み、フォーマットの細かい指摘は不要です。
   重大な問題と改善提案のみ、各3件以内で簡潔に報告してください。

   ドキュメント: {ファイルパス}
   ```
   Codexからの指摘は「重大」「改善」レベルのみ採用し、「情報」レベルは破棄する。

### 3b. フィードバック統合

全レビューアの結果を統合する:
- 重複する指摘は統合し、指摘元を明記する
- Codexの指摘は高レベルのもののみ残す（細かい指摘はフィルタ）
- 総合評価は最も厳しい評価を採用する（PASSとREVISEならREVISE）

### 3c. 判定と分岐

- PASS → ステップ4（完了報告）へ
- REVISE → ステップ3dへ（修正→再レビュー）
- REJECT → ユーザーにエスカレーション。根本的な書き直しが必要な旨を報告する

### 3d. 修正実行

フィードバックの「重大な問題」と「改善提案」をもとに、ドキュメントを修正する。

修正方法:
- odinコンテキストがある場合: odinにフィードバックを返却し、元のthink系スキルで修正を実行する
- ユーザー直接呼び出しの場合: Edit/Writeツールで直接修正する

修正後、ステップ3aに戻り、修正版を新しいレビューアで再評価する。

### 3e. ループ上限

- 最大3回まで反復する
- 3回でPASSにならない場合、ユーザーにエスカレーション
  - これまでのレビュー結果と修正履歴を提示する
  - 続行/中止/手動修正の判断をユーザーに委ねる

### 反復ログ

各反復のサイクルを記録する:
```
ピアレビュー反復ログ:
- Cycle 1: REVISE → 重大2件・改善3件 → 修正実施
- Cycle 2: REVISE → 重大0件・改善1件 → 修正実施
- Cycle 3: PASS → ループ終了
```

### 完了チェックポイント（ステップ3）

- レビュー結果がPASSであること、またはループ上限に達してユーザーに報告済みであること
- 反復ログが記録されていること

## ステップ4: 完了報告

レビュー結果と修正履歴をまとめて報告する:

```
## ドキュメントピアレビュー結果

対象: {ファイル名}（{種別}）
反復回数: {N}回
最終評価: PASS

### レビュー経緯
- Cycle 1: {概要}
- Cycle 2: {概要}

### 修正サマリー
- {修正した主要な項目を箇条書き}

### 残存する注意点
- {PASSしたが注意が必要な点があれば記載}
```

### 完了チェックポイント（ステップ4）

- レビュー結果サマリーが出力されていること
- 反復ログが含まれていること

## odinからの自動発動条件

odinの実行ループ（Phase 5）で以下のタイミングで自動発動する:

- think-design 完了後: design-*.md に対して実行
- think-requirements 完了後: requirements-*.md / clarified-requirements-*.md に対して実行
- think-plan 完了後: plan-*.md / tasks-*.md に対して実行（タスク分解の妥当性確認）

以下のthink系スキルでは自動発動しない（情報収集フェーズのため品質ゲートは不要）:
- think-research: 調査の網羅性はodinが判断する
- think-investigate: 調査結果は次のアクションで検証される
- think-analyze: 分析結果はthink-designへの入力として検証される

## Examples

### Example 1: odinから自動呼び出し（設計書レビュー）

```
ステップ1: レビュー対象特定
  odinコンテキストから document_path = ".claude/artifacts/design-20260320-1600.md"
  document_type = "design"

ステップ2: レビュー基準構築
  共通基準（明確性、完全性、一貫性、実行可能性）
  + 設計書基準（アーキテクチャ妥当性、トレードオフ、実装可能性、非機能要件）

ステップ3: レビュー実行
  Cycle 1:
    レビューア起動 → ドキュメント内容のみ渡す
    結果: REVISE（重大1件: トレードオフの検討が不足、改善2件）
    修正: odin-think-design に差し戻して修正
  Cycle 2:
    新しいレビューア起動 → 修正版を渡す
    結果: PASS（改善1件のみ、重大0件）

ステップ4: 完了報告
  反復2回でPASS。修正サマリーを出力
```

### Example 2: ユーザー直接呼び出し

ユーザー: 「この要件定義書をピアレビューして」

```
ステップ1: レビュー対象特定
  .claude/artifacts/ 内の最新 requirements-*.md を検出
  AskUserQuestionで確認 → ユーザーが対象を選択

ステップ2: レビュー基準構築
  共通基準 + 要件定義基準（テスト可能性、優先度、スコープ、ユーザーストーリー）

ステップ3: レビュー実行
  Cycle 1:
    レビューア → REVISE（テスト可能性が低い要件が3つ、スコープ外の明示なし）
    Editツールで直接修正
  Cycle 2:
    レビューア → PASS

ステップ4: 完了報告
```
