### 5-3.5. Generator-Evaluator反復ループ

Anthropic公式原則: "separating the agent doing the work from the agent judging it proves to be a strong lever"

do-implement / do-refactor 完了後、以下の反復ループを実行する:

#### スプリント契約（完了基準の事前合意）

> "sprint contract mechanism - generator and evaluator agree on 'definition of done' before implementation"

Phase 4（タスク分解）の時点で、各タスクの完了基準を具体的に定義する。
完了基準は以下を含む:

- 機能要件: 何ができるか（例: 「ユーザーがフォームを送信できる」「APIレスポンスが返る」）
- 非機能要件: パフォーマンス、アクセシビリティ、エラーハンドリング等の品質基準
- フロントエンドプロジェクトの変更の場合は加えて: ブラウザでの動作確認、コンソールエラーなし、影響範囲のリグレッションなし
- UI系タスクの場合は更に加えて: レスポンシブ対応、デザイントークン準拠

評価エージェントはこの完了基準を判定軸として使用する。
完了基準が未定義のままループを開始しない。

#### 発動条件

- 変更行数が100行以上の場合（小規模変更では不要）
- UI/デザイン系タスクの場合（行数に関係なく常に発動）
- フロントエンドプロジェクトでの変更の場合（行数に関係なく常に発動）

#### フロントエンドプロジェクト判定（正本）

他ファイル（auto-verify等）はこのセクションを参照すること。

判定条件（いずれかに該当）:
- package.jsonに react / next / vue / nuxt / svelte / angular / remix / gatsby / astro / solid / qwik が含まれる
- src/app/, src/pages/, app/, pages/ 等のフロントエンド規約ディレクトリが存在する
- .tsx / .jsx / .vue / .svelte ファイルが変更に含まれる

ブラウザ検証除外条件（該当する場合はテスト + ビルド + 型チェックのみで検証する）:
- UIを持たないパッケージ（shared library, CLI, SDK等）
- サーバーサイドのみの変更（API route, middleware, DB操作等でUIに影響しない）
- ビルド設定・型定義・lint設定のみの変更
- コンポーネントライブラリでアプリ起動対象がない場合

除外条件に該当しないフロントエンドプロジェクトの変更では、UI実装に限らず（バグ修正、Locale対応、API連携、ルーティング変更等でも）ビヘイビア検証を実施する。人間のエンジニアが修正後にブラウザで動作確認するのと同じ考え方。

#### ループ手順

1. 評価（Evaluate）:

   評価は「コードレビュー」と「ビヘイビア検証」の2段階で行う。

   コードレビュー:
   - Agentツールで新しいエージェントを起動（別コンテキスト）し、変更差分を渡す
   - エージェントには以下を指示:
     - 変更がスプリント契約（Phase 4で定義）の完了基準の各項目を満たしているか判定
     - 品質上の問題点を具体的に指摘（ファイル:行番号、問題の説明、改善案）
   - 評価エージェントはcode-reviewerサブエージェントタイプを使用する

   ビヘイビア検証:

   > "Evaluator tested the app by navigating through it using Playwright MCP"

   品質担保テスト設計（ビヘイビア検証の前に必ず実施）:
   - 「この変更が正しいことを証明するには何を確認すべきか？」を明文化する
   - 修正対象の直接的な動作確認項目を列挙する
   - 影響範囲の確認項目を列挙する（変更したモジュールを参照している画面・機能）
   - バグ修正の場合: 再現手順の確認 + リグレッションがないことの確認
   - 確認項目ごとに「テストコードで検証」「ブラウザで検証」「両方」を判定する

   テストコードによる検証:
   - 上記で「テストコードで検証」と判定した項目について、テストケースが存在するか確認する
   - 不足しているテストケースがある場合はREFINE判定に含め、修正フェーズでGeneratorがodin-do-testに委譲してテストを追加する（Evaluatorは生成作業を行わない）
   - 既存テストを全て実行して通過を確認する

   ブラウザによる検証（フロントエンドプロジェクトの場合は必須。除外条件該当時はスキップ）:
   - ToolSearchで "chrome" または "playwright" を検索し、MCPツールの利用可否を判定する
   - 利用可能な場合:
     - dev server起動: package.jsonのscriptsから `dev` / `start` を検出して実行する。ポートはstdout出力または既定ポート（3000, 5173等）から特定する。既に起動済みの場合はそのまま使用する
     - 対象画面の特定: ルート変更ならそのrouteを使用、page/component変更ならimport graphから親pageを探索、特定不能な場合はトップページで確認する
     - 修正対象の動作を確認する（画面遷移、インタラクション、表示内容等）
     - 影響範囲の画面もナビゲーションして、リグレッションがないことを確認する
     - コンソールエラーの有無を確認する
     - レスポンシブ表示、デザイントークン準拠（UI系タスクの場合）
   - 利用不可の場合: テスト実行結果とビルド成功をもって代替検証とする

   総合評価: コードレビューとビヘイビア検証の両結果を統合して判定する。
   - PASS（問題なし） / REFINE（改善推奨） / REJECT（要修正）

2. 判定:
   - PASS → ループ終了、次のWaveへ
   - REFINE → 改善推奨。ユーザー確認なしで自動的に修正を実行する（自律進行の原則）。ただし以下の場合はユーザーにスコープを確認する: (1) 変更が50行超または3ファイル超 (2) 公開APIシグネチャの変更 (3) DBマイグレーション・スキーマ変更 (4) 認証・認可ロジックの変更
   - REJECT → 修正必須。ステップ3（修正）へ

3. 修正（Refine）:
   - 評価結果の指摘事項をodin-do-implementに渡し、修正を実行
   - 修正後、再度ステップ1（評価）へ

4. ループ上限:
   - 最大3回まで反復する
   - 3回でPASSにならない場合、ユーザーにエスカレーション

#### 反復ログ

各反復のサイクルを記録する:
- Cycle 1: 評価結果（REFINE）→ 指摘3件 → 修正実施
- Cycle 2: 評価結果（PASS）→ ループ終了
