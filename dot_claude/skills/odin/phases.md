# odin Phase詳細

## Phase 1: 初期入力分析 {#phase-1}

### 1-1. テキストからの自動抽出

入力テキスト（自然言語、音声文字起こし、箇条書き、URL等）から以下を識別する:

- 目的: ユーザーが最終的に達成したいこと
- 制約: 技術制約、期限、品質要件、スコープ制限
- 既知情報: ユーザーが既に持っている情報（ファイルパス、URL、エラーメッセージ等）
- 暗黙の前提: 明示されていないが推論できる前提（プロジェクト種別、使用技術等）
- 不足情報: 目的達成に必要だが入力に含まれていない情報

音声文字起こし特有の処理:
- 言い直し・繰り返しの正規化
- フィラー（「えーと」「あのー」等）の除去
- 同音異義語の文脈判定（例: 「コンポーネント」と「コンポネント」の統一）

### 1-2. コンテキスト自動補完

以下のコンテキストを並列で自動収集する:

収集1: プロジェクト構造（Exploreエージェントに委譲）
- プロジェクトルートのファイル構成
- package.json / Cargo.toml 等のプロジェクト定義
- フレームワーク・言語・ツールチェーンの判定
- ディレクトリ構成の概要

収集2: 既存成果物の確認
- `.claude/artifacts/` 内の全ファイルを列挙し、最新のものを把握する
- 特に最新のresearch-*.md, requirements-*.md, design-*.md, plan-*.mdの有無を確認する
- 前セッションの成果物があれば、その内容を踏まえて継続判断する

収集3: 作業文脈の推測
- `git log --oneline -10` で直近の作業を推測する
- `git branch --show-current` で現在のブランチを確認する
- `git status --short` で未コミットの変更を確認する

収集4: ベストプラクティス調査（WebSearch）
- ユーザーの要望に関連する技術領域のベストプラクティスをWebSearchで調査する
- 例: 「通知機能」→ リアルタイム通知のベストプラクティス、UIパターン
- 例: 「認証」→ 最新の認証ベストプラクティス、OAuthフロー
- 例: 「パフォーマンス改善」→ 該当フレームワークの最新最適化手法
- 調査結果はPhase 4のタスク分解で活用する
- 調査不要な場合（品質チェック等の定型作業）はスキップする

#### 収集5: odin-boardタスクID検出

入力テキストの末尾に `odin-board-task: {id}` マーカーがあるか確認する。
存在する場合:
- `board_task_id` にIDを設定
- `artifacts_dir` を `~/.local/share/odin-board/docs/{board_task_id}` に設定
- マーカー行を入力テキストから除去（以降の処理に影響しないよう）

存在しない場合:
- `board_task_id` は空文字列
- `artifacts_dir` は `.claude/artifacts`（従来通り）

### 完了チェックポイント（Phase 1）

- 目的が1文で要約できること
- 制約が列挙されていること（0件でもOK）
- プロジェクトの技術スタックが把握できていること
- 既存成果物の有無が確認できていること
- 不足情報が明確になっていること

---

## Phase 2: 深掘りヒアリング {#phase-2}

### 2-1. スキップ判定

Phase 1で不足情報が0件の場合、このPhaseをスキップしてPhase 3に直行する。

### 2-2. 質問の設計

不足情報がある場合、以下の原則で質問を設計する:

- 質問は最小限に絞る（1〜3問を目安）
- 選択式を優先する（自由回答は最後の手段）
- 選択肢は2〜4個に絞り、各選択肢に日本語で明確な説明を付ける
- 複数の不足情報を1つの質問で解消できる場合はまとめる
- 「わからない/おまかせ」の選択肢を含める（odinが最適判断する旨を伝える）

### 2-3. AskUserQuestionで確認

不足情報をAskUserQuestionで1つずつ確認する。回答を記録する。

### 完了チェックポイント（Phase 2）

- 全ての不足情報が解消されていること（または「おまかせ」でodinに委任されていること）
- ユーザーの回答が記録されていること

---

## Phase 3: 理解要約（自動進行） {#phase-3}

### 3-1. 要約の構成

Phase 1-2の結果を以下の形式で要約する:

```
こう理解しました:

[目的]
{ユーザーが達成したいこと。1〜2文}

[成果物]
{最終的に得られるもの。箇条書き}

[制約]
{守るべき条件。箇条書き}

[アプローチ概要]
{どのような流れで進めるか。3〜5ステップ}

[使用するodinスキル]
{主に使用するスキルの一覧}
```

### 3-2. 自動進行

要約をユーザーに出力し、承認を待たずにPhase 4へ自動的に進行する。

ユーザーは要約を見て問題があればいつでも介入できる。odinは中断されない限り進行を続ける。設計・計画の承認はユーザー確認不要（鉄則4参照）。その代わり、think系成果物にはauto-peer-review → codex:rescue、do系成果物にはtalk-review → codex:rescueの必須レビューチェーンで品質を担保する（正本: execution-loop.md § 5-3）。

### 完了チェックポイント（Phase 3）

- 理解要約がユーザーに提示されていること

---

## Phase 4: タスク分解・計画 {#phase-4}

### 4-1. タスクリスト生成

Phase 1-3で確定した目的・制約・アプローチに基づき、タスクリストを生成する。

タスク分解の原則:
- 1タスク = 1つの配下スキルで完結する作業単位
- タスク名は具体的な動詞で始める（「調査する」「設計する」「実装する」等）
- 各タスクに完了基準を明記する

### 4-2. odinスキルへのマッピング

各タスクに最適なodinスキルを割り当てる。

マッピングルール:
- 調査・分析系 → think系スキル
- 実装・テスト・コミット・PR系 → do系スキル
- ヒアリング・提案・レビュー・説明系 → talk系スキル
- 品質チェック・レビュー・検証 → auto系スキル（PostToolUseフックが自動リマインド。条件はexecution-loop § 5-3参照）
- UI/Webデザイン系 → design系（dissect/generate/knowledge）+ ux系（odin-ux-design/banner/slides）

マッピングできないタスクがある場合:
- Skillツールで `odin-auto-evolve` を実行し、不足スキルを生成する
- 生成後、そのスキルをタスクに割り当てる

### 4-3. 依存関係・並列可否の判定

タスク間の依存関係を分析する:

- 必須依存: タスクAの出力がタスクBの入力に必要
- 推奨依存: 先にやった方が効率的
- 独立: 並列実行可能

クリティカルパスを特定し、全体の所要時間を見積もる。

### 4-4. Wave形式の計画提示

タスクをWave（並列実行グループ）に整理して提示する:

```
実行計画:

Wave 1（並列可能）:
  T-01: [odin-think-research] コードベース調査
  T-02: [odin-talk-clarify] 要件の曖昧点解消

Wave 2（Wave 1完了後）:
  T-03: [odin-think-design] 設計ドキュメント作成

Wave 3（Wave 2完了後、並列可能）:
  T-04: [odin-do-implement] 機能A実装
  T-05: [odin-do-implement] 機能B実装
  （各タスクにauto-quality + simplify + auto-verify必須チェーン）

Wave 4（全do系タスク完了後）:
  T-06: [odin-talk-review] 品質・セキュリティレビュー
  T-07: [codex:rescue] Codex独立レビュー

Wave 5:
  T-08: [odin-do-commit] コミット
  T-09: [odin-do-pr] PR作成

見積もり: 逐次 {N}分 / 並列 {N}分
```

### 4-5. 自動進行（レビューゲート）

計画をユーザーに出力し、承認を待たずに自動的にPhase 5へ進行する。

ユーザー承認ゲートの代わりに、以下のレビューゲートで品質を担保する:
- think系スキル（research, requirements, design, plan, investigate, analyze）の成果物は、完了後にauto-peer-review → codex:rescue（独立レビュー）を必ず実施する（正本: execution-loop.md § 5-3）
- do系スキル完了後は、auto-quality → simplify → auto-verify（各タスク）、全do完了後にtalk-review → codex:rescue → auto-verify（最終検証）の必須チェーンを実行する（正本: execution-loop.md § 5-3）
- レビューで重大な問題（設計の根本的矛盾、要件との不整合等）が検出された場合は、Findingベースのフェーズルーティング（§ 5-3.7）で自動差し戻しする

ユーザーは計画を見て問題があればいつでも介入できる。odinは中断されない限り進行を続ける。

### 完了チェックポイント（Phase 4）

- 全タスクにodinスキルが割り当てられていること
- マッピング不可のタスクが0件であること（auto-evolveで解消済み）
- do系タスクを含む計画の場合、既存成果物のコヒーレンスが確認されていること（Phase 5進入時に `odin-auto-coherence validate` で検証）

## Phase 6: 完了報告 {#phase-6}

### 6-1. 最終検証

Skillツールで `odin-auto-verify` を実行し、全体の成果を最終検証する。
superpowers:verification-before-completion の原則に従い、証拠を収集してから完了を主張する。

### 6-1.5. 知見の自動保存（バックグラウンド・まとめ保存）

§5-X.5（随時保存）でカバーされなかった成果物を対象に、セッション完了時のまとめingestを行う。
§5-X.5との二重ingestは `_log.md` の重複チェックで防止する。

発動条件: 使用スキルに think-design / think-investigate / talk-propose が含まれる場合、
またはADR（adr-*.md）が生成された場合。

- ユーザーへの確認は不要。作業をブロックしない
- 対象: セッション中に生成されたthink系成果物のうち、§5-X.5で未ingestのもの
- 完了したら「知見をknowledgeに保存しました」と通知のみ

スキップ条件:
- do系スキルのみの計画（実装・テスト・コミットのみ）
- `~/.claude/knowledge/_log.md` で全成果物のingestが記録済み

### 6-2. 成果物サマリーの提示

以下の形式でユーザーに報告する:

```
完了しました。

[成果物]
- {成果物1のパスまたはURL}: {説明}
- {成果物2のパスまたはURL}: {説明}
- ...

[実行したスキル]
1. odin-think-research → research-YYYYMMDD-HHMM.md
2. odin-think-design → design-YYYYMMDD-HHMM.md
3. odin-do-implement → src/components/... (5ファイル)
4. odin-auto-quality → 全チェック PASS
5. odin-do-commit → feat: 通知機能を追加
6. odin-do-pr → PR #123

[検証結果]
- テスト: {N}/{N} passed
- ビルド: pass
- lint: pass

[次のアクション推奨]
- {次にやるべきことの提案。PRレビュー待ち、E2Eテスト追加、等}

[振り返りサマリー]
- 本セッションの気づき: {件数}
- 主な学び: {最も重要な1-2個の気づき}
- 改善提案: {odin-auto-improveへの入力として有用な気づきがあれば記載}
```

### 完了チェックポイント（Phase 6）

- 最終検証がPASSしていること
- 成果物サマリーがユーザーに提示されていること
- 次のアクション推奨が含まれていること
