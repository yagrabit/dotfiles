# Next.js バンドル分析

Next.jsプロジェクトのバンドルを分析し、改善提案を日本語でチャット上に出力する。
ファイルは作成せず、テキスト出力のみ行う。

## 1. パッケージマネージャの判定

プロジェクトルートのファイルを確認し、使用するパッケージマネージャを決定する。

- `package-lock.json` が存在する場合: npm を使用（コマンド実行は `npx`）
- それ以外: pnpm を使用（コマンド実行は `pnpx`）

以降の手順では `npx` / `pnpx` を判定結果に応じて読み替える。

## 2. Next.jsプロジェクトの確認

`package.json` を読み込み、以下を確認する。

- `dependencies` または `devDependencies` に `next` が含まれていること
- `next` のバージョンを記録する（`next experimental-analyze` の利用可否判定に使う）

`next` が見つからない場合は「Next.jsプロジェクトではないため分析できません」と報告して終了する。

## 3. ビルド実行と出力キャプチャ

Next.jsのバージョンに応じてビルドコマンドを切り替える。

- Next.js 16.1 未満（webpackビルド）:

```bash
npx next build 2>&1
```

- Next.js 16.1 以上（Turbopackビルド対応）:

```bash
TURBOPACK_STATS=1 npx next build 2>&1
```

TURBOPACK_STATS 環境変数は Turbopack でのビルド時にのみ効果がある。Next.js 15.x 以下は webpack ビルドのため、この環境変数を設定しても webpack-stats.json は生成されない。

ビルドの標準出力をキャプチャし、ルート別サイズテーブル（Route, Size, First Load JS）をパースする。このビルド出力が主要なデータソースとなる。

## 4. ビルド成果物の分析

ビルド完了後、以下のファイルから詳細情報を収集する。

### チャンクファイルの実サイズ確認

`.next/static/chunks/` 配下のチャンクファイルサイズを `ls -lhS` で取得し、大きいファイルから順に確認する。

### ルートとチャンクのマッピング

- `.next/app-build-manifest.json` を読み込み、App Router の各ルートが依存するチャンクファイルを把握する
- `.next/build-manifest.json` を読み込み、Pages Router のページと共有チャンクの対応を確認する

これらのマニフェストにより、特定ルートのサイズが大きい原因（どのチャンクが影響しているか）を逆引きで特定できる。

### webpack-stats.json（Next.js 16.1以上のみ）

Next.js 16.1 以上でビルドした場合、`.next/server/webpack-stats.json` が生成されることがある。存在する場合は読み込み、以下の情報を抽出する。

- 各チャンクの名前とサイズ
- チャンクに含まれるモジュール一覧とそれぞれのサイズ
- モジュール間の依存関係

ファイルが大きい場合は、サイズの大きいチャンク・モジュールを優先的に確認する。

## 5. next experimental-analyzeの実行（条件付き）

`next experimental-analyze` は Next.js 16.1 以降で利用可能な機能。
ステップ2で記録した Next.js のバージョンが 16.1 未満の場合はこのステップをスキップする旨をユーザーに伝える。

バージョンが対応している場合、以下を実行する。

```bash
npx next experimental-analyze --output
```

`.next/diagnostics/analyze/` に出力されたファイルを読み込み、補助的な分析情報として活用する。

コマンドが失敗した場合（未対応バージョン、機能が変更された等）は、エラー内容を記録してスキップし、他の情報のみで分析を続行する。

## 6. ソースコード分析

ビルド成果物の分析に加え、ソースコードの静的分析を行う。

### `'use client'` ディレクティブの使用状況

プロジェクト内の `'use client'` ディレクティブを grep で検索し、Client Components の分布を把握する。Client Components が多すぎる場合や、本来 Server Component で済むファイルに付与されている場合を検出する。

### 大きなライブラリのバレルインポート

以下のようなバレルインポート（トップレベルからの一括インポート）を検出する。

- `import ... from 'lodash'`（`lodash-es` や `lodash/pick` 等の個別インポートを推奨）
- `import ... from 'moment'`（`date-fns` や `dayjs` への移行を推奨）
- `import ... from '@mui/icons-material'`（`@mui/icons-material/IconName` の個別インポートを推奨）
- その他、ツリーシェイキングを阻害するバレルインポートパターン

### `next/dynamic` の使用有無

`next/dynamic` による動的インポートの使用箇所を確認する。使用されていない場合、コード分割の改善余地があると判断する。

## 7. 総合分析

ステップ3のビルド出力（ルート別サイズテーブル）、ステップ4のビルド成果物（チャンクファイルサイズ、マニフェスト）、ステップ5のexperimental-analyze結果（利用可能な場合）、ステップ6のソースコード分析結果を統合し、以下の観点で分析する。

### バンドルサイズの概要

- ルート別（ページ・レイアウト単位）のバンドルサイズ
- チャンク別のサイズと、共有チャンクの割合
- First Load JS のサイズと目標値（200KB以下が理想）との比較

### 大きいモジュールの特定とサイズ削減提案

- サイズ上位のモジュールを一覧化する
- 軽量な代替ライブラリがあれば提案する（例: moment.js → date-fns / dayjs）
- 部分インポートで削減できるものを特定する（例: `import { pick } from 'lodash-es'`）

### 重複依存関係の検出

- 異なるバージョンで重複している依存パッケージを特定する
- 複数チャンクに同じモジュールが含まれている場合を指摘する

### ツリーシェイキングの改善余地

- barrel export（index.ts からの re-export）が原因で不要なコードが含まれていないか確認する
- side effects の設定が適切か確認する
- CommonJS モジュールがツリーシェイキングを阻害していないか確認する

### コード分割（dynamic import）の提案

- 初回ロードに不要なコンポーネントを特定する
- `next/dynamic` による遅延読み込みの候補を提案する
- ルートごとのコード分割が適切か確認する

### CSS最適化の余地

- 未使用CSSの検出（Tailwind CSSの場合はpurge設定の確認）
- CSSファイルのサイズと分割状況
- CSS-in-JSライブラリのランタイムコストの確認

### Server Components / Client Componentsの境界の最適化提案

- `'use client'` ディレクティブの配置が適切か確認する
- Client Components にサーバー側で処理できるロジックが含まれていないか確認する
- Client Components のバンドルサイズが大きい場合、境界の見直しを提案する

## 8. 出力

分析結果を以下のルールで日本語でチャット上に出力する。

### 出力ルール

- 日本語で出力する
- 太字（`**text**`）は使わない
- マークダウンの見出し（`##`）、箇条書き（`-`）、コードブロックを使用する
- ファイルは作成しない（チャット上にテキスト出力のみ）

### 出力構造

```
# バンドル分析結果

## 概要
- プロジェクト情報（Next.jsバージョン、パッケージマネージャ等）
- ビルドの全体サイズ
- First Load JS のサイズ

## ルート別サイズ
- 各ルートのサイズ一覧（大きい順）

## 大きいモジュール
- サイズ上位のモジュールと削減提案

## 重複依存関係
- 検出された重複（なければ「検出なし」）

## ツリーシェイキング
- 改善余地がある箇所

## コード分割の提案
- dynamic importの候補

## CSS最適化
- 改善余地がある箇所

## Server / Client Components の境界
- 最適化提案

## 改善アクション一覧
- 優先度順に具体的なアクションを箇条書きで整理
- 各アクションに期待される効果（サイズ削減の見積もり）を記載
```

改善アクションは影響度の大きいものから順に並べ、すぐ実行できる具体的な手順を含める。
