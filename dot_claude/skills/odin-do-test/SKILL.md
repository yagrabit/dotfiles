# odin-do-test

テスト計画・実装・実行スキル。odin司令塔のdoフェーズで使用する。
設計ドキュメント・要件定義書・既存のテスト計画を入力として、テストシナリオの洗い出し・テスト実装・実行・結果レポートまでを担う。

## ガードレール（安全制御）

このスキルは以下の安全制御を遵守する:

- git push / git commit は実行しない（それぞれodin-do-pr / odin-do-commitスキルの責務）
- `git reset --hard` / `git checkout .` / `git clean -f` 等の破壊的操作を実行しない
- テストファイルの削除や大幅な変更が必要な場合は、Editツールで明示的に修正する

## Instructions

### 完了チェックポイントの原則

各ステップの最後には「完了チェックポイント」を設けている。
チェックポイントに記載された全ての条件を満たさない限り、次のステップに進んではならない。

### コンテキスト検出

$ARGUMENTS を確認し、odinコンテキストJSONが含まれているか判定する。

odinコンテキストJSONの例:
```json
{
  "odin_context": {
    "task": "通知機能のテスト",
    "artifacts": {
      "design": ".claude/artifacts/design-20260320-1600.md",
      "requirements": ".claude/artifacts/requirements-20260320-1500.md",
      "test_plan": ".claude/artifacts/test-plan-20260320-1800.md"
    }
  }
}
```

- odinコンテキストがある場合: artifactsから入力ファイルのパスを取得する
- odinコンテキストがない場合: 成果物管理ルールに従い入力ファイルを探索する

### 成果物管理

入力ファイルの探索順序:
1. $ARGUMENTSにファイルパスが明示指定されていればそれを使う
2. odinコンテキストのartifactsにパスがあればそれを使う
3. `.claude/artifacts/` 内の最新の `design-*.md`, `requirements-*.md`, `test-plan-*.md` を使う
4. 該当ファイルが見つからない場合はAskUserQuestionでユーザーに確認する

出力ファイル: `.claude/artifacts/test-plan-{yyyyMMdd-HHmm}.md`

### ステップ1: テスト対象の確認

1. 入力ファイルを読み込む（設計・要件・既存テスト計画）
2. テストの種別を特定する:

| 種別 | 対象 | 目的 |
|------|------|------|
| ユニットテスト | 個別の関数・コンポーネント | 単体の振る舞いを検証 |
| 統合テスト | モジュール間の連携 | 複数の部品の協調動作を検証 |
| E2Eテスト | ユーザーフロー全体 | エンドツーエンドの動作を検証 |

3. テスト対象のファイル・コンポーネント・APIエンドポイントを列挙する
4. AskUserQuestionで確認が必要な場合（テスト種別・優先度）は確認する

#### 完了チェックポイント（ステップ1）

- テスト対象の機能・コンポーネントが特定されていること
- テスト種別（ユニット/統合/E2E）が決定されていること
- テスト対象ファイルの一覧が把握できていること

### ステップ2: テストシナリオの洗い出し

各テスト対象について、以下の観点でシナリオを洗い出す:

正常系（Happy Path）:
- 期待した入力に対して期待した出力が返ること
- 標準的なユーザー操作フローが正しく動作すること

異常系（Sad Path）:
- 無効な入力に対してエラーが正しく返ること
- 必須フィールドが欠けた場合の挙動
- 認証・認可エラーの挙動

境界値（Boundary Values）:
- 最小値・最大値・ちょうどの値
- 空文字・null・undefined
- 文字列の長さ制限の境界

エッジケース（Edge Cases）:
- 同時実行・競合状態
- ネットワークエラー・タイムアウト
- 大量データ・極端なケース

モック・スタブ関連:
- 複数のカスタムフック呼び出しがある場合のモック設計
- 片方のクエリだけエラーの場合の挙動
- codegen未実行時のimportエラー

非機能要件（Non-functional）:
- レスポンスタイム（必要な場合）
- アクセシビリティ（UIコンポーネントの場合）

UIコンポーネント固有（UIを含むタスクの場合）:
- アクセシビリティ詳細: ARIA属性の正確性、キーボードナビゲーション（Tab/Enter/Escape）、スクリーンリーダー対応
- インタラクション: クリック・キーボード・タッチ操作の全パターン
- レスポンシブ: ブレークポイント別の表示確認（sm/md/lg/xl）
- 状態表示: ローディング/エラー/空/無効の各状態のレンダリング
- ビジュアルリグレッション: Playwrightのスクリーンショット比較（`toHaveScreenshot()`）が導入済みの場合

テストシナリオのフォーマット:
```
- シナリオID: TC-001
  種別: ユニット
  対象: createNotification関数
  前提条件: DBに接続済み、有効なユーザーIDが存在する
  操作: 有効なtitleとbodyで createNotification を呼ぶ
  期待結果: 作成された通知オブジェクトが返り、DBに保存されている
```

作成したシナリオを `.claude/artifacts/test-plan-{yyyyMMdd-HHmm}.md` に出力する:

```markdown
# テスト計画

作成日: {yyyy-MM-dd HH:mm}
対象機能: {機能の概要}
参照設計: {design-*.md のパス}
参照要件: {requirements-*.md のパス（あれば）}

## テスト対象一覧

| 対象 | 種別 | 優先度 |
|------|------|--------|
| {関数/コンポーネント名} | {ユニット/統合/E2E} | {高/中/低} |

## テストシナリオ

### {対象名}

#### TC-001: {シナリオ名}
- 種別: {ユニット/統合/E2E}
- 前提条件: {セットアップ内容}
- 操作: {実行する操作}
- 期待結果: {検証内容}
- 優先度: {高/中/低}

...

## カバレッジ目標

- 全体: {X}%以上
- ビジネスロジック: {X}%以上（推奨: 90%以上）
- エッジケース: 主要なものを全て網羅
```

#### 完了チェックポイント（ステップ2）

- 正常系・異常系・境界値・エッジケースのシナリオが揃っていること
- `test-plan-*.md` が出力されていること
- 全シナリオに前提条件・操作・期待結果が記載されていること

### テストコマンドの自動検出

テスト実装・実行の前に、プロジェクトのテストコマンドを自動検出する:

1. package.jsonの "scripts.test" を確認 → あればそのコマンドを使用
2. vitest.config.* の存在確認 → あれば `pnpm vitest run`
3. jest.config.* の存在確認 → あれば `pnpm jest`
4. Cargo.toml の存在確認 → あれば `cargo test`
5. pytest.ini / pyproject.toml の [tool.pytest] 確認 → あれば `pytest`
6. 上記いずれにも該当しない場合はAskUserQuestionでテストコマンドを確認する

E2Eテストコマンドの自動検出:

1. playwright.config.* の存在確認 → あれば `pnpm playwright test`
2. cypress.config.* の存在確認 → あれば `pnpm cypress run`
3. 上記いずれにも該当しない場合はE2Eテストをスキップする

以降のステップでは、検出したテストコマンドを使用する。

### ステップ3: テストの実装

superpowers:test-driven-development のパターンに従い実装する。

> 注: superpowersプラグインが未インストールの場合は、TDDサイクル（Red→Green→Refactor）を手動で実行する。テストを先に書き、失敗を確認してから実装する。

Arrange-Act-Assert パターンに従う:

```typescript
// Arrange: テストデータとセットアップ
const userId = 'user-001';
const input = { title: 'テスト通知', body: '本文' };

// Act: テスト対象の操作を実行
const result = await createNotification(userId, input);

// Assert: 期待結果を検証
expect(result.title).toBe('テスト通知');
expect(result.userId).toBe(userId);
```

テスト実装の原則:
- テスト名は「何をテストしているか」が一目でわかるように記述する（日本語可）
- モックは最小限にし、実際の依存を使えるなら使う
- テストの独立性を保つ（テスト間の依存・実行順序への依存を避ける）
- `beforeEach` / `afterEach` でテストデータをクリーンアップする

モック設計の指針:
- カスタムフック（useGraphQL等）を複数回呼び出すコンポーネントのテストでは、クエリ名ベースでモック返却値を制御する
- `callCount` や呼び出し順序に依存するモックは禁止（クエリ追加時に全テストが壊れるため）
- MSWハンドラーを使える場合はMSWを優先する（`vi.mock` よりテストの信頼性が高い）
- 各クエリの loading/error 状態を独立してテストするケースを含める

codegen前提の確認:
- GraphQL型（`src/generated/`）を使うテストでは、`pnpm codegen` が実行済みであることを前提とする
- CIパイプラインにも `pnpm codegen` ステップが含まれていることを確認する
- import解決エラーが発生した場合、まず codegen 未実行を疑う

テストファイルの命名:
- ユニット: `src/__tests__/{対象ファイル名}.test.ts`
- 統合: `src/__tests__/integration/{機能名}.integration.test.ts`
- E2E: `e2e/{シナリオ名}.spec.ts`

#### 完了チェックポイント（ステップ3）

- テストシナリオが全てコードに実装されていること
- Arrange-Act-Assert パターンに従っていること
- テスト名が日本語で「何をテストするか」を表現していること

### ステップ4: テスト実行と結果レポート

1. テストを実行する:

   ユニット・統合テスト:
   ```bash
   # 自動検出したテストコマンドに --coverage オプションを付けて実行する
   ```

   E2Eテスト:
   ```bash
   # 自動検出したE2Eテストコマンドを実行する（未検出の場合はスキップ）
   ```

2. 結果を集計する:
   - 合格（Pass）: テストが期待通りに通過
   - 失敗（Fail）: テストが失敗（原因を特定して記録）
   - スキップ（Skip）: 意図的にスキップしたテスト（理由を記録）
   - カバレッジ: ステートメント・ブランチ・関数・行のカバレッジ率

3. 失敗したテストへの対応:
   - 実装のバグ → バグを修正してから再実行
   - テストの誤り → テストを修正してから再実行
   - 環境依存の問題 → 原因を記録し、AskUserQuestionで確認

4. 結果レポートをユーザーに提示する:

```markdown
## テスト結果レポート

実行日時: {yyyy-MM-dd HH:mm}

### サマリー

| 種別 | 合計 | 合格 | 失敗 | スキップ |
|------|------|------|------|--------|
| ユニット | {N} | {N} | {N} | {N} |
| 統合 | {N} | {N} | {N} | {N} |
| E2E | {N} | {N} | {N} | {N} |

カバレッジ: {X}% (目標: {X}%)

### 失敗したテスト（あれば）

- TC-XXX: {テスト名} - {失敗原因}

### 対応が必要な項目

- {要対応事項}
```

#### 完了チェックポイント（ステップ4）

- 全テストが実行されていること
- 失敗テストがない（または失敗の原因と対応方針が記録されていること）
- カバレッジが目標値以上であること
- 結果レポートがユーザーに提示されていること

### ステップ5: ハードニングモード（オプション: --harden）

通常のテストでは検出困難なエッジケースを、プロパティベーステスト（PBT）・ファジング・ミューテーションテストで機械的に発見する。

発動条件:
- odinコンテキストに `harden: true` が指定されている場合
- ユーザーが `--harden` フラグを指定した場合
- ステップ4で全テスト合格かつカバレッジ目標達成済みの場合にのみ実行する（基本テストが不安定な状態でハードニングしない）

#### 5-1. 言語・ツール自動検出

プロジェクトの言語を検出し、対応するハードニングツールを選択する:

| 言語 | PBTツール | Mutation Testing | Fuzzing |
|------|----------|-----------------|---------|
| TypeScript/JavaScript | fast-check | Stryker | - |
| Python | Hypothesis | mutmut | - |
| Rust | proptest | cargo-mutants | cargo-fuzz |
| Go | rapid | go-mutesting | go-fuzz |

検出方法:
1. package.json存在 → TypeScript/JavaScript
2. pyproject.toml / requirements.txt存在 → Python
3. Cargo.toml存在 → Rust
4. go.mod存在 → Go
5. 複数該当する場合はメイン言語（ソースファイル数が最多）を選択

ツールがプロジェクトに未導入の場合:
- AskUserQuestionで「{ツール}をdevDependenciesに追加してよいか」を確認する
- 拒否された場合はハードニングをスキップする

#### 5-2. PBT（プロパティベーステスト）の実行

対象: ビジネスロジック中心の純粋関数・バリデーション関数・デー��変換関数
（UIコンポーネント・副作用を持つ関数は対象外）

手順:
1. テスト対象の関数を特定する（ステップ1のテスト対象一覧から純粋関数を抽出）
2. 各関数に対してプロパティ（常に成立すべき性質）を定義する
3. PBTテストコードを生成する
4. 実行して反例（failing case）を検出する

プロパティの例:
- 逆変換: `decode(encode(x)) === x`
- 冪等性: `f(f(x)) === f(x)`
- 単調性: `x < y → f(x) <= f(y)`
- 不変条件: `validate(generate()) === true`

#### 5-3. Mutation Testingの実行

対象: テストカバレッジが高い（80%以上）モジュール
（カバレッジが低いモジュールは先に通常テストを追加すべき）

手順:
1. Mutation Testingツールを設定する（対象ディレクトリ、除外パターン）
2. ��行する（ミュータント生成 → テスト実行 → 生存ミュータント検出）
3. 生存ミュータントを分析する:
   - テスト不足が原因 → テスト追加を推奨
   - 等価ミュータント → 無視してよい
   - ロジックの冗長性が原因 → リファクタリングを推奨

#### 5-4. ハードニング結果レポート

```markdown
## ハードニング結果

### PBT
- 対象関数: {N}個
- プロパティ数: {N}
- 反例検出: {N}件
  - {関数名}: {反例の説明}（入力: {値}、期待: {値}、実際: {値}）

### Mutation Testing
- ミュータント生成数: {N}
- 殺害済み: {N} ({X}%)
- 生存: {N} ({X}%)
  - {ファイル:行番号}: {ミュータントの説明} → テスト追加を推奨

### 推奨アクション
- {反例に対する修正案}
- {生存ミュータントに対するテスト追加案}
```

#### 完了チェックポイント（ステップ5）

- 発動条件を満たす場合のみ実行されていること
- PBTまたはMutation Testingの少なくとも一方が実行されていること
- 反例・生存ミュータントに対する推奨アクションが記載されていること

## Examples

### 通知APIのユニットテストを書く

ユーザー: 「design-20260320-1600.md をもとに通知APIのテストを書いて」

```

  対象: createNotification, getNotifications, markAsRead

  各シナリオをAAA形式で実装

  自動検出したテストコマンド --coverage で実行
  全10件合格、カバレッジ92%
```
