# odin-do-test

テスト計画・実装・実行スキル。odin司令塔のdoフェーズで使用する。
設計ドキュメント・要件定義書・既存のテスト計画を入力として、テストシナリオの洗い出し・テスト実装・実行・結果レポートまでを担う。

## Instructions

### 完了チェックポイントの原則

各ステップの最後には「完了チェックポイント」を設けている。
チェックポイントに記載された全ての条件を満たさない限り、次のステップに進んではならない。

### コンテキスト検出

$ARGUMENTS を確認し、odinコンテキストJSONが含まれているか判定する。

odinコンテキストJSONの例:
```json
{
  "odin_context": {
    "task": "通知機能のテスト",
    "artifacts": {
      "design": ".claude/artifacts/design-20260320-1600.md",
      "requirements": ".claude/artifacts/requirements-20260320-1500.md",
      "test_plan": ".claude/artifacts/test-plan-20260320-1800.md"
    }
  }
}
```

- odinコンテキストがある場合: artifactsから入力ファイルのパスを取得する
- odinコンテキストがない場合: 成果物管理ルールに従い入力ファイルを探索する

### 成果物管理

入力ファイルの探索順序:
1. $ARGUMENTSにファイルパスが明示指定されていればそれを使う
2. odinコンテキストのartifactsにパスがあればそれを使う
3. `.claude/artifacts/` 内の最新の `design-*.md`, `requirements-*.md`, `test-plan-*.md` を使う
4. 該当ファイルが見つからない場合はAskUserQuestionでユーザーに確認する

出力ファイル: `.claude/artifacts/test-plan-{yyyyMMdd-HHmm}.md`

### ステップ1: テスト対象の確認

1. 入力ファイルを読み込む（設計・要件・既存テスト計画）
2. テストの種別を特定する:

| 種別 | 対象 | 目的 |
|------|------|------|
| ユニットテスト | 個別の関数・コンポーネント | 単体の振る舞いを検証 |
| 統合テスト | モジュール間の連携 | 複数の部品の協調動作を検証 |
| E2Eテスト | ユーザーフロー全体 | エンドツーエンドの動作を検証 |

3. テスト対象のファイル・コンポーネント・APIエンドポイントを列挙する
4. AskUserQuestionで確認が必要な場合（テスト種別・優先度）は確認する

#### 完了チェックポイント（ステップ1）

- テスト対象の機能・コンポーネントが特定されていること
- テスト種別（ユニット/統合/E2E）が決定されていること
- テスト対象ファイルの一覧が把握できていること

### ステップ2: テストシナリオの洗い出し

各テスト対象について、以下の観点でシナリオを洗い出す:

正常系（Happy Path）:
- 期待した入力に対して期待した出力が返ること
- 標準的なユーザー操作フローが正しく動作すること

異常系（Sad Path）:
- 無効な入力に対してエラーが正しく返ること
- 必須フィールドが欠けた場合の挙動
- 認証・認可エラーの挙動

境界値（Boundary Values）:
- 最小値・最大値・ちょうどの値
- 空文字・null・undefined
- 文字列の長さ制限の境界

エッジケース（Edge Cases）:
- 同時実行・競合状態
- ネットワークエラー・タイムアウト
- 大量データ・極端なケース

非機能要件（Non-functional）:
- レスポンスタイム（必要な場合）
- アクセシビリティ（UIコンポーネントの場合）

テストシナリオのフォーマット:
```
- シナリオID: TC-001
  種別: ユニット
  対象: createNotification関数
  前提条件: DBに接続済み、有効なユーザーIDが存在する
  操作: 有効なtitleとbodyで createNotification を呼ぶ
  期待結果: 作成された通知オブジェクトが返り、DBに保存されている
```

作成したシナリオを `.claude/artifacts/test-plan-{yyyyMMdd-HHmm}.md` に出力する:

```markdown
# テスト計画

作成日: {yyyy-MM-dd HH:mm}
対象機能: {機能の概要}
参照設計: {design-*.md のパス}
参照要件: {requirements-*.md のパス（あれば）}

## テスト対象一覧

| 対象 | 種別 | 優先度 |
|------|------|--------|
| {関数/コンポーネント名} | {ユニット/統合/E2E} | {高/中/低} |

## テストシナリオ

### {対象名}

#### TC-001: {シナリオ名}
- 種別: {ユニット/統合/E2E}
- 前提条件: {セットアップ内容}
- 操作: {実行する操作}
- 期待結果: {検証内容}
- 優先度: {高/中/低}

...

## カバレッジ目標

- 全体: {X}%以上
- ビジネスロジック: {X}%以上（推奨: 90%以上）
- エッジケース: 主要なものを全て網羅
```

#### 完了チェックポイント（ステップ2）

- 正常系・異常系・境界値・エッジケースのシナリオが揃っていること
- `test-plan-*.md` が出力されていること
- 全シナリオに前提条件・操作・期待結果が記載されていること

### ステップ3: テストの実装

superpowers:test-driven-development のパターンに従い実装する。

Arrange-Act-Assert パターンに従う:

```typescript
// Arrange: テストデータとセットアップ
const userId = 'user-001';
const input = { title: 'テスト通知', body: '本文' };

// Act: テスト対象の操作を実行
const result = await createNotification(userId, input);

// Assert: 期待結果を検証
expect(result.title).toBe('テスト通知');
expect(result.userId).toBe(userId);
```

テスト実装の原則:
- テスト名は「何をテストしているか」が一目でわかるように記述する（日本語可）
- モックは最小限にし、実際の依存を使えるなら使う
- テストの独立性を保つ（テスト間の依存・実行順序への依存を避ける）
- `beforeEach` / `afterEach` でテストデータをクリーンアップする

テストファイルの命名:
- ユニット: `src/__tests__/{対象ファイル名}.test.ts`
- 統合: `src/__tests__/integration/{機能名}.integration.test.ts`
- E2E: `e2e/{シナリオ名}.spec.ts`

#### 完了チェックポイント（ステップ3）

- テストシナリオが全てコードに実装されていること
- Arrange-Act-Assert パターンに従っていること
- テスト名が日本語で「何をテストするか」を表現していること

### ステップ4: テスト実行と結果レポート

1. テストを実行する:

   ユニット・統合テスト:
   ```bash
   pnpm vitest run --coverage
   ```

   E2Eテスト（Playwrightの場合）:
   ```bash
   pnpm playwright test
   ```

2. 結果を集計する:
   - 合格（Pass）: テストが期待通りに通過
   - 失敗（Fail）: テストが失敗（原因を特定して記録）
   - スキップ（Skip）: 意図的にスキップしたテスト（理由を記録）
   - カバレッジ: ステートメント・ブランチ・関数・行のカバレッジ率

3. 失敗したテストへの対応:
   - 実装のバグ → バグを修正してから再実行
   - テストの誤り → テストを修正してから再実行
   - 環境依存の問題 → 原因を記録し、AskUserQuestionで確認

4. 結果レポートをユーザーに提示する:

```markdown
## テスト結果レポート

実行日時: {yyyy-MM-dd HH:mm}

### サマリー

| 種別 | 合計 | 合格 | 失敗 | スキップ |
|------|------|------|------|--------|
| ユニット | {N} | {N} | {N} | {N} |
| 統合 | {N} | {N} | {N} | {N} |
| E2E | {N} | {N} | {N} | {N} |

カバレッジ: {X}% (目標: {X}%)

### 失敗したテスト（あれば）

- TC-XXX: {テスト名} - {失敗原因}

### 対応が必要な項目

- {要対応事項}
```

#### 完了チェックポイント（ステップ4）

- 全テストが実行されていること
- 失敗テストがない（または失敗の原因と対応方針が記録されていること）
- カバレッジが目標値以上であること
- 結果レポートがユーザーに提示されていること

## Examples

### 通知APIのユニットテストを書く

ユーザー: 「design-20260320-1600.md をもとに通知APIのテストを書いて」

```
ステップ1: 設計確認
  対象: createNotification, getNotifications, markAsRead
  種別: ユニットテスト（APIロジック）+ 統合テスト（DBアクセス）

ステップ2: シナリオ洗い出し
  TC-001: 有効なパラメータで通知が作成できること（正常系）
  TC-002: userIdが存在しない場合に404が返ること（異常系）
  TC-003: titleが空の場合に400が返ること（境界値）
  TC-004: 1000件の通知を取得しても正常に動作すること（エッジケース）
  → test-plan-20260320-1900.md に出力

ステップ3: テスト実装
  src/__tests__/notifications.test.ts を作成
  各シナリオをAAA形式で実装

ステップ4: 実行・レポート
  pnpm vitest run --coverage
  全10件合格、カバレッジ92%
```
