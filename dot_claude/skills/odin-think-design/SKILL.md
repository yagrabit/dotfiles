# odin-think-design

設計ドキュメント・ADR作成スキル。odin司令塔のthinkフェーズで使用する。
要件定義書とコードベース調査レポートを入力として、設計ドキュメントとADRを作成し、ユーザーの承認を得る。

## Instructions

### 完了チェックポイントの原則

各ステップの最後には「完了チェックポイント」を設けている。
チェックポイントに記載された全ての条件を満たさない限り、次のステップに進んではならない。

### コンテキスト検出

$ARGUMENTS を確認し、odinコンテキストJSONが含まれているか判定する。

odinコンテキストJSONの例:
```json
{
  "odin_context": {
    "task": "通知機能の追加",
    "artifacts": {
      "requirements": ".claude/artifacts/requirements-20260320-1500.md",
      "research": ".claude/artifacts/research-20260320-1430.md"
    }
  }
}
```

- odinコンテキストがある場合: artifactsから入力ファイルのパスを取得する
- odinコンテキストがない場合: 成果物管理ルールに従い入力ファイルを探索する

### 成果物管理

入力ファイルの探索順序:
1. $ARGUMENTSにファイルパスが明示指定されていればそれを使う
2. odinコンテキストのartifactsにパスがあればそれを使う
3. `.claude/artifacts/` 内の最新の `requirements-*.md` と `research-*.md` を使う
4. 該当ファイルが見つからない場合はAskUserQuestionでユーザーに確認する

### ステップ1: 入力ファイルの読み込み

1. 要件定義書（requirements-*.md）を読み込む
2. コードベース調査レポート（research-*.md）を読み込む
3. 既存の設計ドキュメントがある場合は差分設計、ない場合はフル設計を行う

フル設計 vs 差分設計の判定:
- 新規機能・新規プロジェクト → フル設計（全セクションを新規作成）
- 既存機能の変更・拡張 → 差分設計（変更影響箇所を中心に設計し、変更なし箇所は「既存のまま」と記載）

#### 完了チェックポイント（ステップ1）

- 要件定義書の内容を把握していること
- コードベース調査レポートの内容を把握していること（なければ「調査レポートなし」として続行）
- フル設計か差分設計かが決定されていること

### ステップ2: アプローチの検討

brainstorming（superpowers:brainstorming）スキルまたは直接思考で2〜3のアプローチを検討する。

各アプローチについて以下を評価する:
- 実装コスト（高/中/低）
- 保守性（高/中/低）
- 既存コードへの影響範囲（大/中/小）
- リスク（高/中/低）
- トレードオフ

推奨アプローチを1つ選び、選定理由を明記する。

#### 完了チェックポイント（ステップ2）

- 2〜3のアプローチが検討されていること
- 各アプローチのトレードオフが明記されていること
- 推奨アプローチが選定されていること

### ステップ3: 設計ドキュメントの出力

1. `.claude/artifacts/` ディレクトリが存在しなければ作成する
2. 現在時刻を取得し、`yyyyMMdd-HHmm` 形式のタイムスタンプを生成する
3. 以下のフォーマットで設計ドキュメントを出力する

出力先: `.claude/artifacts/design-{yyyyMMdd-HHmm}.md`

```
# 設計ドキュメント

作成日: {yyyy-MM-dd HH:mm}
対象機能: {機能の概要}
設計種別: {フル設計 / 差分設計}
参照要件: {requirements-*.md のパス}
参照調査: {research-*.md のパス（あれば）}

## 概要

{設計の全体像・目的の1〜3文のサマリー}

## アーキテクチャ

{選定したアプローチの詳細説明}
{アーキテクチャ図（テキストまたはMermaid記法）}

検討したアプローチ:
1. {アプローチA}: {メリット/デメリット}
2. {アプローチB}: {メリット/デメリット}
3. {アプローチC（あれば）}: {メリット/デメリット}

採用理由: {なぜこのアプローチを選んだか}

## コンポーネント設計

{新規作成・変更するファイルの一覧と役割}

| ファイルパス | 役割 | 新規/変更/削除 |
|-------------|------|--------------|
| {パス} | {役割} | {新規/変更/削除} |

{各コンポーネントのインターフェース定義（型・関数シグネチャ）}

## データフロー

{データの流れ（ユーザー操作 → コンポーネント → API → DB の流れ等）}
{Mermaid記法のシーケンス図やフローチャートを活用}

## エラーハンドリング

{エラーケースと対処方針の一覧}

| エラーケース | 対処方針 | ユーザーへの表示 |
|------------|---------|---------------|
| {ケース} | {方針} | {表示内容} |

## テスト方針

{ユニットテスト・統合テスト・E2Eテストの対象と方針}
{テスト難易度が高い箇所の特定とモック方針}

## 影響範囲

{既存機能・コンポーネントへの影響}
{破壊的変更の有無と移行方針}

## セキュリティ考慮事項

{認証・認可、入力検証、データ保護等のセキュリティ対策}
```

#### 完了チェックポイント（ステップ3）

- `.claude/artifacts/design-*.md` が出力されていること
- 全セクションが埋まっていること（該当なしの場合は「該当なし」と明記）
- コンポーネント設計にファイルパスが具体的に記載されていること

### ステップ4: ADRの作成

設計上の重要な意思決定（技術選定、アーキテクチャパターン、外部サービス採用等）についてADRを作成する。

ADRを作成すべき判断の目安:
- 複数の選択肢を検討して1つを選んだ場合
- 将来のメンテナーが「なぜこうしたのか」と疑問に思いそうな選択
- 後から変更するとコストが高い選択

1. 既存のADRがある場合は連番を継続する（`.claude/artifacts/adr-*.md` を確認）
2. 新規の場合は adr-001 から始める

出力先: `.claude/artifacts/adr-{NNN}-{yyyyMMdd-HHmm}.md`

```
# ADR-{NNN}: {意思決定のタイトル}

作成日: {yyyy-MM-dd}
ステータス: {Proposed / Accepted / Deprecated / Superseded}
関連設計: {design-*.md のパス}

## コンテキスト

{なぜこの決定が必要だったか。背景と問題の説明}

## 判断

{採用した選択肢とその概要}

## 選択肢

### 選択肢1: {選択肢名}
- メリット: {利点}
- デメリット: {欠点}

### 選択肢2: {選択肢名}
- メリット: {利点}
- デメリット: {欠点}

### 選択肢3（あれば）: {選択肢名}
- メリット: {利点}
- デメリット: {欠点}

## 結果

{採用した選択肢の理由・期待される効果・受け入れた制約}
```

#### 完了チェックポイント（ステップ4）

- 重要な意思決定についてADRが作成されていること（意思決定がない場合は「ADR不要」と記録）
- ADRにステータスが記載されていること
- 選択肢と採用理由が明記されていること

### ステップ5: ユーザー承認

1. 設計ドキュメントとADRの概要をユーザーに提示する
2. 以下を確認する:
   - 設計方針に問題がないか
   - 見落としている考慮事項がないか
   - ADRの内容に同意できるか
3. AskUserQuestionで承認を求める
4. フィードバックがあれば設計ドキュメントを更新してから承認を得る
5. 承認が得られたらodinに設計ドキュメントのパスを返す

#### 完了チェックポイント（ステップ5）

- ユーザーの承認が得られていること
- フィードバックがあった場合は設計ドキュメントに反映されていること
- 出力ファイルパスが確認済みであること

## Examples

### 既存プロジェクトに通知機能を追加する設計

ユーザー: 「通知機能の設計をして。要件: requirements-20260320-1500.md、調査: research-20260320-1430.md」

```
ステップ1: 入力ファイル読み込み
  要件: FR-001〜FR-003（通知作成・既読・削除）
  調査: Next.js App Router, Prisma, SSEの前例なし
  → 差分設計（既存APIルートに追加）

ステップ2: アプローチ検討
  A: WebSocket → リアルタイム性高いがインフラコスト大
  B: Server-Sent Events (SSE) → 単方向だが実装シンプル
  C: ポーリング → シンプルだがリアルタイム性低い
  → B（SSE）を推奨: 要件的にリアルタイム受信のみ必要

ステップ3: 設計ドキュメント出力
  → .claude/artifacts/design-20260320-1600.md

ステップ4: ADR作成
  → .claude/artifacts/adr-001-20260320-1600.md
     「WebSocketではなくSSEを採用」

ステップ5: ユーザー承認
  「SSE採用の設計で問題ありませんか？」→ 承認
```
