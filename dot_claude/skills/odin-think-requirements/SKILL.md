# odin-think-requirements

要件分析・構造化スキル。odin司令塔のthinkフェーズで使用する。
Confluenceページ、テキスト、口頭説明から要件を取得・整理し、機能要件・非機能要件・制約・前提・用語を明確化した要件定義書を出力する。

## Instructions

### 完了チェックポイントの原則

各ステップの最後には「完了チェックポイント」を設けている。
チェックポイントに記載された全ての条件を満たさない限り、次のステップに進んではならない。

### コンテキスト検出

$ARGUMENTS を確認し、odinコンテキストJSONが含まれているか判定する。

odinコンテキストJSONの例:
```json
{
  "odin_context": {
    "task": "ユーザー認証機能の追加",
    "requirements_source": "https://confluence.example.com/pages/viewpage.action?pageId=123",
    "requirements_text": "ログイン・ログアウト機能を実装する...",
    "artifacts": {
      "research": ".claude/artifacts/research-20260320-1430.md"
    }
  }
}
```

- odinコンテキストがある場合: そこから要件ソースを読み取り、ステップ1を短縮する
- odinコンテキストがない場合: ステップ1からAskUserQuestionで確認する

### ステップ1: 要件ソースの取得

以下のいずれかの方法で要件を取得する:

方法A: Confluenceページから取得
- $ARGUMENTSまたはodinコンテキストにConfluenceのURLがある場合
- AtlassianプラグインのgetConfluencePageツールでページ内容を取得する
- ページが複数ある場合は全て読み込む

方法B: テキストから取得
- $ARGUMENTSまたはodinコンテキストに要件テキストが直接含まれている場合
- そのまま使用する

方法C: 口頭/対話形式
- 要件ソースが不明な場合、AskUserQuestionで以下を確認する:
  1. 実装したい機能の概要
  2. 対象ユーザー（誰が使うのか）
  3. 背景・動機（なぜ必要か）
  4. 参考となるConfluenceページやドキュメントのURL（あれば）
  5. 期待する完成イメージ・ゴール

#### 完了チェックポイント（ステップ1）

- 要件の生データ（テキスト）が取得できていること
- 機能の概要が把握できていること

### ステップ2: 要件の構造化

取得した要件を以下の観点で整理する:

機能要件（FR）:
- ユーザーストーリー形式: 「{ロール}として、{機能}したい。なぜなら{価値}だから。」
- 受け入れ基準（GIVEN/WHEN/THEN形式）を各ストーリーに付ける
- FR-001, FR-002 の連番を付ける

非機能要件（NFR）:
- パフォーマンス（応答時間、スループット等）
- セキュリティ（認証、認可、データ保護等）
- アクセシビリティ（WCAG基準等）
- 可用性・信頼性
- スケーラビリティ
- NFR-001, NFR-002 の連番を付ける

制約:
- 技術的制約（使用技術、既存システムとの互換性等）
- ビジネス制約（期限、予算、法規制等）
- スコープ外（明示的に対象外とするもの）

前提条件:
- 本機能が依存する他機能・システム
- 実装前提となるインフラ・設定

用語定義:
- ドメイン固有の用語とその定義

#### 完了チェックポイント（ステップ2）

- 機能要件がユーザーストーリー形式で列挙されていること
- 非機能要件が少なくとも1つ以上明記されていること
- 制約・前提条件が整理されていること

### ステップ2.5: EARS形式変換（オプション）

核心的な振る舞い要件（機能要件のうち、システムの動的な振る舞いを定義するもの）をEARS（Easy Approach to Requirements Syntax）形式に変換する。EARS形式は要件のテスト可能性を構造的に保証する。

発動条件:
- odinコンテキストに `ears: true` が指定されている場合
- ユーザーが「EARS形式で」「テスト可能な形式で」と指定した場合
- 機能要件が5つ以上あり、テスト可能性を高める価値がある場合（自動判定）
- 上記いずれにも該当しない場合はスキップする

EARS形式の5パターン:

| パターン | テンプレート | 適用場面 |
|---------|------------|---------|
| Ubiquitous | The [system] shall [behavior] | 常に成立すべき制約 |
| Event-driven | When [trigger], the [system] shall [response] | イベント駆動の振る舞い |
| State-driven | While [condition], the [system] shall [behavior] | 状態依存の振る舞い |
| Unwanted | If [unwanted condition], then the [system] shall [response] | 異常系・エラー処理 |
| Optional | Where [feature is enabled], the [system] shall [behavior] | オプション機能 |

変換手順:
1. ステップ2の機能要件（FR-*）から振る舞い要件を抽出する
2. 各要件に最適なEARSパターンを選択する
3. EARS形式に変換する（元のユーザーストーリーも保持する）
4. 変換結果をステップ5の出力に「EARS形式要件」セクションとして追加する

出力形式（ステップ5の要件定義書内に追加）:
```markdown
## EARS形式要件（テスト可能な仕様）

### EARS-001: {FR-001に対応}
パターン: Event-driven
When ユーザーがロ���インフォームに有効な認証情報を入力して送信する,
the 認証システム shall セッショントークンを生成し、ダッシュボードにリダイレクトする

### EARS-002: {FR-001の異常系に対応}
パターン: Unwanted
If ユーザーが無効な認証情報で3回連続ログインに失敗した,
then the 認証システム shall アカウントを一時ロックし、ロック通知メールを送信する
```

注意:
- 全FRをEARS化する必要はない。振る舞い要件のみ変換する（UI仕様やデータ定義は対象外）
- EARS形式はodin-do-testがテストケース自動生成の入力として使用する

#### 完了チェックポイント（ステップ2.5）

- 発動条件を満たす場合のみ実行されていること
- 変換対象のFRにEARS-{NNN}が紐づいていること
- 各EARS要件にパターン名が明記されていること

### ステップ3: requirements-analystによる品質チェック

requirements-analystサブエージェントに構造化した要件を渡し、以下を検出させる:

- 曖昧な記述（「適切に」「なるべく」等の定義不明な表現）
- 矛盾している要件
- 不足している情報（エラーケース、境界値、権限制御等）
- 未定義の用語

エージェントの出力をリストアップする。

#### 完了チェックポイント（ステップ3）

- requirements-analystの分析結果が得られていること
- 指摘事項のリストが作成されていること

### ステップ4: AskUserQuestionで明確化

ステップ3で検出した指摘事項を1つずつAskUserQuestionで確認する。

- 一度に1つの質問をする（まとめて質問しない）
- 選択肢がある場合は2〜4個に絞って提示する
- 回答を構造化済み要件に反映する
- 全ての指摘事項を確認し終えたら次のステップに進む

優先度の高い確認事項:
1. スコープ境界（何が入って何が入らないか）
2. エラーケース・異常系の扱い
3. 権限・認可（誰が何をできるか）
4. 既存機能との整合性

#### 完了チェックポイント（ステップ4）

- 全ての指摘事項に対して確認が完了していること
- 未解決の確認ポイントがある場合はその理由が記録されていること
- 構造化要件が最新の回答を反映していること

### ステップ5: 要件定義書の出力

1. `.claude/artifacts/` ディレクトリが存在しなければ作成する
2. 現在時刻を取得し、`yyyyMMdd-HHmm` 形式のタイムスタンプを生成する
3. 以下のフォーマットで要件定義書を出力する（先頭にodin_coherence frontmatterを付与する。`depends_on` には入力として使用したresearch成果物のidを記載する）

出力先: `.claude/artifacts/requirements-{yyyyMMdd-HHmm}.md`

```
---
odin_coherence:
  id: "requirements-{yyyyMMdd-HHmm}"
  kind: "requirements"
  depends_on:
    - "{入力で使用したresearch成果物のid 例: research-20260320-1430}"
  updated: "{yyyy-MM-ddTHH:mm}"
---

# 要件定義書

作成日: {yyyy-MM-dd HH:mm}
対象機能: {機能の概要}
ステータス: {Draft / Review中 / 承認済み}

## 背景・目的

{なぜこの機能が必要か、誰が使うのか}

## 機能要件

### FR-001: {要件名}

ユーザーストーリー:
{ロール}として、{機能}したい。なぜなら{価値}だから。

受け入れ基準:
- GIVEN {前提条件}
  WHEN {操作・イベント}
  THEN {期待する結果}

### FR-002: {要件名}
...

## 非機能要件

### NFR-001: {要件名}

{具体的な数値・基準を含む説明}

### NFR-002: {要件名}
...

## 制約

- {技術的制約}
- {ビジネス制約}

## スコープ外

- {明示的に対象外とするもの}

## 前提条件

- {依存する機能・システム}

## 用語定義

| 用語 | 定義 |
|------|------|
| {用語} | {定義} |

## 未解決の確認ポイント

| No. | 質問 | 影響 | 優先度 |
|-----|------|------|--------|
| Q-001 | {質問内容} | {影響を受ける要件} | {高/中/低} |
```

4. 要件定義書の概要をユーザーに提示する
5. odinから呼ばれた場合は、出力ファイルパスをodinに返す

#### 完了チェックポイント（ステップ5）

- `.claude/artifacts/requirements-*.md` が出力されていること
- 全セクションが埋まっていること（該当なしの場合は「該当なし」と明記）
- FR-001形式の連番が付いていること
- 未解決の確認ポイントが記録されていること（0件でもセクションを明記する）

### odin-talk-clarifyとの棲み分け

- odin-talk-clarify: コードベースを読む前の段階で、対話を通じて要件の曖昧点を解消する
- odin-think-requirements: コードベース調査後の段階で、技術スタック・制約条件を加味した要件の構造化を行う

このスキルはコードベース調査結果をもとに要件を構造化する。曖昧な要件の対話的解消が先に必要な場合はodin-talk-clarifyを先に実行する。

## Examples

### ログイン機能の要件を整理する場合

ユーザー: 「ConfluenceのURL: https://... からログイン機能の要件を整理して」

```
ステップ1: Confluenceページから要件取得
  → ページ内容: 「メールアドレスとパスワードでログインできること...」

ステップ2: 構造化
  FR-001: メールアドレス・パスワードでのログイン
  FR-002: ログアウト
  FR-003: パスワードリセット
  NFR-001: ログイン試行は5回失敗でロック

ステップ3: requirements-analyst分析
  指摘: 「ロック解除方法が未定義」「セッション有効期限の記載なし」

ステップ4: AskUserQuestionで確認
  Q: 「アカウントロック後の解除方法はどうしますか？」
  A: 「管理者によるロック解除またはメールによる自動解除」

ステップ5: 要件定義書出力
  → .claude/artifacts/requirements-20260320-1500.md
```
