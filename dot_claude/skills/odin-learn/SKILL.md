# odin-learn

技術力成長のための学習セッション管理スキル。
フロントエンドをリードエンジニアレベルまで、Webデザインを芸大卒デザイナーと張り合えるレベルまで、周辺技術を実務で困らないレベルまで引き上げることを目標とする。

スキルマップで習熟度を追跡し、間隔反復学習で知識の長期定着を図り、実際のコードを題材にした実践的な学習を提供する。

## ブルームのタキソノミー（学習段階）

| レベル | 名称 | 意味 |
|--------|------|------|
| 1 | Remember | 構文・API名を思い出せる |
| 2 | Understand | コードの動作を説明できる |
| 3 | Apply | 仕様をコードに翻訳できる |
| 4 | Analyze | 構造を分解し問題を特定できる |
| 5 | Evaluate | 設計の代替案を比較判断できる |
| 6 | Create | 抽象的な問題の解決策を設計できる |

## Instructions

### 完了チェックポイントの原則

各ステップの最後には「完了チェックポイント」を設けている。
チェックポイントに記載された全ての条件を満たさない限り、次のステップに進んではならない。

### モード判定

$ARGUMENTS から実行モードを判定する:

- `init` → スキルマップ初期化（ステップ A）
- `status` / `スキルマップを見せて` → 進捗レポート（ステップ B）
- `review` / `復習して` → フラッシュカード復習（ステップ C）
- `session` / `学習して` / `今日の学習` / 引数なし → 学習セッション（ステップ D）
- odinコンテキストJSON → コンテキストのtaskフィールドから判定

---

### 保存場所に関する注意

skillmap-latest.md と flashcards-latest.md はプロジェクト横断の個人学習データである。
必ずユーザーレベルの `~/.claude/artifacts/` に保存すること。
プロジェクトの `.claude/artifacts/` に保存してはならない（他のリポジトリからアクセスできなくなる）。

`~/.claude/artifacts/` ディレクトリが存在しない場合は作成する。

### ステップ C: フラッシュカード復習

C-1. `~/.claude/artifacts/flashcards-latest.md` を読み込む。
   - ファイルが存在しない場合: 「まだフラッシュカードがありません。PRレビューの教育モード（/odin-learn-review）でカードが自動生成されます。または学習セッション（/odin 学習して）でも生成されます。」と案内して終了。

C-2. 今日が「次回復習」日以前のカードを抽出する（期限切れ + 今日が期限のカード）。

C-3. 各カードについて以下の復習フローを実行する:

3a. 質問を提示する:
```
復習カード {N}/{total}
領域: {スキル領域}

Q: {質問}
```

3b. AskUserQuestionで回答を求める:
1. [正解候補A]
2. [正解候補B]
3. 考え中... ヒントがほしい
4. （自由記述）

3c. 正誤判定とフィードバック:
- 正解: 「正解です。」+ 簡潔な補足 → 次回復習間隔を延長
- 不正解: 「惜しい / 違います。」+ 正解の解説 → 次回復習間隔をリセット（+1日）

3d. カードの次回復習日を更新する。

C-4. 復習結果サマリーを提示する:
```
--- 復習完了 ---
正答率: {N}/{total} ({X}%)
次回復習: {最も近い復習日}
```

#### 完了チェックポイント（ステップ C）

- 全ての期限切れカードを復習していること
- 復習結果に基づいてカードの次回復習日が更新されていること

---

### ステップ D: 学習セッション

D-1. `~/.claude/artifacts/skillmap-latest.md` を読み込む。
   - 存在しない場合: ステップ A（初期化）を先に実行する。

D-2. 学習トピックの選定。以下の優先順位で選ぶ:

優先度1: ユーザーが明示的にトピックを指定した場合 → そのトピック
優先度2: 復習期限が超過している領域 → 復習セッション（ステップ C に遷移）
優先度3: 目標レベルとの差が最も大きい領域
優先度4: 最終学習日が最も古い（または未学習の）領域

D-3. AskUserQuestionでトピックを提案する:
```
今日の学習トピックの提案です:

1. {推奨トピック}（理由: {選定理由}）
2. {代替トピック1}
3. {代替トピック2}
4. 自分で選びたい
```

D-4. 選択されたトピックについてWebSearchで最新のベストプラクティスを調査する。

D-5. レッスンの構成。ユーザーの現在レベルに応じた内容を構成する:

現在 Remember → Understand を目指す:
- 概念の説明（図解的に）
- 「なぜ」の解説
- 実際のコード例での動作説明
- 理解度チェック: 「このコードは何をしている？」

現在 Understand → Apply を目指す:
- パターンの紹介
- 小さなコーディング課題
- 「この仕様を実装してみてください」
- WebSearchで見つけた実務的なtipsの紹介

現在 Apply → Analyze を目指す:
- 問題のあるコードの分析課題
- 「このコードの問題点は？」
- デバッグシナリオ
- 実際のプロジェクトコードを題材にした分析

現在 Analyze → Evaluate を目指す:
- 設計の比較課題
- 「AとBのアプローチ、どちらを選ぶ？理由は？」
- トレードオフの議論
- 実際のアーキテクチャ判断シナリオ

現在 Evaluate → Create を目指す:
- ゼロからの設計課題
- 「この要件のアーキテクチャを設計してください」
- RFC/ADRの作成演習
- 実際のOSSプロジェクトの設計分析

D-6. ソクラテス式対話で学習を進める（odin-learn-reviewのステップ3と同じフレームワーク）。

D-7. 学んだ内容からフラッシュカードを生成し、flashcards-latest.md に追記する。

D-8. スキルマップを更新する。

D-9. 学習サマリーを提示する:
```
--- 学習セッション完了 ---

トピック: {学習した領域}

学んだこと:
1. {ポイント1}
2. {ポイント2}
3. {ポイント3}

スキルマップ更新:
- {領域}: {旧レベル} → {新レベル}

フラッシュカード: {N}枚追加

次回のおすすめ:
- {次に学ぶべきトピックと理由}
```

#### 完了チェックポイント（ステップ D）

- 学習セッションが完了していること
- フラッシュカードが生成されていること
- スキルマップが更新されていること
- 学習サマリーが提示されていること
