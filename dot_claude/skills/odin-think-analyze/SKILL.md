# odin-think-analyze

品質・パフォーマンス分析スキル。odin司令塔のthinkフェーズで使用する。
コード品質・パフォーマンス・バンドルサイズ・影響範囲・依存関係の5種類の分析を実施し、改善提案付きのレポートを出力する。

## Instructions

### 完了チェックポイントの原則

各ステップの最後には「完了チェックポイント」を設けている。
チェックポイントに記載された全ての条件を満たさない限り、次のステップに進んではならない。

### コンテキスト検出

$ARGUMENTS を確認し、odinコンテキストJSONが含まれているか判定する。

odinコンテキストJSONの例:
```json
{
  "odin_context": {
    "task": "パフォーマンス改善のための分析",
    "analysis_type": "performance",
    "target": "src/components/DataTable",
    "artifacts": {}
  }
}
```

- odinコンテキストがある場合: analysis_typeとtargetを読み取る
- odinコンテキストがない場合: ステップ1でAskUserQuestionを使用する

### ステップ1: 分析種類の特定

AskUserQuestionで以下を確認する（odinコンテキストがない場合）:

1. 分析の目的・背景（なぜ分析が必要か）
2. 分析の種類（複数選択可）:
   - コード品質: 複雑度・重複・ファイルサイズ・保守性
   - パフォーマンス: ボトルネック・N+1・メモリリーク
   - バンドルサイズ: チャンクサイズ・不要な依存・コード分割
   - 影響範囲: 変更の影響先・依存グラフ・importチェーン
   - 依存関係: バージョン・脆弱性・未使用パッケージ
3. 分析対象（特定のファイル・ディレクトリ・全体）

#### 完了チェックポイント（ステップ1）

- 分析の種類が1つ以上特定されていること
- 分析対象の範囲が明確であること

### ステップ2: 分析種別ごとの処理

特定された分析種類に応じて、以下の処理を実行する（複数種類の場合は並列で実行する）。

#### 分析A: コード品質分析

調査内容:
1. 複雑度の高いファイル・関数の特定
   - ファイルサイズ（行数）が300行を超えるファイルを探す
   - ネスト深度が4以上の条件分岐を探す
   - 引数が5つ以上の関数を探す
2. コードの重複検出
   - 類似したロジックが複数箇所に存在しないか確認する
   - コピペされたコードブロックを探す
3. 命名・構造の一貫性
   - 命名規約違反を探す
   - コンポーネントの責任分離が適切か確認する
4. 型安全性
   - `any` 型の使用箇所を探す
   - 型アサーション（`as`）の過度な使用を探す

使用ツール: Grep（パターン検索）、Glob（ファイル一覧）、Bash（行数カウント）

#### 分析B: パフォーマンス分析

調査内容:
1. レンダリングのボトルネック
   - 不必要な再レンダリングを引き起こすパターンを探す
   - useEffect/useCallbackの依存配列の問題を探す
   - 重い計算処理でuseMemoが使われていないケースを探す
2. N+1クエリ問題
   - ループ内でAPIコール・DB呼び出しが発生しているパターンを探す
   - ウォーターフォールなデータフェッチを探す
3. メモリリークリスク
   - クリーンアップされないイベントリスナーを探す
   - クリーンアップされないタイマー（setInterval/setTimeout）を探す
   - unmount後のsetStateを探す
4. 画像・アセット最適化
   - 最適化されていない画像（next/imageを使っていない等）を探す

使用ツール: Grep（パターン検索）、Read（ファイル詳細確認）

#### 分析C: バンドルサイズ分析

調査内容:
1. ビルド成果物の分析
   - `.next/` または `dist/` のチャンクファイルサイズを確認する
   - 大きなチャンク（500KB以上）を特定する
2. 重い依存パッケージの特定
   - package.jsonの全依存パッケージを確認する
   - バンドルサイズが大きい既知のパッケージを特定する（moment.js, lodash等）
   - tree-shakingが効かないインポートパターンを探す（`import * as`等）
3. コード分割の改善余地
   - dynamic import が使われていない大きなコンポーネントを探す
   - 初期ロード不要なモジュールを特定する
4. 未使用コードの検出
   - exportされているが使用されていない関数・コンポーネントを探す

使用ツール: Bash（ファイルサイズ確認）、Grep（importパターン）、Read（package.json）

#### 分析D: 影響範囲分析

調査内容:
1. 対象ファイル・モジュールのimportチェーン
   - 対象ファイルをimportしている全ファイルを特定する（直接・間接）
   - 影響を受けるページ・コンポーネントをリストアップする
2. 依存グラフの可視化
   - 対象→依存先（何を使っているか）
   - 依存元→対象（誰が使っているか）
3. 影響度の評価
   - 影響を受けるユーザーフロー（重要度で分類）
   - テストが必要な範囲を特定する
4. 変更による破壊的影響の特定
   - インターフェース変更が伝播する範囲
   - 型の変更が影響するファイル

使用ツール: Grep（import/require検索）、Glob（ファイル探索）

#### 分析E: 依存関係分析

調査内容:
1. バージョン管理の状態
   - package.jsonのdependencies/devDependenciesを確認する
   - 古いバージョンのパッケージを特定する
   - バージョン固定（exact version）vs レンジ指定の方針確認
2. 脆弱性チェック
   - `npm audit` / `pnpm audit` を実行して脆弱性を確認する
   - Critical/Highの脆弱性を優先的にリストアップする
3. 未使用パッケージの検出
   - package.jsonに記載されているがコード中で使われていないパッケージを探す
4. 重複・代替可能なパッケージの特定
   - 同じ機能を提供する複数のパッケージが入っていないか確認する
   - Node.js標準APIで代替できるパッケージを特定する

使用ツール: Bash（audit実行）、Read（package.json）、Grep（import検索）

#### 完了チェックポイント（ステップ2）

- 全ての指定された分析種別について調査が完了していること
- 各分析で問題箇所が特定されていること（問題なしの場合もその旨を記録）
- 改善提案の根拠となるデータが集まっていること

### 上流判断が必要な項目

該当なしの場合は「該当なし — 全て技術的改善で対応可能」と記載する。

| 項目 | 分類 | 判断が必要な内容 | 確認先 | 背景 |
|------|------|---------------|------|------|
| {問題の概要} | {仕様見直し/設計変更/アーキテクチャ判断} | {何を決める必要があるか} | {PO/デザイナー/テックリード等} | {なぜ技術的改善だけでは不十分か} |

分類の基準:
- 仕様見直し: 分析で発見した問題の根本原因が仕様の曖昧さ・矛盾にある場合（例: 複雑度が高い関数が、仕様自体の複雑さに起因する）
- 設計変更: 改善にUIやユーザー体験の変更が伴う場合（例: パフォーマンス改善のためにページネーション導入が必要だが、UX設計が未着手）
- アーキテクチャ判断: 改善に構造的な変更が必要で、技術的負債の返済方針を決める必要がある場合（例: N+1問題の根本解決にデータアクセス層の再設計が必要）

## Examples

### パフォーマンス分析の実行

ユーザー: 「DataTableコンポーネントのパフォーマンスを分析して」

```
ステップ1: 分析種類の特定
  分析種別: パフォーマンス
  対象: src/components/DataTable/

ステップ2: 分析処理
  - 分析B（パフォーマンス）を実行
  - useEffect依存配列に[]を指定しているが、内部でpropsを参照しているパターン発見
  - ループ内でAPI呼び出し（N+1）を発見
  - クリーンアップなしのsetIntervalを発見

ステップ3: レポート出力
  → .claude/artifacts/analysis-20260320-1530.md
  サマリー: 「3つの重大な問題を発見。N+1クエリが最優先。推定改善工数2時間」
```

### 依存関係と脆弱性チェック

ユーザー: 「プロジェクトの依存関係と脆弱性を確認して」

```
ステップ1: 分析種類の特定
  分析種別: 依存関係
  対象: プロジェクト全体

ステップ2: 分析処理
  - 分析E（依存関係）を実行
  - pnpm audit実行: Critical 2件、High 3件
  - 未使用パッケージ: moment（date-fnsで代替可能）
  - 重複: axios と node-fetch の両方が入っている

ステップ3: レポート出力
  → .claude/artifacts/analysis-20260320-1600.md
```
