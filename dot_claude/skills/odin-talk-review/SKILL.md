# odin-talk-review

多角的コードレビュースキル。odin司令塔のtalkフェーズで使用する。
PRの差分、ブランチの変更、指定ファイルを対象に、品質・セキュリティ・パフォーマンスの3軸で並列レビューを実行し、優先度付きで統合した結果を提供する。

## Instructions

### 完了チェックポイントの原則

各ステップの最後には「完了チェックポイント」を設けている。
チェックポイントに記載された全ての条件を満たさない限り、次のステップに進んではならない。

### コンテキスト検出

$ARGUMENTS を確認し、odinコンテキストJSONが含まれているか判定する。

odinコンテキストJSONの例:
```json
{
  "odin_context": {
    "task": "通知機能の実装レビュー",
    "review_target": "pr:123",
    "artifacts": {
      "design": ".claude/artifacts/design-20260320-1600.md"
    }
  }
}
```

レビュー対象の指定形式:
- `pr:{番号}` → GitHub PRの差分
- `branch:{ブランチ名}` → 指定ブランチとmainの差分
- `file:{パス}` → 特定ファイル
- 未指定 → ステップ1でAskUserQuestionで確認

- odinコンテキストがある場合: そこからレビュー対象を読み取る
- odinコンテキストがない場合: ステップ1からAskUserQuestionで確認する

### ステップ1: レビュー対象の特定

レビュー対象が不明な場合はAskUserQuestionで以下を確認する:

1. レビュー対象:
   - PR番号（例: #123）
   - ブランチ名（例: feature/notification）
   - ファイルパス（例: src/app/api/notifications/route.ts）
   - 現在ブランチのmainとの差分

2. レビューの焦点（複数選択可）:
   - 品質・保守性全般
   - セキュリティ重点
   - パフォーマンス重点
   - 設計・アーキテクチャ

対象別の差分取得方法:
```bash
# PR番号の場合
gh pr diff {PR番号}

# ブランチの場合
git diff $(git merge-base HEAD main)..HEAD

# ファイル指定の場合
# Readツールで直接ファイルを読み込む
```

変更ファイル一覧も取得する:
```bash
# ブランチの場合
git diff --name-only $(git merge-base HEAD main)..HEAD

# PRの場合
gh pr view {PR番号} --json files --jq '.files[].path'
```

#### 完了チェックポイント（ステップ1）

- レビュー対象の差分が取得できていること
- 変更ファイル一覧が把握できていること

### ステップ2: 並列レビューの実行

以下の4つのレビューを1回のレスポンスで同時起動する。

1. Agentツールで code-reviewer サブエージェントを起動（model: sonnet）
   - 変更差分を渡し、品質・保守性の観点でレビューを依頼する
   - コードの複雑さ、命名の適切さ、テストの有無を確認する

2. Agentツールで security-reviewer サブエージェントを起動（model: sonnet）
   - 変更差分を渡し、OWASP Top 10ベースでセキュリティレビューを依頼する
   - 認証・認可、入力バリデーション、秘密情報の扱いを確認する

3. Skillツールで `simplify` を実行
   - 変更コードの再利用性・効率性をチェックする

4. Agentツールで `coderabbit:code-reviewer` サブエージェントを起動
   - プラグインが未インストールの場合はスキップし、スキップした旨を記録する

全レビューの完了を待つ。

#### 完了チェックポイント（ステップ2）

- 4つのレビュー（またはスキップ理由の記録）が完了していること
- 全レビューの結果が得られていること

### ステップ3: 結果統合と優先度付け

4つのレビュー結果を統合し、以下の3段階で優先度付けを行う。

優先度の定義:
- 重大（即座に修正が必要）: セキュリティ脆弱性、バグ、データ損失リスク、テスト失敗
- 改善（推奨）: コード品質、保守性、パフォーマンス、テストカバレッジ
- 情報（参考）: スタイル、ベストプラクティス、将来の改善候補

重複指摘の統合:
- 複数のレビュアーが同じ問題を指摘した場合は1つにまとめ、指摘元を明記する
- 矛盾する指摘がある場合は両方を記載し、どちらを採用するか判断材料を提示する

#### 完了チェックポイント（ステップ3）

- 全指摘が3段階で優先度付けされていること
- 重複指摘が統合されていること

### ステップ4: フィードバックの提示

superpowers:receiving-code-review の原則に従い、技術的に正確なフィードバックを提供する。

> superpowersプラグインが未インストールの場合: 以下のフィードバック原則を直接適用する。

フィードバックの原則:
- 根拠のない賛辞は禁止（「良いコードですね」等）
- 全ての指摘には具体的な根拠を付ける
- 「なぜ問題か」と「どう修正するか」をセットで提示する
- 修正例はコードスニペットで示す（曖昧な説明ではなく具体的なコードで）
- 技術的に不明な点は「確認が必要」と正直に記載する

フィードバックの形式:

```
## レビュー結果サマリー

対象: {PR番号 / ブランチ名 / ファイル名}
変更ファイル数: {N}ファイル
総指摘数: {重大: N件, 改善: N件, 情報: N件}

---

### 重大な問題（即座に修正が必要）

#### [security] {問題の概要}
ファイル: {パス}:{行番号}
問題: {なぜ問題か}
修正例:
```{言語}
{修正後のコード}
```

#### [quality] {問題の概要}
...

---

### 改善提案（推奨）

#### [quality] {改善内容}
...

---

### 情報（参考）

- [{カテゴリ}] {内容}
...

---

### 総評

{全体的な品質評価と次のアクション}
```

カテゴリのタグ一覧:
- `[security]`: セキュリティ
- `[quality]`: コード品質
- `[perf]`: パフォーマンス
- `[test]`: テスト
- `[simplify]`: 簡略化・重複排除
- `[arch]`: アーキテクチャ・設計

#### 完了チェックポイント（ステップ4）

- フィードバックが3段階で整理されていること
- 全ての重大な問題に修正例が付いていること
- 根拠なき賛辞が含まれていないこと

## Examples

### PRのコードレビューを実行する

ユーザー: 「PR #42 をレビューして」

```
ステップ1: レビュー対象の特定
  gh pr diff 42 → 差分取得
  変更ファイル: src/app/api/notifications/route.ts, src/lib/notification.ts

ステップ2: 並列レビュー
  code-reviewer: 命名・テストカバレッジを確認
  security-reviewer: 入力バリデーション・認証を確認
  simplify: 重複コードのチェック
  coderabbit:code-reviewer: 全体的なレビュー

ステップ3: 統合・優先度付け
  重大: SQLインジェクション可能性（security + coderabbitが両方指摘）
  改善: エラーハンドリングが不足（quality）
  情報: 変数名をより説明的に（quality）

ステップ4: フィードバック提示
  重大問題に具体的な修正コードを付けて提示
```

### 現在のブランチ変更をレビューする

ユーザー: 「このブランチの変更をレビューして」

```
ステップ1: レビュー対象の特定
  git diff $(git merge-base HEAD main)..HEAD で差分取得

ステップ2〜4: 同様の流れでレビュー実行
```
