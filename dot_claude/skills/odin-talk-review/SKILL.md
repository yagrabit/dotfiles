# odin-talk-review

多角的コードレビュースキル。odin司令塔のtalkフェーズで使用する。
PRの差分、ブランチの変更、指定ファイルを対象に、品質・セキュリティ・パフォーマンスの3軸で並列レビューを実行し、優先度付きで統合した結果を提供する。

## Instructions

### 完了チェックポイントの原則

各ステップの最後には「完了チェックポイント」を設けている。
チェックポイントに記載された全ての条件を満たさない限り、次のステップに進んではならない。

### コンテキスト検出

$ARGUMENTS を確認し、odinコンテキストJSONが含まれているか判定する。

odinコンテキストJSONの例:
```json
{
  "odin_context": {
    "task": "通知機能の実装レビュー",
    "review_target": "pr:123",
    "artifacts": {
      "design": ".claude/artifacts/design-20260320-1600.md"
    }
  }
}
```

レビュー対象の指定形式:
- `pr:{番号}` → GitHub PRの差分
- `branch:{ブランチ名}` → 指定ブランチとmainの差分
- `file:{パス}` → 特定ファイル
- 未指定 → ステップ1でAskUserQuestionで確認

- odinコンテキストがある場合: そこからレビュー対象を読み取る
- odinコンテキストがない場合: ステップ1からAskUserQuestionで確認する

### ステップ1: レビュー対象の特定

レビュー対象が不明な場合はAskUserQuestionで以下を確認する:

1. レビュー対象:
   - PR番号（例: #123）
   - ブランチ名（例: feature/notification）
   - ファイルパス（例: src/app/api/notifications/route.ts）
   - 現在ブランチのmainとの差分

2. レビューの焦点（複数選択可）:
   - 品質・保守性全般
   - セキュリティ重点
   - パフォーマンス重点
   - 設計・アーキテクチャ

対象別の差分取得方法:
```bash
# PR番号の場合
gh pr diff {PR番号}

# ブランチの場合
git diff $(git merge-base HEAD main)..HEAD

# ファイル指定の場合
# Readツールで直接ファイルを読み込む
```

変更ファイル一覧も取得する:
```bash
# ブランチの場合
git diff --name-only $(git merge-base HEAD main)..HEAD

# PRの場合
gh pr view {PR番号} --json files --jq '.files[].path'
```

#### 完了チェックポイント（ステップ1）

- レビュー対象の差分が取得できていること
- 変更ファイル一覧が把握できていること

### コンテキスト隔離の原則

VCSDDのAdversary設計に倣い、レビューエージェントにはコンテキスト隔離を徹底する:

- レビューエージェントは必ず新しいAgentツールで起動する（同一コンテキストで実行しない）
- レビューエージェントには変更差分のみを渡す。実装時の意図・経緯・会話履歴は渡さない
- レビュー結果は「コードの品質」のみで判断する（実装者の意図で正当化しない）
- これにより「エントロピー抵抗の欠如」（会話が長いほどAIが正当化する傾向）を構造的に排除する

渡してよい情報:
- git diffの出力（変更差分）
- 変更ファイル一覧
- プロジェクトのCLAUDE.mdに記載されたコーディング規約

渡してはいけない情報:
- 実装時の設計判断の経緯
- ユーザーとの会話履歴
- 「この変更は〇〇のために行った」等の意図説明

### ステップ2: 並列レビューの実行

以下の5つのレビューを1回のレスポンスで同時起動する。

1. Agentツールで code-reviewer サブエージェントを起動（model: sonnet）
   - 変更差分のみを渡し、品質・保守性の観点でレビューを依頼する
   - コードの複雑さ、命名の適切さ、テストの有無を確認する

2. Agentツールで security-reviewer サブエージェントを起動（model: sonnet）
   - 変更差分のみを渡し、OWASP Top 10ベースでセキュリティレビューを依頼する
   - 認証・認可、入力バリデーション、秘密情報の扱いを確認する

3. Skillツールで `simplify` を実行
   - 変更コードの再利用性・効率性をチェックする

4. Agentツールで subagent_type に "coderabbit:code-reviewer" を指定して起動する
   - スキップ条件: system-reminderのスキル一覧に `coderabbit:` プレフィックスのスキルが存在しない場合のみスキップ可。存在する場合は必ず実行する
   - スキップ時は「coderabbit:プレフィックスのスキルがsystem-reminderに見つからないためスキップ」と記録する

5. Skillツールで `codex:review` を実行する（Codex経由のレビュー）
   - `--base main --wait` オプションを付けて実行する
   - スキップ条件: system-reminderのスキル一覧に `codex:` プレフィックスのスキルが存在しない場合のみスキップ可。存在する場合は必ず実行する
   - スキップ時は「codex:プレフィックスのスキルがsystem-reminderに見つからないためスキップ」と記録する

全レビューの完了を待つ。

#### 完了チェックポイント（ステップ2）

- 5つのレビューが全て実行されていること（スキップは「プラグイン未インストール」の場合のみ許可。利用可能なプラグインのスキップは違反）
- スキップした場合は具体的な理由（「system-reminderにプレフィックスが見つからない」等）が記録されていること
- 全レビューの結果が得られていること

### ステップ3: 結果統合と優先度付け

5つのレビュー結果を統合し、以下の3段階で優先度付けを行う。

優先度の定義:
- 重大（即座に修正が必要）: セキュリティ脆弱性、バグ、データ損失リスク、テスト失敗
- 改善（推奨）: コード品質、保守性、パフォーマンス、テストカバレッジ
- 情報（参考）: スタイル、ベストプラクティス、将来の改善候補

重複指摘の統合:
- 複数のレビュアーが同じ問題を指摘した場合は1つにまとめ、指摘元を明記する
- 矛盾する指摘がある場合は両方を記載し、どちらを採用するか判断材料を提示する

#### 完了チェックポイント（ステップ3）

- 全指摘が3段階で優先度付けされていること
- 重複指摘が統合されていること

### ステップ4: フィードバックの提示

superpowers:receiving-code-review の原則に従い、技術的に正確なフィードバックを提供する。

> superpowersプラグインが未インストールの場合: 以下のフィードバック原則を直接適用する。

フィードバックの原則:
- 根拠のない賛辞は禁止（「良いコードですね」等）
- 全ての指摘には具体的な根拠を付ける
- 「なぜ問題か」と「どう修正するか」をセットで提示する
- 修正例はコードスニペットで示す（曖昧な説明ではなく具体的なコードで）
- 技術的に不明な点は「確認が必要」と正直に記載する

フィードバックの形式:

```
## レビュー結果サマリー

対象: {PR番号 / ブランチ名 / ファイル名}
変更ファイル数: {N}ファイル
総指摘数: {重大: N件, 改善: N件, 情報: N件}

---

### 重大な問題（即座に修正が必要）

#### [security] {問題の概要}
ファイル: {パス}:{行番号}
問題: {なぜ問題か}
修正例:
```{言語}
{修正後のコード}
```

#### [quality] {問題の概要}
...

---

### 改善提案（推奨）

#### [quality] {改善内容}
...

---

### 情報（参考）

- [{カテゴリ}] {内容}
...

---

### 総評

{全体的な品質評価と次のアクション}
```

カテゴリのタグ一覧:
- `[security]`: セキュリティ
- `[quality]`: コード品質
- `[perf]`: パフォーマンス
- `[test]`: テスト
- `[simplify]`: 簡略化・重複排除
- `[arch]`: アーキテクチャ・設計
- `[codex]`: Codexレビューからの指摘

#### 完了チェックポイント（ステップ4）

- フィードバックが3段階で整理されていること
- 全ての重大な問題に修正例が付いていること
- 根拠なき賛辞が含まれていないこと

### ステップ4.5: レビュー完了フラグの記録

pre-push-review-gateフックと連携するため、レビュー完了フラグを作成する。
フィーチャーブランチ上でレビューを実施した場合のみ実行する。

```bash
# レビュー完了フラグを作成（push前のゲートチェック用）
mkdir -p /tmp/claude-sessions
date +%s > "/tmp/claude-sessions/review-passed-$(git branch --show-current | tr '/' '-')"
```

mainブランチの場合はスキップする（pushゲートはmainには適用されないため）。

### ステップ 5: 学習ポイントの自動提示

レビュー結果の提示後、以下の条件を満たす場合に「学びポイント」を自動で総評に含める:

条件:
- レビュー結果に改善提案または情報が1件以上含まれている
- odinコンテキストに `skip_learning: true` が設定されていない

5-1. レビュー結果から教育的価値の高い指摘を最大2つ選び、総評セクションの末尾に以下を追記する:

```
--- 学びポイント ---
1. [{スキル領域}] {なぜこのパターンが良い/悪いのか、1-2文の解説}
2. [{スキル領域}] {なぜこのパターンが良い/悪いのか、1-2文の解説}
```

学びポイントの選定基準:
- 表面的な指摘（命名、フォーマット等）ではなく、設計原則やアーキテクチャに関わるもの
- ユーザーのスキルマップ（~/.claude/artifacts/skillmap-latest.md）があれば、目標レベルとの差が大きい領域を優先
- スキルマップがなくても学びポイントは提示する（スキルマップなしでも動作する）

5-2. 学びポイント提示後、AskUserQuestionで以下を提示する:
```
学びポイントについてソクラテス式で深掘りしますか?

1. 深掘りする（対話で「なぜ」を理解し、スキルマップに反映）
2. 今回はスキップ
```

選択肢1の場合: Skillツールで `odin-learn-review` を実行する。引数にレビュー結果と対象PRの情報を渡す。ソクラテス式対話での回答精度に基づいてスキルマップを更新する（理解を確認できた場合のみレベルアップ）。
選択肢2の場合: そのまま終了する。スキルマップは更新しない（学びポイントの表示だけでは理解の確認ができないため）。

## Examples

### PRのコードレビューを実行する

ユーザー: 「PR #42 をレビューして」

```
ステップ1: レビュー対象の特定
  gh pr diff 42 → 差分取得
  変更ファイル: src/app/api/notifications/route.ts, src/lib/notification.ts

ステップ2: 並列レビュー
  code-reviewer: 命名・テストカバレッジを確認
  security-reviewer: 入力バリデーション・認証を確認
  simplify: 重複コードのチェック
  coderabbit:code-reviewer: 全体的なレビュー

ステップ3: 統合・優先度付け
  重大: SQLインジェクション可能性（security + coderabbitが両方指摘）
  改善: エラーハンドリングが不足（quality）
  情報: 変数名をより説明的に（quality）

ステップ4: フィードバック提示
  重大問題に具体的な修正コードを付けて提示
```

### 現在のブランチ変更をレビューする

ユーザー: 「このブランチの変更をレビューして」

```
ステップ1: レビュー対象の特定
  git diff $(git merge-base HEAD main)..HEAD で差分取得

ステップ2〜4: 同様の流れでレビュー実行
```
