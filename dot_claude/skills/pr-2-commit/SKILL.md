以下の手順で変更をコミットしてください。

## 1. 変更の確認

- `git status` で未追跡ファイル・変更ファイルを一覧取得する
- `git diff` と `git diff --cached` でステージ済み・未ステージの差分を確認する
- `git log --oneline -5` で直近のコミットメッセージのスタイルを確認する

## 2. セキュリティチェック

コミット対象の全ファイルについて、公開リポジトリに含めてはいけない機密情報がないか確認する。

チェック対象:
- APIキー、トークン、シークレット
- パスワード、認証情報のハードコード
- 個人のメールアドレス（noreplyアドレスやダミーは除く）
- 秘密鍵、証明書
- 内部URL、プライベートIPアドレス
- .env ファイルや credentials ファイル

問題が見つかった場合:
- 該当箇所をユーザーに報告し、コミット対象から除外するか確認する
- 機密情報を環境変数やキーチェーンからの動的取得に置き換える方法を提案する

## 3. コミット計画の作成

変更内容を分析し、論理的にまとまった単位でコミットグループに分割する。

グループ分割の方針:
- 同じ機能・目的に関連するファイルをまとめる
- 設定ファイルの変更はまとめる
- テストファイルは対応する実装ファイルと同じコミットに含める
- 不要ファイルの削除はまとめる
- .env, credentials, secrets など機密ファイルが含まれていたら警告し、コミット対象から除外する

コミットメッセージの規約:
- Conventional Commits 形式の Prefix (feat:, fix:, docs:, refactor:, chore: など)
- 日本語で記述し「なぜ」を重視する

## 4. 計画の提示と承認

以下の形式でコミット計画を表示し、AskUserQuestion でユーザーの承認を得る:

```
コミット計画:

[1] feat: 〇〇を追加
  - src/app/page.tsx
  - src/components/foo.tsx

[2] chore: 不要ファイルを削除
  - public/old-file.svg
```

承認されなかった場合は、ユーザーのフィードバックに基づいて計画を修正し、再度承認を求める。

## 5. コミットの実行

承認後、計画通りの順番でコミットを実行する:
- 各コミットグループのファイルを `git add` でステージング（ファイル名を個別に指定）
- `git commit` でコミット（フックをスキップしない）
- フックが失敗した場合は問題を修正して新しいコミットを作成する（--amend しない）
- コミットメッセージは HEREDOC 形式で渡す

## 6. 結果の報告

全コミット完了後、`git log --oneline` で結果を表示して報告する。
