# Design

明確化された要件とコードベース調査結果をもとに、設計ドキュメントを作成するスキル。
既存プロジェクトでは差分設計（変更箇所のみ）、新規プロジェクトではフル設計を行う。

## Instructions

### 完了チェックポイントの原則

各ステップの最後には「完了チェックポイント」を設けている。
チェックポイントに記載された全ての条件を満たさない限り、次のステップに進んではならない。

### ステップ1: 入力ファイルの読み込みと設計モード判定

1. `.claude/artifacts/` から以下のファイルを探す:
   - `clarified-requirements-*.md`（必須）
   - `research-*.md`（任意: 存在すれば既存プロジェクト）
   - $ARGUMENTSでパスが指定されていればそれを使用
2. 設計モードを判定する:
   - `research-*.md` が存在する → 差分設計モード（既存プロジェクトへの機能追加）
   - `research-*.md` が存在しない → フル設計モード（新規プロジェクト）
3. 要件のリスクレベルを確認する（clarified-requirements-*.md に記載）

#### 完了チェックポイント（ステップ1）

- clarified-requirements-*.md が読み込まれていること
- 設計モード（差分/フル）が決定していること
- リスクレベルが確認されていること

### ステップ2: 設計の作成

設計モードに応じて以下を実行する。各セクションの調査はExploreエージェントに委譲してよい。

差分設計モード（既存プロジェクト）:

1. 変更が必要なファイルの特定
   - research-*.md の「関連する既存実装」を参照
   - 追加/変更/削除されるファイルを一覧化
2. 既存の型定義・インターフェースへの影響分析
   - 変更が必要な型、新規作成する型
3. API変更（該当する場合）
   - 追加/変更されるエンドポイント、リクエスト/レスポンスの型
4. コンポーネント変更（該当する場合）
   - 新規/変更コンポーネントとそのprops
   - 既存コンポーネントの再利用計画
5. 状態管理の変更（該当する場合）
   - 新規/変更されるstate、store
6. DBスキーマの変更（該当する場合）
   - マイグレーション内容

フル設計モード（新規プロジェクト）:

1. ディレクトリ構成の設計
2. データモデルの設計
3. API設計（エンドポイント一覧、リクエスト/レスポンス）
4. コンポーネント設計（コンポーネントツリー、props定義）
5. 状態管理の設計
6. ルーティング設計

#### 完了チェックポイント（ステップ2）

- 設計モードに応じた全セクションが記述されていること
- 変更の影響範囲が明確であること

### ステップ3: ADR作成（必要な場合）

以下のいずれかに該当する場合、ADR（Architecture Decision Record）を作成する:
- 複数の技術的選択肢があり、いずれかを選択した
- 既存のアーキテクチャパターンから逸脱する
- チーム内で議論が必要な設計判断がある

ADRフォーマット:
```
# ADR-{NNN}: {タイトル}

## ステータス
提案

## コンテキスト
{この判断が必要になった背景}

## 決定
{選択した方針}

## 選択肢
{検討した選択肢の一覧と各々の利点・欠点}

## 結果
{この決定による影響}
```

ADRが不要な場合はこのステップをスキップする。

#### 完了チェックポイント（ステップ3）

- 重要な設計判断がある場合はADRが作成されていること
- ADRが不要な場合はスキップの判断が適切であること

### ステップ4: 設計ドキュメントの出力

1. `.claude/artifacts/` ディレクトリが存在しなければ作成する
2. 設計ドキュメントを出力する:

出力先: `.claude/artifacts/design-{yyyyMMdd-HHmm}.md`

```
# 設計ドキュメント

作成日: {yyyy-MM-dd}
設計モード: {差分設計 / フル設計}
対象要件: {clarified-requirements-*.md のパス}
リスクレベル: {高/中/低}

## 概要

{設計の全体像を2-3文で}

## 変更対象ファイル一覧

| ファイルパス | 操作 | 概要 |
|---|---|---|
| {パス} | {追加/変更/削除} | {変更内容} |

## 型定義・インターフェース

{新規/変更される型定義}

## API設計

{エンドポイント、リクエスト/レスポンス}

## コンポーネント設計

{コンポーネント構成、props}

## 状態管理

{state、storeの変更}

## DBスキーマ（該当する場合）

{マイグレーション内容}

## 設計判断

{主要な設計判断の概要。ADRがある場合はパスを参照}
```

3. ADRがある場合は `.claude/artifacts/adr-{NNN}-{title}.md` も出力する

#### 完了チェックポイント（ステップ4）

- `.claude/artifacts/design-*.md` が出力されていること
- 変更対象ファイル一覧が記載されていること

### ステップ5: 設計レビュー（ゲート1）

1. 設計ドキュメントの要約をユーザーに提示する
2. AskUserQuestionで以下の選択肢を提示する:
   - 承認する: 設計に問題なし、タスク分解に進む
   - 修正する: 指摘箇所を修正してから再度レビューする（ステップ2に戻る）
   - 中止する: ワークフローを中止する
3. リスクレベルに応じたレビュー深度の案内:
   - 高リスク: 全セクションの詳細レビューを推奨
   - 中リスク: 変更対象ファイル一覧と型定義の確認を推奨
   - 低リスク: 概要の確認で十分

#### 完了チェックポイント（ステップ5）

- ユーザーが「承認する」を選択していること
- 「修正する」の場合はステップ2に戻って修正が完了していること

## Examples

### 既存Next.jsプロジェクトに通知機能を追加（差分設計）

ユーザー: 「設計して」（research-*.mdとclarified-requirements-*.mdが既にある状態）

```
ステップ1: 入力読み込み
  clarified-requirements-20260320-1500.md → 通知機能の要件
  research-20260320-1430.md → Next.js 14, App Router, Resend利用中
  設計モード: 差分設計
  リスクレベル: 中

ステップ2: 差分設計
  変更ファイル:
    追加: src/app/api/notifications/route.ts（API）
    追加: src/components/notification/NotificationBell.tsx（UI）
    追加: src/components/notification/NotificationList.tsx（UI）
    追加: src/lib/notifications.ts（ビジネスロジック）
    追加: src/types/notification.ts（型定義）
    変更: src/components/layout/Header.tsx（通知ベル追加）
    追加: prisma/migrations/xxx（DBスキーマ）

ステップ3: ADR
  ADR-001: リアルタイム通知の実装方式
    SSE vs WebSocket → SSEを選択（サーバーサイドの複雑さが低い）

ステップ4: 出力
  → .claude/artifacts/design-20260320-1530.md
  → .claude/artifacts/adr-001-realtime-notification-method.md

ステップ5: 設計レビュー（ゲート1）
  リスクレベル: 中 → 変更ファイル一覧と型定義の確認を推奨
  → ユーザー承認
```

### 新規プロジェクトでダッシュボードを設計（フル設計）

ユーザー: 「設計ドキュメントを作って」（clarified-requirements-*.mdのみある状態）

```
ステップ1: 入力読み込み
  clarified-requirements-20260320-1600.md → ダッシュボード要件
  research-*.md → なし
  設計モード: フル設計
  リスクレベル: 中

ステップ2: フル設計
  ディレクトリ構成、データモデル、API、コンポーネント、状態管理、ルーティングを設計

ステップ3: ADR不要

ステップ4: 出力
  → .claude/artifacts/design-20260320-1700.md

ステップ5: 設計レビュー（ゲート1）
  → ユーザー承認
```
