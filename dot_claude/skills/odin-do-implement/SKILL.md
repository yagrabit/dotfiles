# odin-do-implement

TDD実装スキル。odin司令塔のdoフェーズで使用する。
実装計画（plan-*.md）を入力として、タスクをTDDサイクル（Red→Green→Refactor）で実装する。
タスクの特性に応じて単一実行・並列実行・逐次実行を使い分ける。

## ガードレール（安全制御）

このスキルは以下の安全制御を遵守する:

- git push / git commit は実行しない（それぞれodin-do-pr / odin-do-commitスキルの責務）
- `git reset --hard` / `git checkout .` / `git clean -f` 等の破壊的操作を実行しない
- 実装中にファイルを元に戻す必要がある場合は、Editツールで手動修正するか、AskUserQuestionでユーザーに確認する

## Instructions

### 完了チェックポイントの原則

各ステップの最後には「完了チェックポイント」を設けている。
チェックポイントに記載された全ての条件を満たさない限り、次のステップに進んではならない。

### コンテキスト検出

$ARGUMENTS を確認し、odinコンテキストJSONが含まれているか判定する。

odinコンテキストJSONの例:
```json
{
  "odin_context": {
    "task": "通知機能の追加",
    "artifacts": {
      "plan": ".claude/artifacts/plan-20260320-1700.md",
      "design": ".claude/artifacts/design-20260320-1600.md"
    }
  }
}
```

- odinコンテキストがある場合: artifactsから入力ファイルのパスを取得する
- odinコンテキストがない場合: 成果物管理ルールに従い入力ファイルを探索する

### 成果物管理

入力ファイルの探索順序:
1. $ARGUMENTSにファイルパスが明示指定されていればそれを使う
2. odinコンテキストのartifactsにパスがあればそれを使う
3. `.claude/artifacts/` 内の最新の `plan-*.md` を使う
4. 該当ファイルが見つからない場合はAskUserQuestionでユーザーに確認する

### ステップ1: 実装計画の読み込みとタスク特定

1. 実装計画（plan-*.md）を読み込む
2. タスク一覧・依存関係・Wave形式並列実行計画を把握する
3. 実装対象タスクを特定する:
   - $ARGUMENTSに特定タスクID（例: T-001）が指定されていればそれを対象にする
   - 指定がない場合: Wave 1の未完了タスクを対象にする
4. 対象タスクのTDDフェーズ・担当ファイル・完了基準を確認する

#### 完了チェックポイント（ステップ1）

- 実装計画の内容を把握していること
- 対象タスクが特定されていること
- 各タスクのTDDフェーズと担当ファイルが明確になっていること

### ステップ2: 実行方式の決定

対象タスクの特性を分析し、実行方式を決定する:

単一タスクの場合:
- 直接TDDサイクルを実行する（ステップ3へ）

複数の独立タスク（並列実行可能）の場合:
- superpowers:dispatching-parallel-agentsを使用する
- 各エージェントに対象タスクのコンテキスト・ファイルパス・完了基準を渡す
- 全エージェントの完了を待ってから結果を統合する
- UIコンポーネントの並列実装時はspec fileパターンを適用する:
  1. design-*.md のコンポーネント設計テーブルから、各コンポーネントの仕様を抽出する
  2. 各エージェントのプロンプトに仕様をインライン化する（「ファイルを読め」ではなく内容を直接渡す）
  3. 仕様が150行を超える場合はコンポーネントをさらに分割する
  4. インタラクションモデル（クリック駆動 vs スクロール駆動）を仕様に必ず含める

複数の依存タスク（逐次実行必要）の場合:
- superpowers:subagent-driven-developmentを使用する
- タスクをWave順に1つずつ実行する
- 各タスク完了後にテストが通ることを確認してから次へ進む

> superpowersプラグインが未インストールの場合: Agentツールで直接サブエージェントを起動し、同等の並列/逐次実行を手動で行う。

#### 完了チェックポイント（ステップ2）

- 実行方式（直接TDD / 並列 / 逐次）が決定されていること
- 並列実行の場合は各エージェントへの渡す情報が整理されていること

### ステップ3: TDDサイクルの実行

#### テストコマンドの自動検出

TDDサイクル開始前に、プロジェクトのテストコマンドを自動検出する:

1. package.jsonの "scripts.test" を確認 → あればそのコマンドを使用
2. vitest.config.* の存在確認 → あれば `pnpm vitest run`
3. jest.config.* の存在確認 → あれば `pnpm jest`
4. Cargo.toml の存在確認 → あれば `cargo test`
5. pytest.ini / pyproject.toml の [tool.pytest] 確認 → あれば `pytest`
6. 上記いずれにも該当しない場合はAskUserQuestionでテストコマンドを確認する

E2Eテストコマンドの自動検出:

1. playwright.config.* の存在確認 → あれば `pnpm playwright test`
2. cypress.config.* の存在確認 → あれば `pnpm cypress run`
3. 上記いずれにも該当しない場合はE2Eテストをスキップする

以降のステップでは、検出したテストコマンドを使用する。

superpowers:test-driven-development のパターンに従い実行する。

> superpowersプラグインが未インストールの場合: 以下のRed→Green→Refactorサイクルを直接実行する。

#### 仕様疑義の検出と対応

TDDサイクルの各フェーズ（特にRedフェーズでテストを書く際）に、設計ドキュメント（design-*.md）や実装計画（plan-*.md）の記述と実際のコードベースとの矛盾・曖昧点に気づくことがある。
以下の兆候を検出した場合、実装を一時停止してエスカレーションする:

検出すべき兆候:
- 設計ドキュメントに記載された振る舞いが曖昧で、テストの期待値を一意に決められない
- 設計ドキュメントの記述と既存コードの動作が矛盾している
- エッジケースの振る舞いが設計ドキュメントに定義されていない
- 修正・実装によってUIや操作フローが変わるが、デザイン指示がない
- 他モジュール・他チームのAPIとの契約が変わる可能性がある

検出時の対応:
1. 疑義の内容を記録する（何が曖昧/矛盾しているか、どの設計箇所か）
2. 自明な仮定で進められる場合: 仮定を明記してTODOコメント（`// TODO: 仕様確認待ち — {疑義内容}`）を残し、実装を続行する
3. 仮定で進められない場合（複数の解釈で実装が大きく分岐する場合）: AskUserQuestionでユーザーにエスカレーションする。質問には「何が曖昧か」「どの選択肢があるか」「誰に確認すべきか」を含める
4. エスカレーションした疑義は、タスク完了時のレポートに「仕様疑義一覧」として集約する

#### Red フェーズ（失敗するテストを書く）

1. テストファイルを作成・開く
2. 実装したい振る舞いを表すテストコードを書く
3. テストの期待値を設計ドキュメントから導出する。一意に決められない場合は「仕様疑義の検出と対応」に従う
4. テストを実行し、失敗（Red）を確認する
5. テストが正しい理由で失敗していることを確認する（コンパイルエラーではなくアサーション失敗）
6. Red証拠ログを保存する（TDD実施の証拠）:
   ```bash
   # Redフェーズの証拠を保存（テスト失敗の出力をログに記録）
   mkdir -p .claude/artifacts/evidence
   {テストコマンド} 2>&1 | tail -50 > .claude/artifacts/evidence/red-{task-id}.log
   echo "phase: red" >> .claude/artifacts/evidence/red-{task-id}.log
   echo "timestamp: $(date -u +%Y-%m-%dT%H:%M:%SZ)" >> .claude/artifacts/evidence/red-{task-id}.log
   ```
   - `{task-id}` はodinコンテキストのtask_id、またはタスク名をkebab-caseにしたもの
   - ログが50行を超える場合は末尾50行を保存する（全量保存は不要）

```bash
# 自動検出したテストコマンドで対象テストファイルを実行する
```

#### Green フェーズ（テストを通す最小実装）

1. テストを通す最小限のコードを書く
2. 過剰な最適化・抽象化はしない（まず動かすことを優先）
3. テストを実行し、成功（Green）を確認する
4. 既存のテストが壊れていないことを確認する（全テスト実行）
5. Green証拠ログを保存する:
   ```bash
   # Greenフェーズの証拠を保存（テスト成功の出力をログに記録）
   {テストコマンド} 2>&1 | tail -50 > .claude/artifacts/evidence/green-{task-id}.log
   echo "phase: green" >> .claude/artifacts/evidence/green-{task-id}.log
   echo "timestamp: $(date -u +%Y-%m-%dT%H:%M:%SZ)" >> .claude/artifacts/evidence/green-{task-id}.log
   ```

```bash
# 全テスト実行（自動検出したテストコマンドを使用）
```

#### Refactor フェーズ（コードの整理）

1. テストが通ったまま、コードの品質を改善する
2. 重複の除去・命名の改善・責務の明確化を行う
3. リファクタリング後もテストが通ることを確認する
4. コードコメントを日本語で追加する（複雑なロジックには必ず）

#### コード品質ガードレール

TDDサイクルの各Green/Refactorフェーズ完了時に、以下の品質チェックを実施する。これらはClaudeがコード生成時に繰り返しやすい問題パターンである:

#### フレームワーク固有ルール（該当する場合のみ適用）

以下はプロジェクトのpackage.jsonにNext.js/React依存が含まれる場合のみ適用する:
- ページコンポーネントのメタデータは `generateMetadata` 関数で辞書から取得しているか（静的 `metadata` 定数のハードコードは禁止）
- ユーザー表示テキストが辞書ファイル経由になっているか（ハードコード文字列の禁止）
- 複数の独立したデータ取得がある場合、各クエリの loading/error 状態を独立管理しているか（`loading = aLoading || bLoading` の合成は禁止）
- `'use client'` が本当に必要な場合のみ付与されているか

テスト固有:
- useGraphQL等のモックが呼び出し順序（callCount）に依存していないか（クエリ名ベースのディスパッチを使う）
- GraphQL型を使うテストでは `pnpm codegen` 実行が前提であることを確認しているか

セキュリティ固有:
- ペイロードサイズ制限に `String.length` を使っていないか（UTF-16文字数であり、マルチバイト文字でバイト数と乖離する。`new TextEncoder().encode(str).length` を使う）
- `target="_blank"` に `rel="noopener noreferrer"` が付与されているか

型安全性:
- 同じ概念の型が複数箇所で定義されていないか（1箇所で定義し re-export する）
- `any` 型が使用されていないか

UIコンポーネント固有（UIを含むタスクの場合）:
- レスポンシブデザイン: モバイルファーストで設計されているか（min-width ブレークポイント使用）
- アクセシビリティ: インタラクティブ要素に適切なARIA属性・キーボード操作・フォーカス管理があるか
- コンポーネント状態: 全状態（ローディング、エラー、空、無効）のUIが実装されているか
- ダークモード: CSS変数・セマンティックトークンを使用し、ハードコードされた色値がないか

各サイクル完了後:
- タスクの完了基準を満たしているか確認する
- 満たしていれば次のタスクへ進む

#### 完了チェックポイント（ステップ3）

- Red: テストが失敗していること（アサーション失敗であること）
- Green: テストが全て通っていること
- Refactor: コードが整理されており、テストが引き続き通っていること
- タスクの完了基準が全て満たされていること

### ステップ4: タスク完了の検証

superpowers:verification-before-completion のパターンに従い検証する。

> superpowersプラグインが未インストールの場合: 以下の検証手順を直接実行する。

1. 全テストの通過確認:
   ```bash
   # 自動検出したテストコマンドで全テストを実行する
   ```

2. ビルドの成功確認:
   ```bash
   pnpm build
   # または
   pnpm typecheck
   ```

3. lintエラーの確認:
   ```bash
   pnpm lint
   # または
   pnpm oxlint
   ```

4. 実装漏れの確認:
   - タスクの完了基準を1つずつ確認する
   - 設計ドキュメントのコンポーネント設計と実装が対応しているか確認する

5. 仕様疑義の集約:
   - `// TODO: 仕様確認待ち` でGrepし、TDDサイクル中に残したTODOを収集する
   - エスカレーション済み事項（AskUserQuestionで確認した内容と回答）を合わせて一覧にする
   - 各疑義について「仮定で続行（仮定内容を明記）」と「未解決（確認先を明記）」に分類する
   - odinから呼ばれた場合は、完了状態に加えて仕様疑義一覧を返す

6. 次のWaveへの通知:
   - 現在のWaveの全タスクが完了したことを確認する
   - odinから呼ばれた場合は、完了状態と次のWaveのタスク一覧を返す

#### 完了チェックポイント（ステップ4）

- 全テストが通過していること（自動検出したテストコマンドが成功）
- ビルドが成功していること（型エラーなし）
- lintエラーがないこと
- タスクの完了基準が全て満たされていること
- 仕様疑義がある場合: 一覧がまとめられ、各項目に「仮定で続行」または「未解決（要確認先を明記）」のステータスが付いていること

## Examples

### Wave 1 のタスクを並列で実装する

ユーザー: 「plan-20260320-1700.md のWave 1を実装して」

```
ステップ1: 実装計画読み込み
  Wave 1: T-001（型定義）, T-002（DBスキーマ）- 独立タスク

ステップ2: 実行方式決定
  → 複数の独立タスク → dispatching-parallel-agents を使用

ステップ3: 並列実行
  Agent 1: T-001 型定義を TDD で実装
    Red: 型を使うテストを書く → 失敗
    Green: 型を定義する → 通過
    Refactor: JSDoc追加、命名改善
  Agent 2: T-002 DBスキーマを TDD で実装
    Red: マイグレーションテストを書く → 失敗
    Green: スキーマ定義 → 通過
    Refactor: コメント追加

ステップ4: 検証
  全テスト通過・ビルド成功・lintエラーなし → 完了
```

### 単一タスクを TDD で実装する

ユーザー: 「T-004（通知作成APIの実装）を実装して」

```
ステップ1: 計画確認
  T-004: Green フェーズ、src/app/api/notifications/route.ts
  依存: T-002（テスト）完了済み

ステップ2: 直接TDD実行

ステップ3: Green フェーズ
  既存テスト（T-002で作成）を確認
  テストを通す最小実装を行う
  全テスト通過を確認

ステップ4: 検証完了
```
