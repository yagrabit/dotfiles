# odin-do-implement

TDD実装スキル。odin司令塔のdoフェーズで使用する。
実装計画（plan-*.md）を入力として、タスクをTDDサイクル（Red→Green→Refactor）で実装する。
タスクの特性に応じて単一実行・並列実行・逐次実行を使い分ける。

## Instructions

### 完了チェックポイントの原則

各ステップの最後には「完了チェックポイント」を設けている。
チェックポイントに記載された全ての条件を満たさない限り、次のステップに進んではならない。

### コンテキスト検出

$ARGUMENTS を確認し、odinコンテキストJSONが含まれているか判定する。

odinコンテキストJSONの例:
```json
{
  "odin_context": {
    "task": "通知機能の追加",
    "artifacts": {
      "plan": ".claude/artifacts/plan-20260320-1700.md",
      "design": ".claude/artifacts/design-20260320-1600.md"
    }
  }
}
```

- odinコンテキストがある場合: artifactsから入力ファイルのパスを取得する
- odinコンテキストがない場合: 成果物管理ルールに従い入力ファイルを探索する

### 成果物管理

入力ファイルの探索順序:
1. $ARGUMENTSにファイルパスが明示指定されていればそれを使う
2. odinコンテキストのartifactsにパスがあればそれを使う
3. `.claude/artifacts/` 内の最新の `plan-*.md` を使う
4. 該当ファイルが見つからない場合はAskUserQuestionでユーザーに確認する

### ステップ1: 実装計画の読み込みとタスク特定

1. 実装計画（plan-*.md）を読み込む
2. タスク一覧・依存関係・Wave形式並列実行計画を把握する
3. 実装対象タスクを特定する:
   - $ARGUMENTSに特定タスクID（例: T-001）が指定されていればそれを対象にする
   - 指定がない場合: Wave 1の未完了タスクを対象にする
4. 対象タスクのTDDフェーズ・担当ファイル・完了基準を確認する

#### 完了チェックポイント（ステップ1）

- 実装計画の内容を把握していること
- 対象タスクが特定されていること
- 各タスクのTDDフェーズと担当ファイルが明確になっていること

### ステップ2: 実行方式の決定

対象タスクの特性を分析し、実行方式を決定する:

単一タスクの場合:
- 直接TDDサイクルを実行する（ステップ3へ）

複数の独立タスク（並列実行可能）の場合:
- superpowers:dispatching-parallel-agentsを使用する
- 各エージェントに対象タスクのコンテキスト・ファイルパス・完了基準を渡す
- 全エージェントの完了を待ってから結果を統合する

複数の依存タスク（逐次実行必要）の場合:
- superpowers:subagent-driven-developmentを使用する
- タスクをWave順に1つずつ実行する
- 各タスク完了後にテストが通ることを確認してから次へ進む

#### 完了チェックポイント（ステップ2）

- 実行方式（直接TDD / 並列 / 逐次）が決定されていること
- 並列実行の場合は各エージェントへの渡す情報が整理されていること

### ステップ3: TDDサイクルの実行

superpowers:test-driven-development のパターンに従い実行する。

#### Red フェーズ（失敗するテストを書く）

1. テストファイルを作成・開く
2. 実装したい振る舞いを表すテストコードを書く
3. テストを実行し、失敗（Red）を確認する
4. テストが正しい理由で失敗していることを確認する（コンパイルエラーではなくアサーション失敗）

```bash
# テスト実行例（プロジェクトに合わせて変更）
pnpm vitest run src/__tests__/対象ファイル.test.ts
```

#### Green フェーズ（テストを通す最小実装）

1. テストを通す最小限のコードを書く
2. 過剰な最適化・抽象化はしない（まず動かすことを優先）
3. テストを実行し、成功（Green）を確認する
4. 既存のテストが壊れていないことを確認する（全テスト実行）

```bash
# 全テスト実行
pnpm vitest run
```

#### Refactor フェーズ（コードの整理）

1. テストが通ったまま、コードの品質を改善する
2. 重複の除去・命名の改善・責務の明確化を行う
3. リファクタリング後もテストが通ることを確認する
4. コードコメントを日本語で追加する（複雑なロジックには必ず）

各サイクル完了後:
- タスクの完了基準を満たしているか確認する
- 満たしていれば次のタスクへ進む

#### 完了チェックポイント（ステップ3）

- Red: テストが失敗していること（アサーション失敗であること）
- Green: テストが全て通っていること
- Refactor: コードが整理されており、テストが引き続き通っていること
- タスクの完了基準が全て満たされていること

### ステップ4: タスク完了の検証

superpowers:verification-before-completion のパターンに従い検証する。

1. 全テストの通過確認:
   ```bash
   pnpm vitest run
   ```

2. ビルドの成功確認:
   ```bash
   pnpm build
   # または
   pnpm typecheck
   ```

3. lintエラーの確認:
   ```bash
   pnpm lint
   # または
   pnpm oxlint
   ```

4. 実装漏れの確認:
   - タスクの完了基準を1つずつ確認する
   - 設計ドキュメントのコンポーネント設計と実装が対応しているか確認する

5. 次のWaveへの通知:
   - 現在のWaveの全タスクが完了したことを確認する
   - odinから呼ばれた場合は、完了状態と次のWaveのタスク一覧を返す

#### 完了チェックポイント（ステップ4）

- 全テストが通過していること（`pnpm vitest run` が成功）
- ビルドが成功していること（型エラーなし）
- lintエラーがないこと
- タスクの完了基準が全て満たされていること

## Examples

### Wave 1 のタスクを並列で実装する

ユーザー: 「plan-20260320-1700.md のWave 1を実装して」

```
ステップ1: 実装計画読み込み
  Wave 1: T-001（型定義）, T-002（DBスキーマ）- 独立タスク

ステップ2: 実行方式決定
  → 複数の独立タスク → dispatching-parallel-agents を使用

ステップ3: 並列実行
  Agent 1: T-001 型定義を TDD で実装
    Red: 型を使うテストを書く → 失敗
    Green: 型を定義する → 通過
    Refactor: JSDoc追加、命名改善
  Agent 2: T-002 DBスキーマを TDD で実装
    Red: マイグレーションテストを書く → 失敗
    Green: スキーマ定義 → 通過
    Refactor: コメント追加

ステップ4: 検証
  全テスト通過・ビルド成功・lintエラーなし → 完了
```

### 単一タスクを TDD で実装する

ユーザー: 「T-004（通知作成APIの実装）を実装して」

```
ステップ1: 計画確認
  T-004: Green フェーズ、src/app/api/notifications/route.ts
  依存: T-002（テスト）完了済み

ステップ2: 直接TDD実行

ステップ3: Green フェーズ
  既存テスト（T-002で作成）を確認
  テストを通す最小実装を行う
  全テスト通過を確認

ステップ4: 検証完了
```
