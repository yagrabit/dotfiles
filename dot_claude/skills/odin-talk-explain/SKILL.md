# odin-talk-explain

説明・レポート作成スキル。odin司令塔のtalkフェーズで使用する。
コードベース・機能・設計・不具合・進捗等を、対象者のレベルに合わせた言葉と粒度で説明する。チャット上への出力またはドキュメントへの保存を選択できる。

## Instructions

### 完了チェックポイントの原則

各ステップの最後には「完了チェックポイント」を設けている。
チェックポイントに記載された全ての条件を満たさない限り、次のステップに進んではならない。

### コンテキスト検出

$ARGUMENTS を確認し、odinコンテキストJSONが含まれているか判定する。

odinコンテキストJSONの例:
```json
{
  "odin_context": {
    "task": "通知機能のアーキテクチャ説明",
    "explain_target": "通知システムの全体構成",
    "audience": "新メンバー",
    "artifacts": {
      "design": ".claude/artifacts/design-20260320-1600.md",
      "research": ".claude/artifacts/research-20260320-1430.md"
    }
  }
}
```

- odinコンテキストがある場合: そこから説明対象と対象者を読み取る
- odinコンテキストがない場合: ステップ1からAskUserQuestionで確認する

### ステップ1: 説明対象と対象者の確認

AskUserQuestionで以下を確認する（未回答の項目のみ質問する）:

1. 説明対象（何を説明するか）:
   - コードベース・アーキテクチャ
   - 特定の機能・機能の仕組み
   - 不具合・エラーの原因
   - 開発の進捗・成果
   - 技術的な意思決定の背景
   - その他

2. 対象者（誰に向けて説明するか）:
   - 開発者（エンジニア）
   - 企画・PO（プロダクトオーナー）
   - ディレクター・経営層
   - 新メンバー（チーム加入したばかりの開発者）

3. 出力形式:
   - チャット上に出力
   - ドキュメントとして `.claude/artifacts/` に保存
   - 両方

4. 参考にすべき既存ドキュメントや成果物があれば確認する

#### 完了チェックポイント（ステップ1）

- 説明対象が明確になっていること
- 対象者が決定されていること
- 出力形式が決定されていること

### ステップ2: 情報収集

説明対象に応じて必要な情報を収集する。

コードベース・アーキテクチャの場合:
- Agentツール（Exploreエージェント）でコードベースを調査する
- ディレクトリ構成・主要ファイルの役割・技術スタックを把握する
- 既存の成果物（research-*.md, design-*.md）があれば読み込む

特定の機能の場合:
- 機能に関連するファイルをGrepとGlobで特定する
- 実装の主要部分をReadで読み込む
- 関連するテストファイルを確認する

不具合・エラーの場合:
- investigation-*.md があれば読み込む
- なければ症状・原因・影響範囲を整理する

開発進捗・成果の場合:
- git logで最近のコミット履歴を確認する:
  ```bash
  git log --oneline -20
  ```
- 変更ファイルの概要を把握する:
  ```bash
  git diff --stat $(git merge-base HEAD main)..HEAD
  ```
- 成果物ドキュメント（requirements-*.md, design-*.md, plan-*.md 等）を読み込む

#### 完了チェックポイント（ステップ2）

- 説明に必要な情報が収集されていること
- 説明対象の全体像が把握できていること

### ステップ3: 対象者別の説明生成

対象者のレベルに応じて説明スタイルを変える。

#### 開発者向け

- 技術用語を使い、正確さを優先する
- 具体的なファイルパスや関数名を参照する
- アーキテクチャの判断理由（なぜそうしたか）を含める
- コードスニペットで具体例を示す
- 「何ができるか」だけでなく「どう動くか」を説明する

例: 「通知機能はServer-Sent Events (SSE) を使って実装されています。
src/app/api/notifications/stream/route.ts がSSEエンドポイントで、
ReadableStreamを使ってリアルタイムに通知を送信します。...」

#### 企画・PO向け

- 技術用語は避けるか、使う場合は括弧内で補足する
- 「ユーザーにどんな影響があるか」を中心に説明する
- 数値・KPIで効果を示す
- リスクや制約は「できないこと」として明示する
- 意思決定が必要な場合は選択肢と影響を整理する

例: 「通知機能では、ページを再読み込みしなくても新しい通知がリアルタイムで表示されます。
ユーザーが気づきやすくなり、重要なお知らせの見逃しを防げます。...」

#### ディレクター・経営層向け

- ビジネスインパクトを先に示す
- 技術的な詳細は最小限にする
- 投資対効果・リスク・判断材料を整理する
- 「今何が起きているか」と「次に何が必要か」を明確にする
- 具体的な数字（工数、コスト、効果）を含める

例: 「通知機能の開発が完了しました。これにより、ユーザーの重要なアクション見逃し率を
推定30%削減できる見込みです。開発工数は2週間、追加インフラコストは月額〇円です。
次の判断として、段階的ロールアウトか全体公開かをご決定ください。...」

#### 新メンバー向け

- 背景・歴史的経緯から始める
- 「なぜそうなっているか」を都度説明する
- ステップバイステップで段階的に説明する
- 専門用語には全て注釈を付ける
- 「次に見るべきファイル・ドキュメント」を案内する

例: 「このプロジェクトはNext.js（Reactのフレームワーク）で作られています。
まず全体の構成から説明します。src/app/ がページのルーティング、
src/components/ が画面部品、src/lib/ がAPIとのやり取りを担当しています。
通知機能を理解するには、まず src/lib/notification.ts を読んでみてください。...」

#### 完了チェックポイント（ステップ3）

- 対象者のレベルに合わせた説明が生成されていること
- 説明に必要な情報が網羅されていること
- 技術用語の使い方が対象者に適切であること

### ステップ4: 出力

出力形式に応じて処理を行う。

チャット上への出力の場合:
- 説明を構造化してチャット上に表示する
- 見出し・箇条書きを活用して読みやすくする

ドキュメントへの保存の場合:
1. `.claude/artifacts/` ディレクトリが存在しなければ作成する
2. 現在時刻を取得し、`yyyyMMdd-HHmm` 形式のタイムスタンプを生成する
3. 以下のフォーマットで保存する

出力先: `.claude/artifacts/explanation-{yyyyMMdd-HHmm}.md`

```
# 説明ドキュメント

作成日: {yyyy-MM-dd HH:mm}
説明対象: {何を説明したか}
対象者: {誰向けか}
参照情報: {参照したファイル・成果物}

## {説明の見出し}

{説明の本文}

---

※ このドキュメントは {対象者} 向けに作成されました。
```

両方の場合はチャット表示後にファイル保存も行う。

出力後、必要であれば追加の質問や深掘り説明を申し出る。

## Examples

### コードベースのアーキテクチャを新メンバーに説明する

ユーザー: 「このプロジェクトのアーキテクチャを新メンバーに説明するドキュメントを作って」

```
ステップ1: 確認
  対象: コードベース・アーキテクチャ
  対象者: 新メンバー
  出力形式: ドキュメント保存

ステップ2: 情報収集
  Exploreエージェントでコードベース調査
  → Next.js App Router, Prisma, TypeScript, Zustand 構成を把握
  research-*.md があれば読み込む

ステップ3: 新メンバー向け説明生成
  背景から始め、段階的に構成を説明
  各ディレクトリの役割・代表的なファイルを案内

ステップ4: ドキュメント出力
  → .claude/artifacts/explanation-20260320-1700.md
```

### 不具合の原因を経営層に報告する

ユーザー: 「昨日の障害の原因と対応状況を経営層に説明するレポートを作って」

```
ステップ1: 確認
  対象: 不具合・障害対応
  対象者: 経営層
  出力形式: ドキュメント保存

ステップ2: 情報収集
  investigation-*.md があれば読み込む
  git logで修正コミットを確認

ステップ3: 経営層向け説明生成
  ビジネス影響→原因（非技術的）→対応状況→再発防止の順で説明
  具体的な影響時間・対象ユーザー数を含める

ステップ4: ドキュメント出力
  → .claude/artifacts/explanation-20260320-1800.md
```
