# odin-talk-propose

UI/UX・技術提案スキル。odin司令塔のtalkフェーズで使用する。
複数のアプローチを生成し、比較表付きで提案する。UI/UX提案時はモックを作成し、技術提案時はサンプルコードで具体的に示す。

## Instructions

### 完了チェックポイントの原則

各ステップの最後には「完了チェックポイント」を設けている。
チェックポイントに記載された全ての条件を満たさない限り、次のステップに進んではならない。

### コンテキスト検出

$ARGUMENTS を確認し、odinコンテキストJSONが含まれているか判定する。

odinコンテキストJSONの例:
```json
{
  "odin_context": {
    "task": "通知UIの提案",
    "proposal_type": "ui_ux",
    "constraints": "モバイルファーストで実装コスト低め",
    "artifacts": {
      "requirements": ".claude/artifacts/requirements-20260320-1500.md",
      "design": ".claude/artifacts/design-20260320-1600.md"
    }
  }
}
```

- odinコンテキストがある場合: artifactsから入力ファイルのパスと提案タイプを取得する
- odinコンテキストがない場合: ステップ1からAskUserQuestionで確認する

### 成果物管理

入力ファイルの探索順序:
1. $ARGUMENTSにファイルパスが明示指定されていればそれを使う
2. odinコンテキストのartifactsにパスがあればそれを使う
3. `.claude/artifacts/` 内の最新の `requirements-*.md` または `design-*.md` を使う
4. 該当ファイルが見つからない場合はAskUserQuestionでユーザーに確認する

### ステップ1: 提案テーマと制約の確認

AskUserQuestionで以下を確認する（未回答の項目のみ質問する）:

1. 提案テーマの種別:
   - UI/UX（画面設計・インタラクション）
   - 技術方針（実装アプローチ・ライブラリ選定）
   - アーキテクチャ（システム構成・データ設計）
   - その他

2. 制約条件:
   - 実装コストの目安（高/中/低）
   - 既存技術スタックとの整合性要件
   - パフォーマンス・セキュリティ要件
   - 期限・スケジュール

3. 参考にすべき既存ドキュメントやコードがあれば確認する

種別が「UI/UX」の場合は、ux-*スキルとの連携を検討する（ステップ3で実施）。

#### 完了チェックポイント（ステップ1）

- 提案テーマの種別が決定されていること
- 主要な制約条件が把握されていること

### ステップ2: 2-3パターンの生成

提案テーマと制約を踏まえて、2〜3つの異なるアプローチを生成する。

各パターンについて以下を整理する:

概要:
- アプローチの核となるアイデア（1〜2文）
- 他のパターンとの差別化ポイント

メリット/デメリット:
- 技術的なメリット
- 技術的なデメリット
- ユーザー体験への影響

実装コスト:
- 工数の見積もり（高/中/低 または 時間の目安）
- 必要なライブラリ・ツールの追加
- 既存コードへの影響範囲

リスク:
- 技術的リスク（未知の問題、パフォーマンス懸念等）
- ビジネスリスク（スケジュール遅延リスク等）
- 保守性リスク（長期的な維持コスト）

推奨度: ★★★ / ★★☆ / ★☆☆

推奨パターンを1つ選び、選定理由を明記する。

#### 完了チェックポイント（ステップ2）

- 2〜3パターンが生成されていること
- 各パターンのメリデメ・工数・リスクが整理されていること
- 推奨パターンが選定されていること

### ステップ3: モック・サンプルの作成

提案テーマの種別に応じてモックやサンプルを作成する。

UI/UX提案の場合:
- ux-3-core スキルを参照し、デザインガイドラインに沿ったモックを作成する
- HTMLモックファイルを `.claude/artifacts/mock-{yyyyMMdd-HHmm}.html` に出力する
- 各パターンのモックを作成する（または代表的なパターンのみ）

技術方針・アーキテクチャ提案の場合:
- 各パターンの核心となるサンプルコードを作成する
- コードはコメントを日本語で付け、意図を明確にする
- 実際のプロジェクトのコードスタイルに合わせる（既存コードをGrepで確認する）

サンプルコードのスタイル確認:
```bash
# 既存コードのパターンを確認
# 言語・フレームワークに応じて適宜調整する
```

#### 完了チェックポイント（ステップ3）

- UI/UX提案の場合: HTMLモックが作成されていること
- 技術提案の場合: 各パターンのサンプルコードが作成されていること
- コードにコメントが付いていること

### ステップ4: 比較表の提示

以下の形式で比較表を提示する。

```
## アプローチ比較

| 観点 | パターンA（推奨 ★★★） | パターンB（★★☆） | パターンC（★☆☆） |
|------|---------------------|----------------|----------------|
| 概要 | {概要} | {概要} | {概要} |
| 実装コスト | {高/中/低} | {高/中/低} | {高/中/低} |
| 保守性 | {高/中/低} | {高/中/低} | {高/中/低} |
| リスク | {高/中/低} | {高/中/低} | {高/中/低} |
| UX影響 | {内容} | {内容} | {内容} |

推奨: パターンA
理由: {選定理由}
```

比較表に加えて、各パターンの詳細（モックまたはサンプルコード）をチャット上に提示する。

#### 完了チェックポイント（ステップ4）

- 比較表が提示されていること
- 推奨パターンとその理由が明示されていること
- モックまたはサンプルコードが提示されていること

### ステップ5: ドキュメント出力と採用パターン確認

1. `.claude/artifacts/` ディレクトリが存在しなければ作成する
2. 現在時刻を取得し、`yyyyMMdd-HHmm` 形式のタイムスタンプを生成する
3. 提案内容をドキュメントとして出力する

出力先: `.claude/artifacts/proposal-{yyyyMMdd-HHmm}.md`

```
# 提案ドキュメント

作成日: {yyyy-MM-dd HH:mm}
提案テーマ: {テーマの概要}
種別: {UI/UX / 技術方針 / アーキテクチャ / その他}
参照ドキュメント: {requirements-*.md, design-*.md 等のパス}

## 背景・目的

{なぜこの提案が必要か}

## パターンA: {パターン名}（推奨 ★★★）

### 概要
{アプローチの説明}

### メリット
- {メリット1}
- {メリット2}

### デメリット
- {デメリット1}
- {デメリット2}

### 実装コスト
- 工数: {見積もり}
- 影響ファイル数: {概算}

### リスク
- {リスク内容}

### サンプル / モック
{コードまたはモックへの参照}

## パターンB: {パターン名}（★★☆）
...

## 比較表

{ステップ4の比較表}

## 推奨パターンと選定理由

採用推奨: {パターン名}
理由: {詳細な選定理由}

## 採用決定

採用パターン: {ユーザーが選んだパターン / 未決定}
決定日: {決定した日付 / 未定}
決定理由: {決定の経緯}
```

4. AskUserQuestionで採用パターンを確認する
5. 採用パターンが決まった場合はドキュメントを更新する
6. odinから呼ばれた場合は、出力ファイルパスと採用パターンをodinに返す

#### 完了チェックポイント（ステップ5）

- `.claude/artifacts/proposal-*.md` が出力されていること
- AskUserQuestionで採用パターンが確認されていること
- 採用パターンがドキュメントに記録されていること（未決定の場合も明記）

## Examples

### 通知UIのデザインを提案する

ユーザー: 「通知機能のUIを提案して。モバイルファーストで実装コストは低めで」

```
ステップ1: テーマと制約確認
  種別: UI/UX
  制約: モバイルファースト、実装コスト低、既存shadcn/uiスタック

ステップ2: 3パターン生成
  A: ベルアイコン + ドロップダウン（推奨 ★★★）
     コスト低、shadcn/ui Popoverで実装可能
  B: 専用通知ページ（★★☆）
     コスト中、UXはシンプル
  C: サイドパネル（★☆☆）
     コスト高、モバイルでの操作が複雑

ステップ3: HTMLモック作成
  → .claude/artifacts/mock-20260320-1700.html

ステップ4: 比較表提示
  パターンA推奨。モバイルでの操作性と実装コストのバランスが最良

ステップ5: ドキュメント出力・採用確認
  → .claude/artifacts/proposal-20260320-1700.md
  Q: 「パターンAのベルアイコン + ドロップダウンで進めますか？」
  A: 「Aで進める」→ ドキュメントに採用決定を記録
```

### 状態管理ライブラリの選定を提案する

ユーザー: 「グローバル状態管理をどうするか提案して。Next.js App Routerで使う」

```
ステップ1: テーマ確認
  種別: 技術方針
  制約: Next.js App Router, TypeScript, Server Components活用

ステップ2: パターン生成
  A: Zustand（推奨 ★★★）- 軽量、Server Components対応
  B: Jotai（★★☆）- アトミック設計、学習コスト中
  C: React Context（★☆☆）- 追加依存なし、パフォーマンス懸念あり

ステップ3: サンプルコード作成
  各パターンの基本的な使い方を日本語コメント付きで記述

ステップ4: 比較表提示

ステップ5: 採用確認
  → .claude/artifacts/proposal-20260320-1800.md
```
