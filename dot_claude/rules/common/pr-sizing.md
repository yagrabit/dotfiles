# PRサイズガイドライン

## 数値基準

| 区分 | 変更行数 | 判定 | 根拠 |
|------|----------|------|------|
| 理想 | 50-100行 | 推奨 | Graphite（150万PR分析）: 50行のPRは250行より40%速くマージ |
| 良好 | 101-200行 | 推奨 | Google推奨値。200行未満で1時間以内にレビュー完了 |
| 許容 | 201-400行 | 注意 | SmartBear/Cisco研究: 200-400行で欠陥検出率70-90% |
| ソフトリミット | 401-600行 | 警告 | Microsoft: 400行超で自動警告を実装し欠陥35%減 |
| ハードリミット | 601行以上 | 分割必須 | Propel研究: 1,000行超で欠陥検出率28%に激減 |

補足指標:
- ファイル数: 1-10ファイル
- レビュー時間: 60分以内
- 関心事の数: 1つ（1PR = 1つの目的）

## 原則

- 1チケット = 1PR を基本とする
- 1PRは1つのユーザー価値またはレビュー可能な単位に対応する
- AI生成コードは人間コードより問題が1.7倍多い（CodeRabbit調査）ため、小さいPRでのレビューがより重要

## 例外ケース

以下はハードリミット超過を許容する。PRの説明文に例外理由を明記すること。

- 自動生成コード: Prismaスキーマ、OpenAPI生成、ロックファイル等（行数カウントから除外）
- DBマイグレーション: 自動生成マイグレーションファイル（実質的なロジック変更行数で判定）
- 大規模リネーム/移動: ファイル名変更やディレクトリ構成変更（実質的な内容変更が少なければ許容）
- 依存パッケージ更新: ロックファイル差分は除外
- 初期セットアップ: 設計レビュー（ゲート1）通過済みであること

## 分割テクニック（優先度順）

1. Vertical Slicing（垂直分割）: アーキテクチャの全レイヤーを薄く貫通し、1PRで1つのユーザー価値を提供
2. Stacked PRs: 3-5 PRが実用的上限。上流PRのフィードバックは下流にカスケード
3. 機能分割: インターフェース先行PR → テスト先行PR → 実装PR
4. Feature Flags: 未完成機能をフラグで隠してmainにマージ

## 超過時の対応

- 400行以下: そのままPR作成
- 401-600行（ソフトリミット超過）: 例外ケース該当ならPR作成、非該当なら分割を提案
- 601行以上（ハードリミット超過）: 例外ケース該当ならPR作成、非該当なら分割必須

odin-do-prスキル内で自動チェックされる。odin-think-planでのタスク分解時にもこの基準を参照する。
